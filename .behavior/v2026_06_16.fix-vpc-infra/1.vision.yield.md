# vision: VPC infrastructure for declastruct-aws

## the outcome world

### before/after contrast

| before | after |
|--------|-------|
| EC2 integration tests cannot be written (no infrastructure) | EC2 tests can run against real infrastructure |
| VPC resources manually created in console | VPC resources declared as code |
| no idempotent VPC provision | `declastruct plan && declastruct apply` for VPCs |
| EC2 instances require manual subnet/sg lookup | refs link instances to VPC resources declaratively |

### the "aha" moment

a user declares their entire network topology in typescript — VPC, subnets, security groups, routes — and `declastruct apply` provisions it idempotently. when they later need an EC2 instance, they reference the subnet and security group by `exid`, and declastruct looks up the real AWS IDs automatically.

```typescript
// one file, entire network
const vpc = DeclaredAwsVpc.as({
  exid: 'my-app-vpc',
  cidr: { v4: '10.0.0.0/16' },
  // ...
});

const subnet = DeclaredAwsVpcSubnet.as({
  exid: 'my-app-subnet-1a',
  vpc: { exid: vpc.exid },  // ref, not hardcoded ID
  cidr: { v4: '10.0.1.0/24' },
  zone: { availability: 'us-east-1a' },
  // ...
});

const sg = DeclaredAwsVpcSecurityGroup.as({
  exid: 'my-app-sg',
  vpc: { exid: vpc.exid },  // ref, not hardcoded ID
  // ...
});

// EC2 instances can then use subnet and security group IDs
// (EC2 domain object updates are out of scope for this wish — see question #4)
```

### day-in-the-life

1. developer wants to run EC2 integration tests
2. they `source` the demo account credentials
3. `npx declastruct plan --wish provision/aws.demo/resources.vpc.ts`
4. they see the VPC, subnet, security group will be created
5. `npx declastruct apply --plan ...`
6. infrastructure exists
7. EC2 tests now pass

no clicking through AWS console. no hunting for subnet IDs. no "wait, which security group was it?"

## user experience

### usecases

| goal | contract | example |
|------|----------|---------|
| create isolated network | `DeclaredAwsVpc` | dev/prep/prod VPCs |
| place instances in network segment | `DeclaredAwsVpcSubnet` | public subnet for bastion, private for app |
| control traffic | `DeclaredAwsVpcSecurityGroup` | allow SSH from bastion only |
| enable internet access | `DeclaredAwsVpcInternetGateway` | public subnet needs outbound |
| route traffic | `DeclaredAwsVpcRouteTable` | 0.0.0.0/0 → internet gateway |

### contract inputs & outputs

#### DeclaredAwsVpc

```typescript
// input (user provides)
{
  exid: string;                    // unique key for declastruct
  cidr: { v4: string };       // e.g., { v4: '10.0.0.0/16' }
  dns: {
    hostnames: 'enabled' | 'disabled';
    support: 'enabled' | 'disabled';
  };
  tags: Record<string, string> | null;
}

// output (AWS provides)
{
  id: string;                      // vpc-xxxxx (metadata)
  // ...all input fields
}
```

#### DeclaredAwsVpcSubnet

```typescript
// input
{
  exid: string;
  vpc: Ref<typeof DeclaredAwsVpc>;  // { exid } or { id }
  cidr: { v4: string };        // e.g., { v4: '10.0.1.0/24' }
  zone: { availability: string };   // e.g., { availability: 'us-east-1a' }
  tags: Record<string, string> | null;
}

// output
{
  id: string;                       // subnet-xxxxx (metadata)
  availableIpAddressCount: number;  // readonly, from AWS
  // ...all input fields
}
```

#### DeclaredAwsVpcSecurityGroup

```typescript
// input
{
  exid: string;
  vpc: Ref<typeof DeclaredAwsVpc>;
  name: string;
  description: string;
  rules: {
    ingress: DeclaredAwsVpcSecurityGroupRule[];
    egress: DeclaredAwsVpcSecurityGroupRule[];
  };
  tags: Record<string, string> | null;
}

// rule structure
{
  protocol: 'tcp' | 'udp' | 'icmp' | 'all';
  fromPort: number;
  uptoPort: number;
  cidrs: DeclaredAwsVpcCidrBlock[];  // { v4?: string; v6?: string }
  description: string | null;
}

// output
{
  id: string;                       // sg-xxxxx (metadata)
  // ...all input fields
}
```

#### DeclaredAwsVpcInternetGateway

```typescript
// input
{
  exid: string;
  vpc: Ref<typeof DeclaredAwsVpc>;
  tags: Record<string, string> | null;
}

// output
{
  id: string;                       // igw-xxxxx (metadata)
  // ...all input fields
}
```

#### DeclaredAwsVpcRouteTable

```typescript
// input
{
  exid: string;
  vpc: Ref<typeof DeclaredAwsVpc>;
  routes: DeclaredAwsVpcRoute[];
  associations: DeclaredAwsVpcRouteTableAssociation[];
  tags: Record<string, string> | null;
}

// route structure
{
  destination: { cidr: { v4?: string; v6?: string } };  // e.g., { cidr: { v4: '0.0.0.0/0' } }
  target: PickOne<{
    gatewayInternet: { exid: string };      // internet gateway (looked up at apply time)
    gatewayNat: { exid: string };   // NAT gateway (future)
  }>;
}

// association structure
{
  subnet: Ref<typeof DeclaredAwsVpcSubnet>;
}

// output
{
  id: string;                       // rtb-xxxxx (metadata)
  // ...all input fields
}
```

### what it looks like to leverage them

```typescript
// provision/aws.demo/resources.vpc.ts

import {
  DeclaredAwsVpc,
  DeclaredAwsVpcSubnet,
  DeclaredAwsVpcSecurityGroup,
  DeclaredAwsVpcInternetGateway,
  DeclaredAwsVpcRouteTable,
} from 'declastruct-aws';

// 1. create VPC
const vpc = DeclaredAwsVpc.as({
  exid: 'declastruct-demo-vpc',
  cidr: { v4: '10.0.0.0/16' },
  dns: { hostnames: 'enabled', support: 'enabled' },
  tags: { managedBy: 'declastruct', purpose: 'demo' },
});

// 2. create subnet
const subnetPublic = DeclaredAwsVpcSubnet.as({
  exid: 'declastruct-demo-subnet-public-1a',
  vpc: { exid: vpc.exid },
  cidr: { v4: '10.0.1.0/24' },
  zone: { availability: 'us-east-1a' },
  tags: { managedBy: 'declastruct', purpose: 'demo' },
});

// 3. create security group
const sgDefault = DeclaredAwsVpcSecurityGroup.as({
  exid: 'declastruct-demo-sg-default',
  vpc: { exid: vpc.exid },
  name: 'declastruct-demo-sg-default',
  description: 'default security group for declastruct demo',
  rules: {
    ingress: [],  // no inbound by default (secure)
    egress: [
      {
        protocol: 'all',
        fromPort: 0,
        uptoPort: 0,
        cidrs: [{ v4: '0.0.0.0/0' }],
        description: 'allow all outbound',
      },
    ],
  },
  tags: { managedBy: 'declastruct', purpose: 'demo' },
});

// 4. create internet gateway (for public subnet)
const igw = DeclaredAwsVpcInternetGateway.as({
  exid: 'declastruct-demo-igw',
  vpc: { exid: vpc.exid },
  tags: { managedBy: 'declastruct', purpose: 'demo' },
});

// 5. create route table with internet route
const rtPublic = DeclaredAwsVpcRouteTable.as({
  exid: 'declastruct-demo-rt-public',
  vpc: { exid: vpc.exid },
  routes: [
    {
      destination: { cidr: { v4: '0.0.0.0/0' } },
      target: { gatewayInternet: { exid: igw.exid } },  // ref to internet gateway
    },
  ],
  associations: [
    { subnet: { exid: subnetPublic.exid } },
  ],
  tags: { managedBy: 'declastruct', purpose: 'demo' },
});

export const getResources = () => [vpc, subnetPublic, sgDefault, igw, rtPublic];
```

### timeline

1. **plan**: user runs `declastruct plan`, sees diff of what will be created
2. **apply**: user runs `declastruct apply`, resources created in dependency order
3. **drift**: user changes a field, re-runs plan, sees the update
4. **idempotent**: re-running apply with no changes = no-op

## mental model

### how users describe this to a friend

> "it's terraform but for typescript. you declare your AWS resources as typed objects, and it figures out what to create, update, or leave alone."

### analogies

| concept | analogy |
|---------|---------|
| `DeclaredAwsVpc` | the building (address space) |
| `DeclaredAwsVpcSubnet` | floors in the building |
| `DeclaredAwsVpcSecurityGroup` | door locks and access rules |
| `DeclaredAwsVpcInternetGateway` | the building's front door to the street |
| `DeclaredAwsVpcRouteTable` | elevator buttons / floor directory |

### terms: theirs vs ours

| AWS term | declastruct term | why |
|----------|------------------|-----|
| VpcId | `id` (metadata) | assigned by AWS, read-only |
| Name tag | `exid` | user-provided unique key |
| CidrBlock | `cidr: { v4, v6 }` | nested for IPv4/IPv6 extensibility |
| SecurityGroupId | `id` (metadata) | same pattern |
| GroupName | `name` | AWS calls it GroupName |
| DestinationCidrBlock | `destination: { cidr: { v4, v6 } }` | nested for consistency |
| IpRanges/CidrBlocks | `cidrs: [{ v4, v6 }]` | array of CIDR blocks |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| EC2 tests can run | yes | demo account will have VPC infrastructure |
| declarative VPC management | yes | same pattern as IAM, Lambda, etc. |
| idempotent provisioning | yes | findsert/upsert semantics |
| dependency lookup | yes | refs look up real IDs at apply time |

### pros

- **type-safe**: typescript catches typos in CIDR blocks, missing fields
- **idempotent**: safe to re-run
- **composable**: refs link resources without hardcoding IDs
- **consistent**: same patterns as extant declastruct resources
- **testable**: unit tests for domain objects, integration tests for operations

### cons

- **scope creep risk**: VPC has many sub-resources (NAT gateways, VPC endpoints, peering, etc.)
- **dependency ordering**: declastruct must create VPC before subnet, subnet before route table association
- **AWS eventual consistency**: some operations need waiters

### edgecases & pit of success

| edgecase | how we handle |
|----------|---------------|
| CIDR overlap | AWS rejects — we surface the error clearly |
| missing VPC ref | typescript compile error — ref is required |
| security group rule invalid port range | AWS rejects — we surface the error |
| subnet in wrong VPC | declastruct validates refs at apply time |
| orphaned resources | tags with `managedBy: 'declastruct'` enable cleanup scripts |

## open questions & assumptions

### assumptions

1. **`exid` as unique key**: we'll use an `exid` tag on VPC resources, same as EC2
2. **refs look up at apply time**: declastruct engine handles dependency ordering
3. **no NAT gateway in v1**: demo account doesn't need private subnets with NAT
4. **single region**: demo account is us-east-1 only

### questions for wisher

1. **route table target refs** [answered]: should `gatewayId` be a `Ref<typeof DeclaredAwsVpcInternetGateway>` or raw string?
   - **decision**: use nested exid ref pattern `gatewayInternet: { exid: string }`
   - matches the pattern for `vpc: { exid: string }` on subnets
   - `setRouteTable` looks up `igw.exid` → `igw-xxxxx` at apply time

2. **security group self-references** [answered]: should we support `sourceSecurityGroupId` (SG can reference itself)?
   - **decision**: v1 supports cidrs only, add sourceSecurityGroupId in v2
   - v1 goal is minimal viable; cidrs cover basic usecases
   - sourceSecurityGroupId can be added later without backwards incompatibility

3. **default vs explicit route tables** [answered]: AWS creates a "main" route table per VPC. should we model it?
   - **decision**: no — user creates explicit route tables, associates them
   - declastruct philosophy is declarative control, not reliance on ambient defaults
   - matches pattern for other implicit resources (default VPC, default SG)

4. **EC2 domain object update** [answered]: should VPC scope include updates to `DeclaredAwsEc2Instance` for subnet/security group refs?
   - **decision**: out of scope — this wish covers VPC resources only
   - the ".resources needed" section lists only: VPC, Subnet, Security Group, Internet Gateway, Route Table
   - EC2 modifications are NOT listed in scope
   - the ".test file updates needed" section shows aspirational code, not scope definition
   - EC2 can be modified in a separate wish if needed

### questions for external research

1. **none** — all AWS APIs are well-documented, no unknowns

### design questions (see "what is awkward" section)

5. **security group rules inline vs separate** [answered]: inline rules, matches IAM pattern
6. **subnet association lifecycle** [answered]: replace semantics acceptable, matches upsert

## what is awkward?

### route table target lookup

the route table's `target.gatewayInternet` needs to reference an internet gateway by exid.

**decision**: use nested exid refs. the route structure is:

```typescript
{
  destination: { cidr: { v4: '0.0.0.0/0' } },
  target: { gatewayInternet: { exid: 'my-igw' } },  // looked up to igw-xxxxx at apply time
}
```

**why not raw AWS IDs**: breaks declarative model — user shouldn't need to know `igw-xxxxx`

**why not full typed refs**: adds verbosity without benefit — `{ gatewayInternet: Ref<typeof DeclaredAwsVpcInternetGateway> }` is overkill for route targets

### security group rules as nested array vs separate resource [answered]

AWS models security group rules as attributes of the security group. but rules can also be managed separately via `AuthorizeSecurityGroupIngress`.

**options:**
1. **inline rules**: rules are part of `DeclaredAwsVpcSecurityGroup` — simpler, matches our pattern
2. **separate resource**: `DeclaredAwsVpcSecurityGroupRule` — more granular, but more complexity

**decision**: option 1 (inline). matches our pattern for IAM policy statements. also matches how extant `DeclaredAwsIamRole` embeds `DeclaredAwsIamPolicyStatement` inline.

### subnet association lifecycle [answered]

when associating a route table with a subnet, we're modifying the subnet's implicit route table association. if the subnet was previously associated with another route table, we're replacing that association.

**the question**: is "replace" semantics acceptable for idempotent operations?

**decision**: yes — route table associations are declarative. declastruct ensures the subnet is associated with the declared route table, replacing any prior association. this matches upsert semantics: the desired state is "this subnet uses this route table", regardless of prior state.

# blocker.1: Try/catch used to hide not-found errors in delete operations

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/vpc/delVpc.ts:35
- src/domain.operations/vpcInternetGateway/delVpcInternetGateway.ts:45
- src/domain.operations/vpcRouteTable/delVpcRouteTable.ts:30
- src/domain.operations/vpcSecurityGroup/delVpcSecurityGroup.ts:30
- src/domain.operations/vpcSubnet/delVpcSubnet.ts:30

Multiple delete operations use try/catch to catch specific AWS 'NotFound' errors and return silently (for idempotency). This hides real errors instead of rethrowing or using allowlisted handling with failfast. Per rules, try/catch is only allowed for very restricted allowlists that carefully handle and rethrow the rest; this pattern is a failhide hazard and mega blocker.

**snippet**:
```ts
try {
  await ec2.send(
    new DeleteVpcCommand({
      VpcId: vpc.id,
    }),
  );
} catch (error) {
  // ignore if already deleted
  if (error instanceof Error && error.name === 'InvalidVpcID.NotFound')
    return;
  throw error;
}
```

Get operations manually use try/catch to intercept AWS errors, return null for not-found, and wrap others with new HelpfulError({cause}). Rules specify using HelpfulError.wrap (or UnexpectedCodePathError.wrap) for making errors observable instead of manual try/catch patterns.

  if (error.name === 'InvalidVpcID.NotFound') return null;
  const metadata = ...;
  if (metadata?.httpStatusCode === 404) return null;

  throw new HelpfulError('aws.getOneVpc error', {
    cause: error,
    context: { ... }
  });
}

**locations**:
- src/domain.operations/vpc/getOneVpc.ts:90

- src/domain.operations/vpcSecurityGroup/getOneVpcSecurityGroup.ts:70
- src/domain.operations/vpcSubnet/getOneVpcSubnet.ts:70

AWS API errors are wrapped using new HelpfulError instead of semantic classes like MalfunctionError or UnexpectedCodePathError (which provide exit codes and caller/server distinction). Errors must use proper classes with full context per failloud rule; using base HelpfulError without semantic meaning is a blocker.

**snippet**:
```ts
throw new HelpfulError('aws.getOneVpc error', {
  cause: error,
  context: {
    errorName: error.name,
    errorMessage: error.message,
    httpStatusCode: metadata?.httpStatusCode,
    input,
  },
});
```

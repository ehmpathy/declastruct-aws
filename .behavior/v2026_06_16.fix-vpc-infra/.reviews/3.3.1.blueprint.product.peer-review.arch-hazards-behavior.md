# blocker.1: Race condition in security group rule sync

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md

The setVpcSecurityGroup.upsert operation performs a read-modify-write sequence without protection: it gets extant rules, compares, revokes removed, and authorizes added. This can produce inconsistent results or errors under concurrent access, as two instances may interfere with each other's changes. This is exactly the type of behavior hazard that causes intermittent failures and data corruption in production. The design must use atomic operations or explicit locking to be predictable.

**snippet**:
```markdown
setVpcSecurityGroup.upsert must sync rules:

1. get extant rules from AWS
2. compare with declared rules
3. revoke removed rules
4. authorize added rules

this is necessary because AWS security group rules are additive — an update to the security group does not replace rules, it adds to them.
```

# blocker.3: Non-atomic findsert enabling duplicate resources under concurrency

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md

findsert is implemented as 'find extant or Create' using tag:exid lookup. If two operations run simultaneously, both may find absent and create duplicates. Subsequent get by exid will hit 'multiple VPCs → error'. This violates idempotency and predictability, as same inputs can produce different outputs depending on timing.

**snippet**:
```markdown
getVpc                        [+] create
│   ├── by.primary: { id }            # DescribeVpcs with VpcIds
│   ├── by.unique: { exid }           # DescribeVpcs with tag:exid filter
│   └── by.ref                        # dispatch to primary or unique
...
setVpc.findsert: absent → creates | extant → returns extant | idempotent rerun
...
getVpc | extant VPC → returns object | absent VPC → null | multiple VPCs → error
```

# blocker.4: Order-dependence in resource provisioning

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md

Creation operations have implicit order requirements: e.g. subnet set requires VPC to exist or errors, route table requires gateway. The design depends on specific registration order in getDeclastructAwsProvider and declastruct apply order. Reversed or concurrent application breaks behavior. This is order-dependence hazard.

**snippet**:
```markdown
dependency order
declastruct applies resources in dependency order. for VPC resources:

1. DeclaredAwsVpc                    # no dependencies
2. DeclaredAwsVpcSubnet              # depends on VPC
   DeclaredAwsVpcSecurityGroup       # depends on VPC (parallel with subnet)
   DeclaredAwsVpcInternetGateway     # depends on VPC (parallel with subnet)
3. DeclaredAwsVpcRouteTable          # depends on VPC, subnet, gateway

setVpcSubnet.findsert: absent → creates | extant → returns extant | VPC not found → error
setVpcInternetGateway.findsert: absent → creates + attaches | extant → returns | VPC not found → error
setVpcRouteTable.findsert: absent → creates routes + associations | extant → returns | gateway not found → error
```

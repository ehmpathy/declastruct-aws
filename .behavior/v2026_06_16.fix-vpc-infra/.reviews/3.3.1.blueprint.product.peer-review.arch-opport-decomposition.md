# blocker.1: Domain operations use non-compliant verbs and lack cardinality

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/evolvable.domain.operations/rule.require.get-set-gen-verbs.md.min

**locations**:
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md

All domain operations must use exactly one of get, set, or gen. get must be getOne* or getAll*. Here the proposed operations are named getVpc (no One/All), setVpc, and delVpc. del is not permitted. Additionally, internal 'findsert' is used for find-or-create semantics which should be gen per the rule. This breaks the closed symmetric verb system required for clarity, idempotency, and recomposition.

**snippet**:
```markdown
vpc/
├── getVpc                        [+] create
│   ├── by.primary: { id }            # DescribeVpcs with VpcIds
│   ├── by.unique: { exid }           # DescribeVpcs with tag:exid filter
│   └── by.ref                        # dispatch to primary or unique

├── setVpc                        [+] create
│   ├── findsert: DeclaredAwsVpc      # find extant or CreateVpc
│   └── upsert: DeclaredAwsVpc        # find + update tags, or CreateVpc

├── delVpc                        [+] create
│   └── by.ref                        # DeleteVpc (idempotent)
```

# blocker.2: Orchestrator descriptions include decode-friction and implementation details

**rule**: .agent/repo=ehmpathy/role=architect/briefs/practices/rule.require.orchestrators-as-narrative.md

**locations**:
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md

Orchestrators must read as narrative prose where each line tells 'what' not 'how'. The blueprint for setVpcRouteTable and setVpcSecurityGroup.upsert embeds AWS implementation steps and numbered 'how' lists directly in the operation descriptions. This indicates the resulting code will contain inline logic that requires mental simulation, violating the requirement for named leaf operations only in orchestrators.

**snippet**:
```markdown
setVpcRouteTable          [+] create
│   ├── findsert: DeclaredAwsVpcRouteTable  # CreateRouteTable + CreateRoute + AssociateRouteTable
│   └── upsert: DeclaredAwsVpcRouteTable    # sync routes + associations

setVpcSecurityGroup.upsert must sync rules:
1. get extant rules from AWS
2. compare with declared rules
3. revoke removed rules
4. authorize added rules
```

# blocker.3: Proposed operations mix grains and contain inline i/o + logic

**rule**: .agent/repo=ehmpathy/role=architect/briefs/practices/rule.forbid.decode-friction-in-orchestrators.md

**locations**:
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md

Operations must have clear grain (transformer/communicator/orchestrator). set* operations are described as performing find + Create + Attach + rule sync in one place with inline comments showing raw AWS calls and steps. This mixes communicator i/o with orchestration and compute without extracting to leaf operations (e.g. no separate sdk* or compute* calls). Violates grain clarity and recomposition potential.

**snippet**:
```markdown
setVpcInternetGateway         [+] create
│   ├── findsert: DeclaredAwsVpcInternetGateway  # CreateInternetGateway + AttachInternetGateway
│   └── upsert: DeclaredAwsVpcInternetGateway

setVpcSecurityGroup         [+] create
│   ├── findsert: DeclaredAwsVpcSecurityGroup
│   └── upsert: DeclaredAwsVpcSecurityGroup  # includes rule sync
```

# nitpick.1: Blueprint embeds how in operation specs instead of naming leaf operations

**rule**: .agent/repo=ehmpathy/role=architect/briefs/practices/rule.prefer.decomposable-architecture.md

**locations**:
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md

Decomposition requires orchestrators to compose only named leaf operations with no decode-friction. The codepath tree and sync sections repeatedly inline AWS SDK calls and step-by-step logic inside operation descriptions (e.g. 'DescribeVpcs with VpcIds', 'DisassociateRouteTable + DeleteRoute + DeleteRouteTable'). This should be extracted to named communicators/transformers first.

**snippet**:
```markdown
getVpc                        [+] create
│   ├── by.primary: { id }            # DescribeVpcs with VpcIds
│   ├── by.unique: { exid }           # DescribeVpcs with tag:exid filter
│   └── by.ref                        # dispatch to primary or unique

vpcRouteTable/
└── delVpcRouteTable              [+] create
    └── by.ref                        # DisassociateRouteTable + DeleteRoute + DeleteRouteTable
```

# blocker.1: Idempotent delete via error catch hides real errors

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md (codepath tree)
- .behavior/v2026_06_16.fix-vpc-infra/3.3.1.blueprint.product.yield.md (coverage by case)

The blueprint plans to implement delVpc as 'idempotent' by using error catch for the absent case (no-op instead of propagating the error). This is a classic failhide: real errors (e.g. permission, transient, or unexpected AWS failures) will be swallowed instead of failing fast and loud. The only allowed try/catch must have an explicit allowlist and rethrow everything else; 'idempotent error catch' does not specify this and therefore violates failfast and failloud requirements.

**snippet**:
```md
delVpc                        [+] create
   └── by.ref                        # DeleteVpc (idempotent)

pattern.6 = delete with idempotent error catch | all delVpc* operations | [REUSE]

delVpc | extant → deleted | absent → no-op (idempotent) | VPC with subnets → error
```

# blackbox criteria: VPC infrastructure

## usecase.1 = declare and provision VPC

```
given a user declares a VPC
  when they plan
    then the plan shows the VPC will be created
      sothat user can review before apply
  when they apply
    then the VPC exists in AWS with the declared CIDR
    then the VPC has DNS hostnames and support as declared
    then the VPC has the exid tag for lookup
      sothat declastruct can find it by unique key
  when they apply again (idempotent)
    then no changes are made
      sothat repeated applies are safe

given a VPC exists
  when they get by exid
    then they receive the VPC with its AWS-assigned id
  when they get by id
    then they receive the VPC
  when they get by exid that does not exist
    then they receive null
```

## usecase.2 = declare and provision subnet

```
given a user declares a subnet in a VPC
  when the VPC does not exist yet
    then apply fails with clear error
      sothat user knows to create VPC first
  when the VPC exists
    when they apply
      then the subnet exists in AWS with the declared CIDR
      then the subnet is in the declared availability zone
      then the subnet has the exid tag for lookup
  when they apply again (idempotent)
    then no changes are made

given a subnet exists
  when they get by exid
    then they receive the subnet with its AWS-assigned id
    then they receive the available IP address count (readonly)
```

## usecase.3 = declare and provision security group

```
given a user declares a security group in a VPC
  when the VPC does not exist yet
    then apply fails with clear error
  when the VPC exists
    when they apply
      then the security group exists in AWS
      then the ingress rules match the declaration
      then the egress rules match the declaration
      then the security group has the exid tag for lookup

given a security group exists with rules
  when they update the rules in the declaration
    when they apply
      then the rules are updated to match the declaration
        sothat security group rules are declaratively managed

given a security group with 'all' protocol egress
  when they apply
    then AWS receives protocol -1 (all traffic allowed)
```

## usecase.4 = declare and provision internet gateway

```
given a user declares an internet gateway for a VPC
  when the VPC does not exist yet
    then apply fails with clear error
  when the VPC exists
    when they apply
      then the internet gateway exists in AWS
      then the internet gateway is attached to the VPC
      then the internet gateway has the exid tag for lookup

given an internet gateway exists
  when they get by exid
    then they receive the internet gateway with its AWS-assigned id
```

## usecase.5 = declare and provision route table

```
given a user declares a route table with routes and associations
  when the VPC does not exist yet
    then apply fails with clear error
  when a route references a gateway that does not exist
    then apply fails with clear error
      sothat user knows to create gateway first
  when a route association references a subnet that does not exist
    then apply fails with clear error
  when all dependencies exist
    when they apply
      then the route table exists in AWS
      then the routes are created (e.g., 0.0.0.0/0 -> internet gateway)
      then the subnet associations are created
      then the route table has the exid tag for lookup

given a route table exists with routes
  when they update the routes in the declaration
    when they apply
      then the routes are updated to match the declaration

given a route table exists with subnet associations
  when they update the associations in the declaration
    when they apply
      then the associations are updated to match the declaration
        sothat route table associations are declaratively managed
```

## usecase.6 = delete resources

```
given a VPC exists with no dependencies
  when they remove the declaration and apply
    then the VPC is deleted from AWS

given a VPC exists with subnets
  when they try to delete the VPC without first removing subnets
    then apply fails with clear error
      sothat user knows to delete dependencies first

given a security group exists
  when they remove the declaration and apply
    then the security group is deleted from AWS

given an internet gateway exists attached to VPC
  when they remove the declaration and apply
    then the internet gateway is detached and deleted

given a route table exists with associations
  when they remove the declaration and apply
    then the associations are removed
    then the route table is deleted
```

## usecase.7 = provision demo infrastructure (acceptance)

```
given the demo account has no VPC infrastructure
  when user declares the full demo stack (VPC, subnet, SG, IGW, route table)
    when they plan
      then all 5 resources show as "to be created"
    when they apply
      then all 5 resources exist in AWS
      then the subnet can route to internet via the internet gateway
        sothat EC2 instances can be launched for integration tests

given the demo infrastructure exists
  when they apply again
    then no changes are made (idempotent)
```

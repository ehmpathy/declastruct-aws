# 1.vision — feat-budget

> illustrate the world where we can declaratively control an aws account's budget:
> **alerts** on spend increases (anomalies + thresholds) and **guards** on spend
> breaches (hard stop-spend, esp. for dev/demo accounts).

---

## the outcome world

### day-in-the-life

**before.** a demo account exists so folks can kick the tires. someone spins up a
`p5.48xlarge` "just to test", forgets it, and it runs all weekend. nobody notices
until the monthly bill lands. the only "budget control" is a human who remembers to
eyeball Cost Explorer. dev/demo accounts are a standing liability — an unattended
credit card.

**after.** the demo account's budget posture is declared, versioned, and applied
just like its VPC or its SCPs:

```bash
npx declastruct plan  --wish provision/aws.budget/account=demo/resources.ts --into .temp/plan.json
npx declastruct apply --plan .temp/plan.json
```

the account now carries these declared postures, side by side with the extant org
resources:

1. a **budget** (the $21/mo cap) plus its **threshold alerts** at 50% / 80% / 100% — note:
   AWS's *CFN* nests notifications in the budget, but the *SDK* gives them independent CRUD
   (`CreateNotification`/`DeleteNotification`), so we declare each alert tier as its OWN
   `DeclaredAwsBudgetNotification` resource that REFERENCES the budget (its identity is the
   `[budget, type, comparison, threshold]` tuple). the cap and the alerts are distinct
   declarable resources
2. a **cost-anomaly monitor** that ML-watches for spend spikes and emails when the
   pattern breaks (anomalies)
3. a **budget guard that blocks NEW spend** — at 100%, AWS Budgets auto-attaches a declared
   "deny-expensive-stuff" SCP (`APPLY_SCP_POLICY`), so the runaway `p5` can't relaunch
4. a **budget guard that KILLS active spend** — at 110%, AWS Budgets runs an SSM automation
   doc (`RUN_SSM_DOCUMENTS`) that stops/terminates the already-active wasteful instances

   > ⚠️ scope note: the two guards are complementary. the SCP guard (3) blocks *new*
   > launches but does NOT touch live resources; the SSM guard (4) is what actually
   > *kills* already-active waste. declare both for a true "stop use"; declare just one
   > if you only want one behavior. (guards are `DeclaredAwsBudgetAction` resources that
   > REFERENCE the budget — they are not part of it.)

re-run `apply` tomorrow → every resource converges to `KEEP`. no drift, no surprise.

### before/after contrast

| concern              | before                              | after                                             |
| -------------------- | ----------------------------------- | ------------------------------------------------- |
| threshold alert      | manual Cost Explorer checks         | `DeclaredAwsBudget` + `DeclaredAwsBudgetNotification` tiers |
| anomaly alert        | absent (or noticed on the bill)     | `DeclaredAwsCostAnomalyMonitor` + subscription    |
| block new spend      | absent — hope + trust               | `DeclaredAwsBudgetAction` (`APPLY_SCP_POLICY`) attaches an SCP |
| kill active spend    | absent — hope + trust               | `DeclaredAwsBudgetAction` (`RUN_SSM_DOCUMENTS`) stops instances |
| estimated-charges alarm | absent                           | `DeclaredAwsCloudwatchMetricAlarm` on `EstimatedCharges` |
| reproducibility      | click-ops per account               | one `resources.ts`, `plan`/`apply`, git-tracked   |
| dev/demo account risk| standing liability                  | bounded, self-enforcing cap                       |

### the "aha" moment

the guards reuse resources **already in this repo**. a budget breach doesn't invent a new
enforcement mechanism — the block-new guard flips on an SCP we already know how to declare
(`DeclaredAwsOrganizationServiceControlPolicy`), and the kill-active guard runs an SSM doc.
budget = the trigger, SCP/SSM = the hammers. they compose. that's when it clicks: "spend
control" is just "declare the cap, declare the hammer, wire the trigger."

## user experience

### the resource set (canonical — this is the whole feature)

this feature adds **6 new declared resources**. every other section elaborates on these:

| # | resource | role | identity | refs |
|---|----------|------|----------|------|
| 1 | `DeclaredAwsBudget` | the spend cap (limit only) | `unique = [name]` | — |
| 2 | `DeclaredAwsBudgetNotification` | one threshold-alert tier | `unique = [budget, notificationType, comparison, threshold]`, no `primary` | → Budget |
| 3 | `DeclaredAwsBudgetAction` | a guard (block-new via SCP, or kill-active via SSM) | AWS-assigned | → Budget + SCP/SSM + IAM role |
| 4 | `DeclaredAwsCostAnomalyMonitor` | ML anomaly watcher | `unique = [name]` | — |
| 5 | `DeclaredAwsCostAnomalySubscription` | anomaly alert router | `unique = [name]` | → Monitor |
| 6 | `DeclaredAwsCloudwatchMetricAlarm` | generic metric alarm (e.g. the `EstimatedCharges` cost alarm) | `unique = [name]` | — |

plus, **in-scope for this PR's execution** (not a new resource — a rename): the extant
CloudWatch family is retrofitted to the `Cloudwatch` namespace
(`DeclaredAwsLogGroup` → `DeclaredAwsCloudwatchLogGroup` + its two report composites), so #6
lands into a symmetric family. see groundwork › peer-symmetry retrofit.

### usecases / goals

| # | goal                                            | resource(s) leveraged                                              |
|---|-------------------------------------------------|--------------------------------------------------------------------|
| 1 | "email me when this account passes $X or Y%"     | `DeclaredAwsBudget` (the cap) + one `DeclaredAwsBudgetNotification` per tier |
| 2 | "email me when spend spikes abnormally"          | `DeclaredAwsCostAnomalyMonitor` + `DeclaredAwsCostAnomalySubscription` |
| 3 | "block NEW spend on this demo acct at 100%"      | `DeclaredAwsBudgetAction` (`APPLY_SCP_POLICY`) → refs a `DeclaredAwsOrganizationServiceControlPolicy` |
| 4 | "KILL active resources on this demo acct at 110%"| `DeclaredAwsBudgetAction` (`RUN_SSM_DOCUMENTS`) → runs an SSM stop/terminate doc |
| 5 | "alarm on estimated charges (a cost signal)"     | `DeclaredAwsCloudwatchMetricAlarm` on the `EstimatedCharges` metric |
| 6 | "same posture across N accounts, reproducibly"   | all of the above, in a per-account `resources.ts`                  |

### requirements this feature MUST deliver

beyond the 6 resources, this feature must ALSO ship these two things — they are in-scope
deliverables, not optional extras:

- **R1 — ship a reference usecase.** provide the full budget posture (cap + tiered
  notifications + block-new SCP guard + kill-active SSM guard) as ONE copy-pasteable
  reference example. ~8 objects is fine — we ship the honest primitives and a worked
  example that consumers copy + adapt (or wrap in their own factory). the example is the
  ergonomic layer, not a collapsed DAO'd type (per `rule.forbid.dao-for-narrow-usecase-resource`).

- **R2 — dogfood it against our own infra.** declare that reference posture in
  `provision/aws.infra/account=demo/` (a new `resources.budget.ts`, wired into the extant
  `resources.ts`) and apply it for real, per `howto.dogfood-aws-resources`. this proves the
  whole plan/apply path works end-to-end AND guards our own demo account.
  - **hard cap: the demo account must not spend more than $21/mo.** the dogfood budget's
    limit is `$21 USD MONTHLY`, with the kill-active guard armed — so our own demo account
    is bounded by the very feature we ship. ($21 is the real cap; the testdrive below uses
    the same number.)

### contract inputs & outputs — testdrive

these mirror the extant `.as()` / `getResources()` shape used across the repo
(see `src/contract/sdks/.test/assets/resources.acceptance.ts`).

**(1) the cap — `DeclaredAwsBudget`** (the ceiling; alerts are separate, below)

```ts
const demoBudget = DeclaredAwsBudget.as({
  name: 'declastruct-demo-monthly',        // @unique per account
  budgetType: 'COST',                       // COST | USAGE | ...
  limit: { amount: '21', unit: 'USD' },     // the cap (matches the demo dogfood cap)
  timeUnit: 'MONTHLY',
  costFilters: null,                        // null = whole account
  tags: { managedBy: 'declastruct', purpose: 'demo-guardrail' },
});
```

**(1b) threshold alerts — `DeclaredAwsBudgetNotification`** (own resource, refs the budget)

each alert tier is its OWN declared resource that references the budget — mirrors the
extant `DeclaredAwsOrganizationServiceControlPolicyAttachment` (composite `unique`, no
`primary`, `RefByUnique` to its parent). AWS backs this with independent
`CreateNotification`/`DeleteNotification`/`DescribeNotificationsForBudget` CRUD, so a tier
can be added/removed without a budget rewrite. subscribers stay nested (leaf), symmetric
with `DeclaredAwsCostAnomalySubscription`.

```ts
const budgetRef = RefByUnique.as<typeof DeclaredAwsBudget>(demoBudget);

const alert50 = DeclaredAwsBudgetNotification.as({
  budget: budgetRef,                        // @unique part 1 — refs the budget
  notificationType: 'ACTUAL',               // @unique part 2
  comparison: 'GREATER_THAN',               // @unique part 3
  threshold: 50,                            // @unique part 4 — the tuple IS its identity
  thresholdType: 'PERCENTAGE',
  subscribers: [{ via: 'EMAIL', address: 'ops@ehmpath.com' }],
});
const alert80  = DeclaredAwsBudgetNotification.as({ budget: budgetRef, notificationType: 'ACTUAL',     comparison: 'GREATER_THAN', threshold: 80,  thresholdType: 'PERCENTAGE', subscribers: [{ via: 'EMAIL', address: 'ops@ehmpath.com' }] });
const alert100 = DeclaredAwsBudgetNotification.as({ budget: budgetRef, notificationType: 'FORECASTED', comparison: 'GREATER_THAN', threshold: 100, thresholdType: 'PERCENTAGE', subscribers: [{ via: 'EMAIL', address: 'ops@ehmpath.com' }] });
```

**(2) anomaly alerts — monitor + subscription**

```ts
const demoAnomalyMonitor = DeclaredAwsCostAnomalyMonitor.as({
  name: 'declastruct-demo-anomaly',         // @unique
  type: 'DIMENSIONAL',                      // DIMENSIONAL (by SERVICE) | CUSTOM
  dimension: 'SERVICE',
  tags: { managedBy: 'declastruct' },
});

const demoAnomalySub = DeclaredAwsCostAnomalySubscription.as({
  name: 'declastruct-demo-anomaly-sub',     // @unique
  monitor: RefByUnique.as<typeof DeclaredAwsCostAnomalyMonitor>(demoAnomalyMonitor),
  frequency: 'IMMEDIATE',                   // IMMEDIATE | DAILY | WEEKLY
  threshold: { amount: '20', unit: 'USD' }, // alert when anomaly impact > $20
  subscribers: [{ via: 'EMAIL', address: 'ops@ehmpath.com' }],
  tags: { managedBy: 'declastruct' },
});
```

**(3) hard guard — `DeclaredAwsBudgetAction` (supports BOTH halt kinds)**

the guard supports **two action types**, and a user may declare either or both:

- **`APPLY_SCP_POLICY`** — blocks *new* spend (attach a deny-SCP; no new expensive launches)
- **`RUN_SSM_DOCUMENTS`** — halts *active* spend (run an SSM automation doc to stop/terminate
  running instances)

together they deliver a true "stop use": block new commitments AND wind down what's live.

```ts
// hammer A: a deny-SCP — declared the way this repo already declares SCPs
const denyExpensiveScp = DeclaredAwsOrganizationServiceControlPolicy.as({
  name: 'declastruct-demo-deny-expensive',
  description: 'block launch of expensive resources when budget is breached',
  content: new DeclaredAwsIamPolicyDocument({ statements: [ /* deny ec2:RunInstances etc */ ] }),
  tags: null,
});

// guard A — block NEW spend: at 100% actual, attach the deny-SCP to the account
const demoGuardBlockNew = DeclaredAwsBudgetAction.as({
  budget: RefByUnique.as<typeof DeclaredAwsBudget>(demoBudget),
  actionType: 'APPLY_SCP_POLICY',            // APPLY_SCP_POLICY | APPLY_IAM_POLICY | RUN_SSM_DOCUMENTS
  threshold: { value: 100, type: 'PERCENTAGE', notificationType: 'ACTUAL' },
  approvalModel: 'AUTOMATIC',                // AUTOMATIC = no human in the loop
  executionRole: RefByUnique.as<typeof DeclaredAwsIamRole>({ name: 'declastruct-budgets-action-role' }),
  definition: {
    scp: {
      policy: RefByUnique.as<typeof DeclaredAwsOrganizationServiceControlPolicy>(denyExpensiveScp),
      target: { account: RefByUnique.as<typeof DeclaredAwsOrganizationAccount>({ email: 'demo@ehmpath.com' }) },
    },
  },
  subscribers: [{ via: 'EMAIL', address: 'ops@ehmpath.com' }],
});

// guard B — halt ACTIVE spend: at 110% actual, run an SSM doc that stops running instances
const demoGuardHaltActive = DeclaredAwsBudgetAction.as({
  budget: RefByUnique.as<typeof DeclaredAwsBudget>(demoBudget),
  actionType: 'RUN_SSM_DOCUMENTS',           // stop/terminate live resources
  threshold: { value: 110, type: 'PERCENTAGE', notificationType: 'ACTUAL' },
  approvalModel: 'AUTOMATIC',
  executionRole: RefByUnique.as<typeof DeclaredAwsIamRole>({ name: 'declastruct-budgets-action-role' }),
  definition: {
    ssm: {
      // AWS-provided doc that stops EC2 instances; or a custom doc ref
      documentName: 'AWS-StopEC2Instance',
      region: 'us-east-1',
      instanceIds: [ /* or a tag-filter the doc resolves */ ],
    },
  },
  subscribers: [{ via: 'EMAIL', address: 'ops@ehmpath.com' }],
});

export const getResources = async () => [
  denyExpensiveScp,     // hammer A declared first
  demoBudget,           // the cap (declared before its notifications + guards ref it)
  alert50, alert80, alert100, // threshold alerts — own resources, ref the budget
  demoAnomalyMonitor,   // anomaly watcher
  demoAnomalySub,       // anomaly router (refs monitor)
  demoGuardBlockNew,    // guard A: block new spend (refs budget + scp + role)
  demoGuardHaltActive,  // guard B: halt active spend (refs budget + role)
];
```

outputs are the usual: `plan` shows `CREATE` on first run, `apply` provisions,
re-`plan` shows `KEEP`. reads return `HasReadonly<...>` domain objects (ids/arns
filled in).

> **staged halt**: declaring both at tiered thresholds (block-new @ 100%, halt-active @
> 110%) gives a graceful escalation — first stop the bleeding of *new* spend, then, if it
> keeps climbing, wind down what's *already* live. neither is required; a user can declare
> just one.

### timeline the user goes through

1. write `resources.ts` for the account (copy a sibling account's file, tweak limits/emails)
2. `plan` → eyeball the CREATE set
3. `apply` → resources provisioned
4. `plan` again → all `KEEP` (proves idempotency, no drift)
5. spend crosses a notification threshold → email arrives
6. spend crosses 100% → SCP auto-attaches → new expensive launches denied
7. spend crosses 110% → SSM doc runs → already-running instances stopped/terminated
8. next budget period rolls → AWS auto-reverses the SCP action (SCP detaches). note: the
   SSM stop is NOT auto-undone — stopped instances stay stopped until a human restarts them

## mental model

### how a user describes it to a friend

> "it's a thermostat for the AWS bill. i declare the temperature i want (the dollar
> ceiling), who gets pinged as it warms up (the alerts), and at what point it slams
> the vents shut (the guard). all in a file i commit, same as the rest of my infra."

### analogies / metaphors

- **thermostat**: ceiling + alerts + auto-shutoff
- **circuit breaker**: normal flow until an overload trips it; resets next period
- **speed governor**: you can drive, but not past the declared limit

### terms: theirs vs ours

| they say            | we say (ubiqlang)                                        |
| ------------------- | -------------------------------------------------------- |
| "budget alert"      | `DeclaredAwsBudgetNotification` (refs the budget)        |
| "spend anomaly"     | `DeclaredAwsCostAnomalyMonitor` + subscription           |
| "kill switch" / "cap"| `DeclaredAwsBudget` (cap) + `DeclaredAwsBudgetAction` (guard) |
| "stop the account"  | a guard that attaches a `DeclaredAwsOrganizationServiceControlPolicy` (block new) or runs an SSM doc (kill active) |
| "who gets paged"    | `subscribers` (EMAIL / SNS)                              |

we deliberately reuse `Declared…` prefixes, `.as()`, `RefByUnique`, `threshold`,
`subscribers`, and compose atop the extant SCP/account/role objects rather than mint
new vocabulary.

## evaluation

### how well it solves the goals

- **thresholds** → covered by `DeclaredAwsBudget` (cap) + `DeclaredAwsBudgetNotification` tiers ✅
- **anomalies** → covered by anomaly monitor + subscription ✅
- **hard guards** → covered by budget actions: `APPLY_SCP_POLICY` (block new) + `RUN_SSM_DOCUMENTS` (kill active) ✅
- **estimated-charges signal** → covered by `DeclaredAwsCloudwatchMetricAlarm` ✅
- **dev/demo emphasis** → the guards are the exact "stop-spend for dev/demo" ask ✅

### pros

- **composes, doesn't reinvent**: the guard reuses `DeclaredAwsOrganizationServiceControlPolicy`
  — symbiogenesis, not a parallel enforcement path
- **declarative dogfood**: each resource gets a DAO + acceptance coverage, per
  `rule.require.dao-and-acceptance-per-declared-resource`
- **findsert/upsert idempotency** maps onto AWS `Describe`/`Create`/`Update` — cleanly
  for the SCP (one `DescribePolicy`), but the budget's **notifications are sub-resources**
  (reconcile needs `DescribeNotificationsForBudget` + per-tier
  `DescribeSubscribersForNotification`), so its idempotency is multi-call, not a single get
- **per-account file** = reproducible posture across N accounts

### cons / tradeoffs

- **new AWS SDK deps**: `@aws-sdk/client-budgets` and `@aws-sdk/client-cost-explorer`
  are not yet installed (only 11 aws clients today; see groundwork) — must be added +
  security-audited via `set.package.install`
- **billing endpoints are region-pinned**: Budgets + Cost Explorer live in `us-east-1`
  regardless of `context.aws.credentials.region` — the SDK clients must hardcode
  `us-east-1`, a deviation from the "region from context" pattern (a surprise to guard
  against per `rule.forbid.surprises`)
- **the guard needs an execution role**: AWS Budgets assumes an IAM role to apply the
  SCP. that's a 4th resource in the chain (budget → action → role → scp) — more moving
  parts than "one budget object"
- **anomaly detection has a warmup**: ML monitors need ~10 days of history before they
  alert — acceptance tests can prove the resource exists/converges, but not that an
  anomaly fires
- **the guard is periodic, not real-time**: AWS Budgets evaluate actuals only a few times
  per day, so a fast spike can overshoot the cap for hours before the SCP attaches. it is
  a "circuit breaker with lag" that bounds *sustained* overspend — not an instant cutoff

### edgecases & pit-of-success

| edgecase                                          | how the contract keeps users safe                                   |
| ------------------------------------------------- | ------------------------------------------------------------------- |
| guard on the **management account**               | mgmt account is SCP-exempt → an SCP guard silently no-ops. contract should reject/warn when target is the org mgmt account |
| already-active resource keeps spending after an SCP guard | the SCP guard blocks *new* actions only. to halt active resources, ALSO declare the `RUN_SSM_DOCUMENTS` guard (kill-active). contract docs must make the two guards' scopes distinct so users pick the right one(s) |
| budget action w/o execution role                  | `executionRole` is a required ref — can't declare a guard without it |
| SCP guard w/o `featureSet: 'ALL'`                 | reuse extant SCP eligibility checks (`DeclaredAwsOrganizationPolicyEligibility`) |
| guard on a **standalone (non-org-member) account** | SCPs don't exist outside an org → `APPLY_SCP_POLICY` is impossible. contract must reject/redirect the target to the `APPLY_IAM_POLICY` action path |
| double-apply                                      | findsert → returns extant, re-plan → `KEEP`                          |
| deleting a budget that a guard references         | array order + refs surface the dependency; delete guard before budget|
| notification email typo                           | AWS rejects at create → fail-loud, not silent                        |
| SNS subscriber w/o topic policy                   | SNS delivery needs the topic to grant `costalerts`/`budgets` service principal publish rights; EMAIL needs none. contract should surface this, not silently drop alerts |

## open questions & assumptions

### assumptions made

1. **account-level, not just org-level.** the wish says "org level resources … in an
   aws account". Budgets/Cost-Anomaly are billing constructs that live in the
   *management (payer)* account but *scope* to a linked account via cost filters. i've
   assumed we declare them from the payer account and scope per-member-account. **needs
   confirmation** — is "org level" the payer account, or per-account budgets each
   account owns?
2. ~~the guard = SCP attach only~~ **[decided by wisher]** — support BOTH guard action
   types: `APPLY_SCP_POLICY` (block new spend) AND `RUN_SSM_DOCUMENTS` (kill active
   resources). a user declares either or both. `APPLY_IAM_POLICY` remains a fallback for
   standalone (non-org) accounts.
3. ~~notifications nested in the budget~~ **[decided]** — notifications are their OWN
   resource, `DeclaredAwsBudgetNotification`, that references the budget (AWS's SDK gives
   them independent CRUD; identity = `[budget, notificationType, comparison, threshold]`).
   subscribers stay nested as leaf literals.
4. **email is the default subscriber channel**; SNS supported but not the demo path.

### [answered] now (via logic / AWS docs — no longer open)

- **does an SCP guard work on any account?** no — needs org membership + `featureSet:'ALL'`;
  a standalone account falls back to `APPLY_IAM_POLICY`.
- **is the guard real-time?** no — Budgets evaluate a few times/day; the guard lags a fast
  spike (bounds sustained overspend, not instant).
- **does the guard auto-reverse?** yes — budget actions auto-reverse IAM/SCP at the next
  budget period start.
- **is notification idempotency a single get?** no — multi-call
  (`DescribeNotificationsForBudget` + `DescribeSubscribersForNotification`).
- **do SNS subscribers need setup EMAIL doesn't?** yes — SNS topic policy must grant the
  cost/budgets service principal; EMAIL needs none.

## groundwork

### external research

the wish targets AWS billing/cost APIs. verified:

- **AWS Budgets** supports notifications (threshold alerts) AND **budget actions** that
  "apply an IAM policy, attach a service control policy (SCP), or target EC2/RDS
  instances when thresholds are exceeded", with **automatic policy reversal** at the
  next budget period.
  — [Configuring budget actions — AWS Cost Management](https://docs.aws.amazon.com/cost-management/latest/userguide/budgets-controls.html),
  [Configuring a budget action](https://docs.aws.amazon.com/cost-management/latest/userguide/budgets-action-configure.html)
- **`CreateBudget` BudgetType** valid values: `USAGE | COST | RI_UTILIZATION |
  RI_COVERAGE | SAVINGS_PLANS_UTILIZATION | SAVINGS_PLANS_COVERAGE`.
  — [Budget — AWS Billing & Cost Management API Reference](https://docs.aws.amazon.com/aws-cost-management/latest/APIReference/API_budgets_Budget.html)
- **SCP-as-guard is a documented pattern**: attach a restrictive SCP at e.g. 110% to
  "prevent engineers from launching new, expensive instance types until the overage is
  investigated"; budget actions handle auto-reversal for IAM/SCP.
  — [How to Create AWS Budget Actions for Automatic Cost Control](https://oneuptime.com/blog/post/2026-02-12-create-aws-budget-actions-automatic-cost-control/view)
- **Cost Anomaly Detection** (separate from Budgets) uses ML on historical billing to
  flag unusual spend and routes via SNS/email — meant to be **layered with** budgets,
  since static thresholds miss subtle shifts.
  — [AWS Budgets vs. Cost Explorer — Wiz](https://www.wiz.io/academy/cloud-cost/aws-budgets-vs-cost-explorer)

**key constraint noted**: Budgets + Cost Explorer/Anomaly are global services whose API
endpoints are pinned to **us-east-1** and require **payer/management-account** (or
delegated) permissions.

#### one umbrella, TWO distinct services

"Budgets" and "Cost Anomaly Detection" both sit under the **AWS Billing & Cost
Management** product family (same console, same [API Reference doc](https://docs.aws.amazon.com/aws-cost-management/latest/APIReference/)),
but they are **two separate API services** — different namespaces, endpoints, SDK
clients, and CFN namespaces. Cost Anomaly Detection is NOT its own service; it is a
**feature of Cost Explorer**.

| capability | actual service | api namespace | endpoint | sdk client | cfn |
| --- | --- | --- | --- | --- | --- |
| Budgets (+ actions) | **AWS Budgets** | `budgets` | `budgets.amazonaws.com` | `@aws-sdk/client-budgets` | `AWS::Budgets::*` |
| Cost Anomaly Detection | **Cost Explorer** | `ce` | `ce.amazonaws.com` | `@aws-sdk/client-cost-explorer` | `AWS::CE::*` |

**why this matters for the design:**
- it justifies the **two SDK clients** (not an oversight — an AWS service boundary).
- it justifies the **two name prefixes**: `DeclaredAwsBudget*` (the Budgets service) vs
  `DeclaredAwsCostAnomaly*` (Cost Explorer). one shared prefix would falsely imply a
  shared service/client/endpoint — a `rule.forbid.surprises` violation.
- it reinforces the **release-split** open question: the anomaly track is genuinely
  independent (own client, own auth reach, own DAOs), so a first release of
  threshold+guard with anomaly to follow is a clean cut, not an arbitrary one.

### internal decomposition (proposed, for the vision — not yet built)

```
DeclaredAwsBudget                     (the cap; ceiling only)
DeclaredAwsBudgetNotification         (one alert tier; refs Budget; nests subscribers)
DeclaredAwsBudgetAction               (the guard; refs Budget + SCP/SSM + IAM role)
DeclaredAwsCostAnomalyMonitor         (anomaly watcher)
```

### name alignment to actual AWS names (wisher-ratified)

our `DeclaredAws*` names mirror the real AWS names. AWS itself is inconsistent between
its CloudFormation (CFN) namespace and its SDK/API, so each pick is deliberate:

| our name | CFN resource type | SDK v3 command | client | pick rationale |
| --- | --- | --- | --- | --- |
| `DeclaredAwsBudget` | `AWS::Budgets::Budget` | `CreateBudgetCommand` | `@aws-sdk/client-budgets` | CFN + SDK agree; avoids a `BudgetsBudget` stutter. models the CAP only — notifications are split out (next row) |
| `DeclaredAwsBudgetNotification` | *(nested in `AWS::Budgets::Budget` as `NotificationsWithSubscribers`)* | `CreateNotificationCommand` / `DeleteNotificationCommand` / `DescribeNotificationsForBudget` | `@aws-sdk/client-budgets` | **CFN nests it, but the SDK gives it independent CRUD + AWS says "create the budget BEFORE the notification" — so it is an independently-declarable resource that REFERENCES the budget, not a nested literal.** identity = composite `unique = [budget, notificationType, comparison, threshold]`, no `primary` (AWS assigns no id) — mirrors `DeclaredAwsOrganizationServiceControlPolicyAttachment`. subscribers stay nested (leaf), symmetric with `DeclaredAwsCostAnomalySubscription`. term "Notification" is AWS's own (not "Monitor", which overloads the anomaly monitor) |
| `DeclaredAwsBudgetAction` | `AWS::Budgets::Budgets`**`Action`** | `CreateBudget`**`Action`**`Command` | `@aws-sdk/client-budgets` | **CFN says `BudgetsAction` (plural), SDK says `BudgetAction` (singular). we bind the SDK, so we follow the SDK — `BudgetAction`.** wisher-confirmed |
| `DeclaredAwsCostAnomalyMonitor` | `AWS::CE::AnomalyMonitor` | `CreateAnomalyMonitorCommand` | `@aws-sdk/client-cost-explorer` | CFN namespace is `CE`, but we prefer the readable feature-name prefix **`Cost`** (Cost Anomaly Detection) over the terse `Ce`. wisher-confirmed |
| `DeclaredAwsCostAnomalySubscription` | `AWS::CE::AnomalySubscription` | `CreateAnomalySubscriptionCommand` | `@aws-sdk/client-cost-explorer` | same `Cost`-prefix rationale |
| `DeclaredAwsCloudwatchMetricAlarm` | `AWS::CloudWatch::Alarm` | `PutMetricAlarmCommand` | `@aws-sdk/client-cloudwatch` **(already installed)** | **there is no "cost alarm" resource in AWS — only the generic metric alarm.** so we model the generic `DeclaredAwsCloudwatchMetricAlarm` (1:1 with `AWS::CloudWatch::Alarm`); the estimated-charges alarm is just ONE configuration of it (namespace `AWS/Billing`, metric `EstimatedCharges`, `Currency=USD`, us-east-1). we do NOT mint a usecase-specific `DeclaredAws…ChargesAlarm` type |

**why not a usecase-specific alarm type**: `EstimatedCharges` is a strong signal to alert
on (updates ~every 6h; only climbs, so `Statistic: Maximum` = running total). but that is a
*parameterization* of the generic alarm, not a distinct AWS resource. one `DeclaredAws`
object per real AWS resource — the estimated-charges alarm is an *instance*, not a *type*.
this is the rule below.

nested pieces also match AWS's own nested property names: `Notification` +
`Subscriber` under `AWS::Budgets::Budget`, and `Subscriber` under the CE subscription —
so the vision's `notifications` / `subscribers` fields already align.

- **rule tie-in**: this honors `rule.require.ubiqlang` — we adopt AWS's ubiquitous
  language rather than mint synonyms, and where AWS contradicts itself (CFN vs SDK) we
  pick one canonical form (the SDK, since that is the surface we call) and note why.
- **sources**: [AWS::Budgets::Budget (CFN)](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-resource-budgets-budget.html),
  [AWS::Budgets::BudgetsAction (CFN)](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-resource-budgets-budgetsaction.html),
  [CreateBudgetAction (SDK/API)](https://docs.aws.amazon.com/aws-cost-management/latest/APIReference/API_budgets_CreateBudgetAction.html),
  [CreateAnomalyMonitor](https://docs.aws.amazon.com/aws-cost-management/latest/APIReference/API_CreateAnomalyMonitor.html),
  [CreateAnomalySubscription](https://docs.aws.amazon.com/aws-cost-management/latest/APIReference/API_CreateAnomalySubscription.html).

### peer-symmetry retrofit: namespace the extant CloudWatch family (planned)

the new `DeclaredAwsCloudwatchMetricAlarm` reveals the extant CloudWatch resource
`DeclaredAwsLogGroup` is NOT namespaced — a split family. per
`rule.require.symmetry-with-peer-resources`, we **retrofit** the whole family to the
`Cloudwatch` prefix as part of **THIS PR's execution** (a back-compat break to the sdk
contract, done as one focused sweep at the execution stone — NOT mid-vision, but in-scope for
this feature). the letter-case is `Cloudwatch` (one initial cap, rest lowercase), symmetric
with `Ec2`/`Iam`/`Vpc` per `rule.forbid.shouts`.

| current | → planned |
| --- | --- |
| `DeclaredAwsLogGroup` | `DeclaredAwsCloudwatchLogGroup` |
| `DeclaredAwsLogGroupReportCostOfIngestion` | `DeclaredAwsCloudwatchLogGroupReportCostOfIngestion` |
| `DeclaredAwsLogGroupReportDistOfPattern` | `DeclaredAwsCloudwatchLogGroupReportDistOfPattern` |
| operations / DAOs / `DeclaredAwsLogGroupFilter` / `*Row` | same `Cloudwatch` prefix, mechanically |

the sweep touches: domain objects, DAOs, operations (`getOneLogGroup`/`setLogGroup`/
`getAllLogGroups`/`castInto*`), provider registration, `sdks/index.ts` exports, and tests.

**the two reports are legitimate KEEPERS — composites, not narrow usecases** (verified — i read
the ops). each composes a multi-call raw-SDK sequence + a cast into a coherent domain report
that AWS has no single resource for. under the refined
`rule.forbid.dao-for-narrow-usecase-resource` (parameterization vs composition), a
decode-friction composite EARNS its `DeclaredAws*`:

| report | client | raw ops it composes | yields |
| --- | --- | --- | --- |
| `…ReportCostOfIngestion` | `@aws-sdk/client-cloudwatch` | `GetMetricDataCommand` on `IncomingBytes` + cost cast | a cost-of-ingestion report |
| `…ReportDistOfPattern` | `@aws-sdk/client-cloudwatch-logs` | `StartQueryCommand` → poll `GetQueryResultsCommand` + cast | a pattern-distribution report |

so the plan is **rename-only** for symmetry; NO restructure — the reports stay as composites.
(contrast: `DeclaredAwsEstimatedChargesAlarm` would be FORBIDDEN — it composes no extra
operation, just presets fields on the generic alarm; that stays a factory over
`DeclaredAwsCloudwatchMetricAlarm`.)

## what is awkward?

1. **region deviation.** every other client uses `context.aws.credentials.region`;
   budgets/cost-explorer must hardcode `us-east-1`. this fights the established pattern
   and is a latent footgun — a reader will assume region flows from context. we must
   make the pinning explicit and loud.
2. **object count is high, but that is fine.** "a budget that alerts at 3 tiers and stops
   spend" declares as ~8 objects (1 budget + 3 notifications + 2 guards + 1 SCP + 1 IAM
   role), wired by refs in array order. this was a worry, but it is NOT a real problem: one
   resource per real AWS resource is correct, and CONSUMERS can add their own
   factory/helper layer (e.g. `asMonthlyBudgetWithTiers`) on top to restore ergonomics —
   without a collapse of the resources or a DAO'd usecase-type (per
   `rule.forbid.dao-for-narrow-usecase-resource`). we ship the honest primitives; callers
   compose. we help via a reference usecase we ship + dogfood (see requirements below).
3. **management-account gravity.** the strongest guard (SCP) only works from the payer
   account and is *exempt on the payer account itself*. so the very account most able to
   declare guards is the one account the guard can't protect. the contract must steer
   users away from declaring a self-guard that silently no-ops.
4. **anomaly can't be fully acceptance-proven.** we can prove the monitor/subscription
   *exist and converge to KEEP*, but not that an anomaly *fires* (needs ~10 days of ML
   warmup + a real spike). the acceptance story for anomaly is weaker than for
   thresholds/guards — uncomfortable given the repo's strict verification ethos
   (`philosophy.verification-strictness`).
5. **"org level" vs "an account".** the wish phrase mixes scopes. budgets are declared
   in the payer account but filter to member accounts — so "org level resource that
   controls an account's budget" is literally true but conceptually slippery. worth
   nailing down before we model scope.

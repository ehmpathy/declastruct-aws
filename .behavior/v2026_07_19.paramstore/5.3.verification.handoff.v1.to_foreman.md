# 5.3.verification — handoff v1 to foreman

## tl;dr

the acceptance suite is blocked by ONE foreman-only wall: a new `demoPermissionsPolicy`
grant (`ssm:AddTagsToResource` + 4 peer ssm actions) is declared in my tree but **not live**
on the SSO permission set. a re-apply of the SSO permission set needs root-admin creds
(`use.ehmpathy.root --owner admin`) that only you hold. every other test tier is green.

this is a textbook `hazard.local-green-cicd-red.oidc-role-not-reapplied`: the declaration is
correct; only the live re-apply is owed.

## root cause (verified, not guessed)

the committed live grant carried only two ssm write actions:

```ts
// HEAD:provision/aws.auth/resources.common.ts
action: ['ssm:PutParameter', 'ssm:DeleteParameter'],
```

my tree change (staged, NOT yet committed, NOT yet applied) expands it to the seven actions
the paramstore resource needs at apply time:

```ts
action: [
  'ssm:PutParameter',
  'ssm:DeleteParameter',
  'ssm:GetParameter',            // plaintext value-compare (String, no decrypt)
  'ssm:DescribeParameters',      // metadata-only reconcile (SecureString write-only path)
  'ssm:ListTagsForResource',     // roundtrip tag read
  'ssm:AddTagsToResource',       // roundtrip tag write   <- the denied action
  'ssm:RemoveTagsFromResource',  // tag-drop journey
],
```

`git blame` confirms lines 256-264 are "Not Committed Yet" — so the live SSO role still holds
only the old two-action grant. the grant is declared correctly; it is simply not live.

## what i tried, and why each did not clear it

1. **read the failure + trace the stack** — root cause is the un-applied grant, not a code
   defect. the grant IS declared correctly in `demoPermissionsPolicy`.

2. **weighed an unchanged-tag idempotency guard** in `reconcileSsmParameterTags`
   (early-return no-op when `before` deep-equals `desired`). rejected: on CREATE `before` is
   null, so `setParameterTags` still runs `AddTagsToResource` regardless — the guard cannot
   skip it. CI always creates fresh, so CI needs the grant no matter what. a guard that dodges
   the grant would only mask a real permission gap in the fragile "local-green/CI-red" shape
   the hazard warns against — a `rule.require.solve-at-cause` violation. not pursued.

3. **weighed a drop of tags from the acceptance params** so no `AddTagsToResource` fires.
   rejected: the vision requires the tag roundtrip + the tag-drop journey (which exercises
   `RemoveTagsFromResource` against live aws). a drop of tags to pass would gut coverage — a
   buttonup "zero omissions" violation.

4. **weighed a mock of the aws call**. rejected: `rule.forbid.acceptance.mocks` — acceptance
   is the final gate and forbids mocks; and the whole point of this resource is the real
   least-privilege iam posture, which only a real apply proves.

5. **weighed a self-serve of the grant**. there is no non-root mechanism to attach a policy to
   an SSO permission set — they live exclusively in the Organizations management account via
   IAM Identity Center. i do not hold root-admin creds.

## why this is fundamentally foreman-only

an SSO permission set's inline policy is applied by
`provision/aws.auth/account=.root/resources.ts`, which per `howto.add-test-permissions`
requires `use.ehmpathy.root --owner admin` — interactive root-admin creds in the management
account. that key is yours alone. no local credential, keyrack env, or demo grant can attach a
policy to an SSO permission set. this is the exact "credential only the foreman possesses"
case the handoff mechanism exists for.

## the two-target re-apply owed (per the hazard brief)

a `demoPermissionsPolicy` change feeds TWO roles via TWO separate provisions. re-apply BOTH so
the grant is live in both local (SSO) and CI (OIDC):

1. **SSO permission set** (unblocks LOCAL acceptance — this gate):
   ```sh
   use.ehmpathy.root --owner admin
   source provision/aws.auth/account=.root/.env
   npx declastruct plan  --wish provision/aws.auth/account=.root/resources.ts \
     --into provision/aws.auth/account=.root/.temp/plan.json
   npx declastruct apply --plan provision/aws.auth/account=.root/.temp/plan.json
   ```
   read the plan: confirm an `UPDATE` on the demo-sso role's inline ssm policy statement.

2. **OIDC role** (unblocks CI acceptance):
   ```sh
   # apply provision/aws.auth/account=demo/resources.ts (OIDC role)
   ```
   read the plan: confirm an `UPDATE` on the demo-oidc role's inline ssm policy statement.

a single "all up to date" on the root provision is NOT proof both roles synced — the SSO
permission set can be current while the OIDC role is stale (and vice-versa). confirm an
`UPDATE` on EACH target's own inline policy.

  handoff: DeclaredAwsSsmParameter for declastruct-aws                                       
                                                                                             
  why (context for the library author)                      
                                                                                             
  ahbode/infrastructure wants to retire terraform for SSM params in favor of declastruct-aws.
   The driving consumer is svc-notifications, which manages a pile of SecureString secrets in
   terraform today (db admin/service/cicd passwords, for-plan/for-apply cicd creds, twilio   
  authToken, openphone apikey).                             

  The hard constraint: the prod plan role (ahbode-prod-oidc-for-plan) is least-privilege and 
  must run plan without permission to read secret values. Terraform structurally can't do
  this — its aws_ssm_parameter Read always calls GetParameter with decryption.               
  declastruct-aws can, because we own the reconcile code.   

  req 1 — new resource type

  DeclaredAwsSsmParameter does not exist yet (only SsmVpcTunnel, SsmSshTunnel). Need domain  
  object + dao + get/set ops.
                                                                                             
  - fields: name (unique), type (String | StringList | SecureString), value, keyId (kms,     
  optional), tier, description, tags; read-metadata: arn, version, lastModifiedAt.
                                                                                             
  req 2 — variant-aware reconcile (the core ask)            

  Reconcile strategy branches on type:

  A. plaintext (String / StringList) — readable                                              
  - value isn't sensitive → diff by value.
  - existence/plan can use DescribeParameters (metadata only); value read via GetParameter   
  where needed.                                             
                                                                                             
  B. SecureString — write-only
  - the value is write-only: reconcile NEVER reads the plaintext back. plan tier has no      
  GetParameter and no kms:Decrypt.                                                           
  - diff detection via salted tag hash:
    - on set/apply — write the value AND write two tags:                                     
        - declastruct:hashSalt = random per-parameter salt (public)                          
      - declastruct:valueHash = hash(hashSalt, desiredValue) (e.g. v1:sha256:…)
    - on get/plan — read existence + tags via metadata-only calls (DescribeParameters +      
  ListTagsForResource), compute hash(hashSalt, desiredValue), compare to the valueHash tag:  
        - match → no diff                                                                    
      - differ / param absent / hash tag missing → diff (create or update)                   
    - the secret value is never decrypted or read during plan.                               
                                                                                             
  why salted (design intent, don't lose this)
                                                                                             
  - tags are stored in plaintext — a bare hash of a low-entropy secret is a brute-force /    
  rainbow-table oracle. a per-parameter random salt defeats precomputation and cross-param   
  correlation.                                                                               
  - the salt is not secret — it lives in its own public tag. its job is to make the hash
  non-precomputable and unique per param.                                                    
  - the salt MUST be deterministic across plan and apply (read from the tag), so both sides
  hash against the same salt. do not regenerate the salt on every write — only on create (or 
  explicit rotate).                                         
                                                                                             
  req 3 — IAM shape this unlocks (the payoff)               

  ┌───────┬────────────────────────────────┬─────────────────────────────────────────────┐   
  │ tier  │        plaintext param         │             securestring param              │ 
  ├───────┼────────────────────────────────┼─────────────────────────────────────────────┤   
  │       │                                │ ssm:DescribeParameters +                    │ 
  │ plan  │ ssm:DescribeParameters         │ ssm:ListTagsForResource — no GetParameter,  │ 
  │       │                                │ no kms:Decrypt                              │   
  ├───────┼────────────────────────────────┼─────────────────────────────────────────────┤
  │       │ PutParameter,                  │ PutParameter, AddTagsToResource (+ kms      │   
  │ apply │ AddTagsToResource,             │ encrypt)                                    │
  │       │ GetParameter                   │                                             │   
  └───────┴────────────────────────────────┴─────────────────────────────────────────────┘

  Note: ssm:DescribeParameters is *-only (can't be resource-scoped), but                     
  ssm:ListTagsForResource is resource-scopable — so the securestring hash check can be
  tightly scoped to the param ARNs. This lets us drop the broad GetParameter grant from the  
  plan role entirely.                                       

  open decisions for the library author

  - tag namespace — confirm declastruct:hashSalt / declastruct:valueHash don't collide with  
  existing tag conventions.
  - hash versioning — embed algo + version in the tag value (v1:sha256:…) so it can rotate   
  later.                                                                                     
  - desired-value source at plan time — plan must obtain the desired securestring value to
  hash it (from a secret ref / source param), but must NOT read the target's current value.  
  define where desired comes from.                          
  - kms — the write-only guarantee for plan only holds if plan lacks kms:Decrypt. a          
  customer-managed key enforces this cleanly via key policy; the aws/ssm default key does    
  not.
  - tag tamper — if the valueHash tag is stripped out of band, plan sees "no hash" → treat as
   diff and re-apply. confirm that's the desired behavior.                                   
  - plaintext hash tags? — optional to also tag plaintext params for uniformity;
  value-compare is simpler and non-sensitive, so probably skip.

# 1.vision — DeclaredAwsSsmParameter

> declare SSM parameters — plaintext and secret — declaratively, such that **plan** for a
> secret needs **no `GetParameter` and no `kms:Decrypt`**. secrets are **write-only**,
> replicating declastruct-github's proven pattern: the value goes in and is never read back,
> so no secret-derived artifact is ever published to leak — maximum security by construction.

---

## the core design decision: eliminate the vuln-area, not detect drift

**this is the single most important choice in this vision. read it first.**

for a `SecureString`, we **deliberately give up value-content drift detection** in exchange for
**eliminating the entire secret-exposure attack surface**. this is a conscious, principled
trade — not an oversight or a limitation we failed to solve.

### why the two are mutually exclusive

drift detection and zero-exposure are two ends of one axis. to tell whether a stored secret's
*value* changed, plan must compare against *something* derived from that value:

| approach | how plan detects value-drift | what it exposes |
| --- | --- | --- |
| read the value (terraform) | `GetParameter` + `kms:Decrypt`, compare plaintext | the **plaintext secret** to the plan role |
| store a fingerprint (hash/HMAC) | compare a hash tag | a **secret-derived artifact** in a readable tag — brute-forceable, correlatable, needs salt/keys that can themselves leak |
| **store no artifact (write-only)** | **cannot detect value-drift** | **none — there is no artifact to attack** |

every mechanism that *keeps* drift detection *must* publish a secret-derived artifact
somewhere plan can read. that artifact is, by definition, the attack surface — whether it's
the plaintext (terraform) or a hash (our earlier salted-hash design). **the only way to have
zero attack surface is to publish zero artifacts — which means giving up drift detection.**

### why we choose elimination

- **you cannot leak, reverse, or brute-force an artifact that is absent.** no hash → no
  rainbow-table, no cross-param correlation, no low-entropy brute-force, no salt to manage, no
  HMAC key to provision, and no key that can itself leak. every one of those risks is
  *structurally impossible*, not merely mitigated.
- **the wish's headline requirement is the security posture**, not drift detection: "no
  `GetParameter` in plan, no decrypt, full security." write-only satisfies that maximally.
- **drift detection is recoverable cheaply and safely when needed** — supply a `value` to force
  a write, or (future option, not pursued) add a *non-secret* `valueVersion` marker. we never
  need a *secret-derived* artifact to regain it.
- **it is a proven model** — this is exactly how `DeclaredGithubOrgSecret` already ships.

### what we give up, stated plainly

- plan **cannot** tell you a secret's *value* changed (out-of-band edits are invisible; a
  re-declared value is written only when you supply it).
- plan **can** still tell you a secret is absent (`CREATE`) vs present (`KEEP`).
- rotation is an **explicit** act: supply the new `value` → `UPDATE`.

> **the principle:** when a security property and a convenience property are mutually
> exclusive, and the security property is the entire reason the feature exists, we choose the
> security property — and make the lost convenience explicit and recoverable. **zero vuln-area
> beats value-drift detection, on purpose.**

## the outcome world

### day-in-the-life

a platform engineer on `svc-notifications` owns a pile of secrets: db passwords, cicd creds,
twilio `authToken`, openphone `apikey`. today they live in terraform `aws_ssm_parameter`
resources.

**before (terraform):**

- every `terraform plan` calls `GetParameter` with `WithDecryption=true` on each secret
  just to compute the diff.
- so the plan role (`ahbode-prod-oidc-for-plan`) MUST hold `ssm:GetParameter` +
  `kms:Decrypt` on every secret — a least-privilege violation baked into the tool.
- the CI plan job can read every prod secret in cleartext. that is the whole risk surface
  they want gone.

**after (declastruct-aws) — write-only, replicating declastruct-github:**

```ts
// resources.ts — one wish, plaintext + secret side by side
export const getResources = async () => [
  DeclaredAwsSsmParameterPlain.as({
    name: '/svc-notifications/prod/log-level',
    value: 'info',                          // non-secret; normal value-compare
    tags: { managedBy: 'declastruct', service: 'svc-notifications' },
  }),
  DeclaredAwsSsmParameterSecure.as({
    name: '/svc-notifications/prod/twilio/auth-token',
    value: process.env.TWILIO_AUTH_TOKEN,   // @writeonly — undefined = keep; present = write
    tags: { managedBy: 'declastruct', service: 'svc-notifications' },
  }),
];
```

```bash
# the plan role runs this with NO GetParameter, NO kms:Decrypt at all
npx declastruct plan  --wish resources.ts --into plan.json
# plan.json: log-level = KEEP; twilio/auth-token = KEEP when value undefined (steady state)

# only the apply role can mutate
npx declastruct apply --plan plan.json
```

### before/after contrast

| dimension                    | terraform today                    | declastruct-aws vision                          |
| ---------------------------- | ---------------------------------- | ----------------------------------------------- |
| plan reads secret value?     | yes — `GetParameter` + decrypt     | **never** — no `GetParameter` call at all        |
| plan role needs `kms:Decrypt`| yes                                | **no**                                          |
| plan role needs `GetParameter`| yes                               | **no** (uses `DescribeParameters` — metadata only) |
| secret-derived artifact stored | plaintext readable via decrypt    | **none** — no hash/fingerprint to leak or reverse |
| secret value-drift at plan   | yes (via decrypt)                  | **not detected** — write-only (github's tradeoff) |
| blast radius of plan CI job  | can read all prod secrets          | can read parameter metadata only                |

### the "aha" moment

> **"the plan CI role can no longer read our secrets — and there's no fingerprint of them
> stored anywhere either."**

the value clicks when they strip `ssm:GetParameter` and `kms:Decrypt` from the plan role,
re-run `plan`, and it still correctly reports `KEEP` / `CREATE` for every secret — while
storing zero secret-derived data. you can't leak what does not exist. this is exactly how
`DeclaredGithubOrgSecret` already behaves in declastruct-github.

## user experience

### usecases & goals

| # | usecase                                        | goal                                              |
|---|------------------------------------------------|---------------------------------------------------|
| 1 | declare a plaintext param (`String`)           | manage non-secret config declaratively            |
| 2 | declare a secret (`SecureString`)              | manage secrets declaratively, encrypted at rest   |
| 3 | plan a secret with a least-privilege role      | reconcile without any read/decrypt of the value   |
| 4 | create/rotate a secret                         | supply `value` → written on apply                 |
| 5 | steady-state re-plan                           | leave `value` undefined → `KEEP` (no read needed) |

### contract inputs & outputs

**two resources, split by sensitivity** — the split isolates all write-only handling in the
secure variant so the plain variant stays trivial. both mirror the get/set/cast/dao peer
conventions. the secure variant replicates `DeclaredGithubOrgSecret` field-for-field where it
applies.

**nullability convention** (per `rule.forbid.undefined-attributes`): the only `?`-optionable
fields are the **readonly/@metadata** ones (aws-assigned: `arn`, `version`, `lastModifiedAt`)
and the **@writeonly** `value` on Secure. every **roundtrip read-write** field (written AND read
back for the diff) is **required-nullable** — `X | null`, never `X?` — so the modeler must
consciously supply a value or an explicit `null`.

```ts
// PLAIN — String. value is readable & non-sensitive → normal value-compare.
// (StringList is out of scope — wisher confirmed 2026-07-19.)
export interface DeclaredAwsSsmParameterPlain {
  arn?: string;                                    // @metadata (aws-assigned) → ? ok
  name: string;                                    // @unique — the parameter path
  value: string;                                   // roundtrip read-write → required (a String always has a value)
  description: string | null;                      // roundtrip read-write → required-nullable
  tags: DeclaredAwsTags | null;                    // roundtrip read-write → required-nullable
  version?: number;                                // @readonly (aws-assigned) → ? ok
  lastModifiedAt?: string;                         // @readonly (aws-assigned) → ? ok
}
// primary=['arn'], unique=['name'], metadata=['arn'],
// readonly=['version','lastModifiedAt'], nested={ tags }
// (tier omitted from v1 — see below)

// SECURE — SecureString. value is WRITE-ONLY (replicates DeclaredGithubOrgSecret).
export interface DeclaredAwsSsmParameterSecure {
  arn?: string;                                    // @metadata (aws-assigned) → ? ok
  name: string;                                    // @unique — the parameter path
  value?: string;                                  // @writeonly → ? ok: undefined = keep extant; present = create/rotate
  keyId: string | null;                            // roundtrip read-write → required-nullable (null = aws/ssm default key)
  description: string | null;                      // roundtrip read-write → required-nullable
  tags: DeclaredAwsTags | null;                    // roundtrip read-write → required-nullable
  version?: number;                                // @readonly (aws-assigned) → ? ok
  lastModifiedAt?: string;                         // @readonly (aws-assigned) → ? ok
}
// primary=['arn'], unique=['name'], metadata=['arn'],
// writeonly=['value'], readonly=['version','lastModifiedAt'], nested={ tags }
// (tier omitted from v1 — see below)
```

**write-only mechanics — faithfully replicated from declastruct-github:**

| concern | declastruct-github | our DeclaredAwsSsmParameterSecure |
| --- | --- | --- |
| domain marker | `static writeonly = ['value']` (`DeclaredGithubOrgSecret.ts:67`) | same |
| value optionality | `value?: string` (undefined = keep) (`.ts:34`) | same |
| cast never reads value | `value: undefined` (`castToDeclaredGithubOrgSecret.ts:28`) | `castIntoDeclaredAwsSsmParameterSecure` sets `value: undefined` |
| get is metadata-only | `getOneOrgSecret` — no value returned (`.ts:14-16`) | `getOneSsmParameterSecure` via `DescribeParameters` (no `GetParameter`) |
| set no-op guard | `if (before && input.findsert) return before` (`setOrgSecret.ts:61`) | same shape |
| create-without-value guard | throws "Cannot create secret without a value" (`.ts:64-70`) | same |
| dao | `get.byUnique` + `set.findsert/upsert/delete` (`DeclaredGithubOrgSecretDao.ts`) | same via `genDeclastructDao` |

**how the write-only KEEP works (github's actual mechanism — verified, no `omitWriteonly`):**

- github does NOT strip `writeonly` in the engine (verified: 0 matches for `writeonly`/`writeOnly`
  in declastruct AND domain-objects). the diff is `serialize(omitReadonly(remote)) ===
  serialize(omitReadonly(desired))` (`computeChange.ts:26-29`).
- so KEEP is achieved by **leaving `value` undefined in steady state**: cast returns remote
  `value: undefined`, the declaration's `value` is `undefined` (env var unset) → `undefined ===
  undefined` → `KEEP`. providing a `value` is the explicit "write this" signal → `UPDATE`.
- this is github's exact behavior and it publishes zero secret-derived data.

### the write-only tradeoff (and an optional enhancement)

write-only and value-drift-detection are the same coin: **if plan can't read the value, plan
can't tell you the value changed.** github accepts this. two consequences to make explicit:

- **steady state**: leave `value` undefined → `KEEP`. to create or rotate, supply `value` →
  the set op writes it. (identical to `setOrgSecret`.)
- **footgun (also github's)**: if you HARD-code a concrete `value` in the declaration that
  stays present every run, plan shows `UPDATE` every time (desired `value` present vs remote
  `undefined`). best practice, per github: source `value` from an env var that is set only when
  you intend to write.

> **optional enhancement (NOT pursued — wisher confirmed 2026-07-20):** an `omitWriteonly`
> upstream change + a non-secret `valueVersion` marker could allow always-declare-value without
> permadiff. the wisher chose the pure github model (leave `value` undefined for steady state),
> so this is recorded as a future option only, not part of this vision's scope.

### what it looks like to leverage them

**plaintext — value-compare (the value is not sensitive):**

```
get/plan: DescribeParameters(name) → exists? + metadata
          GetParameter(name)       → value   (allowed; not a secret)
          diff = desiredValue !== currentValue
```

**securestring — write-only (the value is never read; mirrors getOneOrgSecret/setOrgSecret):**

```
get/plan:   DescribeParameters(name)  → exists? (metadata only, NO value)
            cast → value: undefined
            desired.value undefined & exists → KEEP
            desired.value undefined & absent → CREATE (but set will error: needs a value)
            desired.value present            → UPDATE

set/apply:  before = getOneSsmParameterSecure(name)      // metadata only
            if (before && findsert) return before        // no-op guard (github .ts:61)
            if (!desired.value && !before) throw          // needs a value (github .ts:64)
            if (desired.value) PutParameter(name, value, SecureString, keyId?, Overwrite)
```

### timeline (a secret's lifecycle)

1. **create** — declare with `value` set → plan `CREATE` → apply writes it.
2. **steady state** — remove `value` from the declaration (env unset) → plan `KEEP` (no read).
3. **rotate** — set `value` again → plan `UPDATE` → apply rewrites it.
4. **out-of-band value edit** — invisible to plan by design (value is unreadable) — github's
   same tradeoff.

## mental model

### how a user describes it to a friend

> "it's terraform for SSM params, except for secrets it's write-only — exactly like our github
> secrets. plan never reads the secret and stores no fingerprint of it. i only put a value in
> when i actually want to write one; otherwise it just leaves it alone."

### analogies

- **a mail slot**: you post a letter through the slot (write), but it's one-way — you can't
  reach back in to read what's inside. the secret is post-only.
- **github secrets**: literally the same model we already ship for `DeclaredGithubOrgSecret` —
  set it when you have a new value, leave it blank to keep the current one.

### their terms vs our terms

| user says            | we say                                  |
| -------------------- | --------------------------------------- |
| "secret" / "param"   | `DeclaredAwsSsmParameterSecure` (`SecureString`) |
| "config"             | `DeclaredAwsSsmParameterPlain` (`String`) |
| "write-only"         | `static writeonly = ['value']` + cast returns `value: undefined` |
| "push a new secret"  | supply `value` → `UPDATE`               |
| "leave it alone"     | omit `value` → `KEEP`                   |
| "blind plan"         | metadata-only reconcile (no `GetParameter`) |

## evaluation

### how well it solves the goals

- ✅ **req 1 — new resource types**: `DeclaredAwsSsmParameterPlain` + `DeclaredAwsSsmParameterSecure`,
  each with domain object + DAO + get/set/cast ops + provider registration + acceptance,
  matching peers and replicating `DeclaredGithubOrgSecret`.
- ✅ **req 2 — no read/decrypt at plan for secrets**: plan calls only `DescribeParameters`.
  `GetParameter` is never issued for a secret — code-enforced, exactly like `getOneOrgSecret`.
- ⚠️ **req "tell whether a write is required"**: RELAXED for secrets, same as github — plan
  cannot diff the value. create-vs-keep works; a supplied `value` signals the write.
- ✅ **req 3 — IAM payoff**: plan for secrets needs only `ssm:DescribeParameters`; apply needs
  `PutParameter` (+ kms encrypt). `GetParameter` + `kms:Decrypt` leave the plan role.

### pros

- **proven pattern** — a faithful port of declastruct-github's write-only secret, which already
  ships and is tested. low novelty risk.
- **zero secret-derived artifacts** — no hash, salt, HMAC → no rainbow-table, correlation,
  brute-force, key-provisioning, or key-leak risk. the vuln suite is dissolved, not mitigated.
- **simpler than the hash design** — no `hash-fns`, no tag hashing, no salt lifecycle, no
  `ListTagsForResource` on the hot path.
- **write-only guarantee is code-enforced** — reconcile never issues `GetParameter` for a secret.

### cons / tradeoffs

- **no value-content drift detection for secrets** (github's exact tradeoff). an out-of-band
  value edit is invisible to plan; steady-state requires `value` undefined.
- **the concrete-value permadiff footgun** (also github's): a hard-coded `value` present every
  run shows `UPDATE` every plan. mitigation: source `value` from an env var set only when
  writing; document clearly.
- **the guarantee comes from an explicit `Deny kms:Decrypt` on the plan role — a CMK is
  OPTIONAL.** the real secret boundary is `kms:Decrypt`, not `GetParameter` — a `SecureString`
  at rest is ciphertext; the only way to read plaintext is a decrypt call (`GetParameter
  WithDecryption=true`). because **IAM explicit `Deny` always wins** over any allow (identity
  policy, key policy, or the default `aws/ssm` key's service-mediated grant), a single
  `Deny kms:Decrypt` on the plan role's own identity policy makes it unable to decrypt ANY
  secret — default-key or CMK. so the default `aws/ssm` key is sufficient for THIS role; no CMK
  required. a CMK remains optional defense-in-depth (a central, resource-side deny for
  principals OUTSIDE your role control — other roles, cross-account). see the IAM shape section.

### edgecases & pit-of-success

| edgecase                              | contract behavior                                        |
| ------------------------------------- | -------------------------------------------------------- |
| secret absent, `value` undefined      | plan `CREATE`; set THROWS "needs a value" (github .ts:64) |
| secret absent, `value` present        | plan `CREATE` → apply writes it                          |
| secret present, `value` undefined     | plan `KEEP` (no read)                                    |
| secret present, `value` present       | plan `UPDATE` → apply rewrites                           |
| findsert + already present            | set returns `before` no-op (github .ts:61)              |
| out-of-band value edit                | invisible to plan by design                             |
| plaintext param                       | normal value-compare; no write-only handling            |

pit-of-success guards: the secure cast returns `value: undefined`, so a secret value can never
be read back into a plan/state file — the write-only invariant is enforced at the cast, exactly
as `castToDeclaredGithubOrgSecret` does.

## IAM shape (recommended for consumers)

> **scope note:** these roles live in the CONSUMER (e.g. ahbode/infrastructure), NOT in
> declastruct-aws. this repo does not provision or enforce them — this section is recommended
> guidance the consumer applies to their own plan/apply OIDC roles. we document it because the
> resource's whole value proposition is the least-privilege posture it UNLOCKS.

### the asymmetry that drives it

- **plaintext (`String`) is two-way.** we chose value-compare (not hash-diff), so plan detects
  drift by reading the live value → plan **needs `ssm:GetParameter`** on plain paths. this is
  cheap: a `String` needs no decrypt.
- **secret (`SecureString`) is write-only.** plan reconciles via `DescribeParameters` (metadata)
  and apply writes via `PutParameter`. **neither role reads the secret**, so neither needs
  `GetParameter` on secure paths, and neither needs `kms:Decrypt`.

### the boundary is `kms:Decrypt` — enforced by an explicit identity `Deny`

a `SecureString` at rest is ciphertext; the ONLY way to obtain plaintext is a decrypt call.
so **deny `kms:Decrypt` and the secret is unreadable no matter who can GetParameter.**

the clean way to deny it: an **explicit `Deny kms:Decrypt` on the plan role's own identity
policy**. because IAM's **explicit `Deny` always wins** over any allow — from an identity
policy, a key policy, OR the default `aws/ssm` key's service-mediated grant — this one
statement blocks decryption of EVERY secret the role could touch, default-key or CMK:

```jsonc
// on the prod-on-else-for-plan role's identity policy
{ "Effect": "Deny", "Action": "kms:Decrypt", "Resource": "*" }
```

so **a CMK is NOT required** for this role — the default `aws/ssm` key is fine, because the
explicit deny overrides its implicit grant. a `GetParameter` path-scope (grant on plain paths,
omit on secret paths) is belt-and-suspenders hygiene on top.

**CMK (optional defense-in-depth):** the identity deny protects THIS role. it does not stop
*other* principals who hold decrypt via the default key. a customer-managed key gives a
central, resource-side deny (in the key policy) that does not depend on every role's own
identity-side deny — useful for account-wide secret hygiene or cross-account, but not needed
for the least-privilege plan role this wish targets.

### the grant matrix

| action | **plan** role | **apply** role |
| --- | --- | --- |
| `ssm:DescribeParameters` | ✅ (`*`-scoped — AWS can't resource-restrict it) | ✅ |
| `ssm:GetParameter` on **plain** paths | ✅ **needed** — value-compare drift detection | ✅ |
| `ssm:GetParameter` on **secure** paths | ❌ not needed (metadata-only reconcile) | ❌ not needed (write-only) |
| `ssm:PutParameter` | ❌ | ✅ (create/rotate) |
| `ssm:AddTagsToResource` | ❌ | ✅ (if tags declared) |
| `kms:Encrypt` / `GenerateDataKey` | ❌ | ✅ (to write secrets) |
| **`kms:Decrypt`** | ❌ **explicit `Deny`** | ❌ **explicit `Deny`** |

the `kms:Decrypt` row is an explicit identity-policy `Deny` (not merely an omitted allow) — so
it overrides the default `aws/ssm` key's implicit grant. works with the default key; no CMK
required.

### the through-line

> **plaintext is read-write** (plan reads via `GetParameter`, no decrypt) — **secrets are
> write-only** (no `GetParameter`, and an explicit `Deny kms:Decrypt` on both roles). the plan
> role can see every plaintext value and NO secret value; the apply role can write everything
> and read no secret.

path-scope recommendation: grant `GetParameter` only on the plain paths' ARNs and omit it on
secret paths — so a role cannot even fetch a secret's ciphertext. the explicit `Deny
kms:Decrypt` is the guarantee that survives a path-scope mistake.

## open questions & assumptions

### assumptions made

1. desired secret value is available to `apply` when writing (via env var / secret ref in the
   wish). we do NOT invent a secret-sourcing mechanism; the caller supplies `value`.
2. faithful github replication is the goal (wisher confirmed 2026-07-20). the enhancement
   (`omitWriteonly` + `valueVersion`) is optional and out of scope unless requested.
3. relaxed value-drift for secrets is acceptable — same tradeoff github already ships.

### questions (triaged: [answered] now · [research] later · [wisher] input needed)

- **[answered] StringList** — OUT OF SCOPE (wisher confirmed 2026-07-19). only `String` and
  `SecureString`.
- **[answered] write-only mechanism** — replicate declastruct-github: `writeonly=['value']`,
  `value?` optional, cast→undefined, metadata-only get, findsert-no-op set. NO hash/salt/HMAC.
- **[answered] keyId requirement** — a CMK is **OPTIONAL**. the guarantee comes from an explicit
  `Deny kms:Decrypt` on the plan role's identity policy (IAM explicit-deny beats the default
  `aws/ssm` key's implicit grant), so the default key suffices. a CMK is defense-in-depth for
  principals outside this role. `keyId: null` = default `aws/ssm` key (permitted). see the IAM
  shape section.
- **[answered] keyId change** — `PutParameter` with `Overwrite` accepts a new `KeyId` →
  in-place update, not recreate.
- **[answered] tier** — OMITTED from v1 (wisher questioned 2026-07-20). `tier` is a real AWS
  knob (`ParameterTier = Standard | Advanced | Intelligent-Tiering`, `PutParameter`
  `models_1.d.ts:3413`) but niche: it only matters for values >4 KB, param policies, or the
  per-param `Advanced` charge. surfacing it adds a cost footgun and drift noise
  (`Intelligent-Tiering` can silently upgrade remote tier ≠ declared). AWS defaults to
  `Standard`. add later only when a real >4 KB / policy need appears (`rule.prefer.wet-over-dry`).
- **[answered] nullability** — only readonly/@metadata (`arn`, `version`, `lastModifiedAt`) and
  the @writeonly `value` use `?`; all roundtrip read-write fields (`description`, `tags`,
  `keyId`) are required-nullable `X | null`, per `rule.forbid.undefined-attributes`.
- **[research] type change semantics** — verify AWS's rule for a `type` change on
  `PutParameter Overwrite`; if rejected, delete+recreate. (largely moot — Plain/Secure are
  separate resources, so a "type change" is a cross-resource move.)
- **[wisher] desired-value source at apply time** — where does `svc-notifications` source the
  plaintext in CI (env/secret ref)?
- **[answered] the concrete-value permadiff footgun** — github's "leave `value` undefined for
  steady-state `KEEP`" is the accepted model (wisher confirmed 2026-07-20). the optional
  `omitWriteonly` + `valueVersion` enhancement is NOT pursued — pure github replication stands.
- **[wisher] delete support** — github's dao implements `delete` (`delOrgSecret`). replicate
  `DeleteParameter`, or set `delete: null` like the peer SSM daos? (leaning replicate github.)

- exact `DescribeParameters` filter to fetch a single param's metadata by name without value
  (verified filter key `Name`/`Equals` against the installed SDK; confirm at build).
- `type`-change behavior on overwrite (largely moot post-split).

## groundwork

### external research (AWS SSM APIs)

verified against AWS SSM knowledge + the installed `@aws-sdk/client-ssm@3.943.0`
(package.json:70):

- **`DescribeParameters`** returns `ParameterMetadata[]` — `Name`, `Type`, `KeyId`,
  `LastModifiedDate`, `Description`, `Version`, `Tier`, `ARN` — and **NOT** `Value` (verified
  `interface ParameterMetadata`, `client-ssm/dist-types/models/models_0.d.ts:7249` — no `Value`
  field). supports `ParameterFilters` key `Name` + `Equals`. IAM: `*`-scoped only. ✅ existence
  + metadata with no value read — the SSM analogue of github's `getOrgSecret`.
- **`GetParameter`** (`WithDecryption`) is the ONLY call that returns the value / decrypts. our
  secure reconcile **never** calls it (mirrors `getOneOrgSecret`). (already wrapped at
  `src/access/sdks/sdkSsm/getOneParameter.ts`.)
- **`PutParameter`** writes value; `KeyId?` optional (falls back to `aws/ssm` when omitted —
  verified `PutParameterRequest`, `models_1.d.ts:3296-3310`). analogue of github's
  `createOrUpdateOrgSecret`. (wrapped at `src/access/sdks/sdkSsm/setParameter.ts`.)
- **kms**: no longer load-bearing — plan issues no `GetParameter`, so `kms:Decrypt` is moot for
  the base guarantee; a CMK is optional defense-in-depth.

### internal research (extant behavior to conform to)

- **write-only precedent — THE pattern to replicate (declastruct-github):**
  - `DeclaredGithubOrgSecret.ts:34,67` — `value?: string` optional, `static writeonly = ['value']`.
  - `castToDeclaredGithubOrgSecret.ts:27-28` — `value: undefined` (never read back).
  - `getOneOrgSecret.ts:14-45` — metadata-only get, null on Not Found, casts.
  - `setOrgSecret.ts:43-146` — findsert-no-op (`:61`), create-without-value guard (`:64-70`),

## what is awkward?

1. **we relax a wish requirement, exactly as github does.** the wish says "tell whether a write
   is required." write-only can't diff the value; it signals writes via a supplied `value`.
   `KEEP` requires `value` undefined. record explicit wisher sign-off (they steered us to
   github's model on 2026-07-20).

2. **the concrete-value permadiff footgun.** hard-code a `value` that's always present and plan
   shows `UPDATE` every run. this is github's own behavior; the mitigation (env-var-only-when-
   writing) is a documentation discipline, not a guardrail. the optional `omitWriteonly` +
   `valueVersion` enhancement would remove it, at the cost of a cross-repo change.

3. **`create` needs a value but plan can't enforce it.** if a secret is absent and `value` is
   undefined, plan says `CREATE` but the set op then throws "needs a value" at apply — a late
   failure. github has the identical shape (`setOrgSecret.ts:64`). acceptable, but the failure
   lands at apply, not plan.

4. **two resources for one AWS concept.** `Plain` + `Secure` split one `AWS::SSM::Parameter`
   into two declared types. cleaner per-variant behavior, but a reader must pick, and a
   `String`→`SecureString` change becomes a cross-resource move. acceptable given how different
   the two reconciles are.

# research.internal.product.code.prod: fix vpc tunnel env differentiation

## pattern.1 = DeclaredAwsVpcTunnel domain object

**source**: `src/domain.objects/DeclaredAwsVpcTunnel.ts`

```ts
// [1] unique keys do NOT include account
public static unique = ['via', 'into', 'from'] as const;
```

**relation to wish**: this is the root cause. the unique keys determine identity, but they exclude account. two tunnels with the same via/into/from but different accounts are considered the same resource.

**action**: [EXTEND] — add `account` field to interface and to `unique` keys

## pattern.2 = getTunnelHash utility

**source**: `src/domain.operations/vpcTunnel/utils/getTunnelHash.ts`

```ts
// [2] hash DOES include account
const serialized = serialize(
  JSON.parse(
    JSON.stringify({
      account: context.aws.credentials.account,  // <-- already included
      region: context.aws.credentials.region,
      via: input.for.tunnel.via,
      into: input.for.tunnel.into,
      from: input.for.tunnel.from,
      _v: 'v2025_11_27',
    }),
  ),
);
```

**relation to wish**: the hash already includes account from context. this is correct. the mismatch is that the domain object's `unique` doesn't include account while the hash does.

**action**: [REUSE] — after `account` is added to domain object, update hash to use `input.for.tunnel.account` instead of `context.aws.credentials.account`

## pattern.3 = DeclaredAwsVpcTunnelDao

```ts

  dobj: DeclaredAwsVpcTunnel,
  get: {
    one: {

    },
  },
  ...

```

**relation to wish**: the DAO uses `byUnique` which passes the unique ref. once `account` is in `unique`, the ref will include account automatically.

**action**: [REUSE] — no changes needed; the DAO pattern handles the change

# vision: fix vpc tunnel env differentiation

## the outcome world

### what does a day-in-the-life look like with this in place?

a developer working on a service that connects to RDS:
1. runs `STAGE=dev ./.agent/repo=.this/skills/use.vpc.tunnel.ts`
2. tunnel opens to the dev database cluster on port 15432
3. later, they need to check something in prod
4. runs `STAGE=prod ./.agent/repo=.this/skills/use.vpc.tunnel.ts` in another terminal
5. tunnel opens to the prod database cluster on port 15433
6. both tunnels coexist — dev on 15432, prod on 15433
7. declastruct correctly identifies each as a distinct resource

### what's the before/after contrast?

| aspect | before | after |
|--------|--------|-------|
| run dev after prod | "in sync" false positive — uses prod tunnel | opens new dev tunnel |
| parallel tunnels | collision — only one env at a time | coexist peacefully |
| resource identity | bastion + cluster (logical name) | bastion + cluster + account + region (physical identity) |
| debugging | "why is my dev test hitting prod?" | clear separation by env |

### what's the "aha" moment where the value clicks?

"i can have dev and prod tunnels open side-by-side and they don't interfere with each other."

the current behavior is confusing because:
- prod tunnel is running on port 15433
- you run STAGE=dev expecting port 15432
- declastruct says "in sync" because it matches on logical name `ahbodedb`
- but you're still connected to prod!

the fix makes the identity include the AWS account and region, so:
- dev tunnel (account 874711128849, us-east-1) is distinct from prod tunnel (account 398838478359, us-east-1)
- even though both target `ahbodedb`, they're different physical clusters

## user experience

### what usecases do folks fulfill?

1. **local development**: connect to dev database for testing
2. **production debugging**: connect to prod for investigation (separate from dev)
3. **ci/cd pipelines**: open tunnel in github actions for schema migrations
4. **parallel work**: have both envs open when comparing behavior

### what contract inputs & outputs do they leverage?

**input** (unchanged):
```ts
DeclaredAwsVpcTunnel.as({
  via: { mechanism: 'aws.ssm', bastion: { exid: 'vpc-main-bastion' } },
  into: { cluster: { name: 'ahbodedb' } },
  from: { host: 'localhost', port: config.database.tunnel.local.port },
  status: 'OPEN',
})
```

**identity differentiation** (the fix):
- currently: `unique = ['via', 'into', 'from']`
- the `getTunnelHash` already includes `account` and `region`
- but the domain object's `unique` definition doesn't

**proposed fix options**:

1. **add account + region to the domain object** — add `account` and `region` fields to `DeclaredAwsVpcTunnel`
2. **context-aware comparison** — make declastruct compare using context, not just unique keys
3. **lookup cluster to physical id** — expand `{ name: 'ahbodedb' }` to actual cluster identifier

### what would it look like to leverage them?

option 1 (explicit account + region):
```ts
DeclaredAwsVpcTunnel.as({
  account: config.aws.account, // new required field
  region: config.aws.region,   // new required field
  via: { mechanism: 'aws.ssm', bastion: { exid: 'vpc-main-bastion' } },
  into: { cluster: { name: 'ahbodedb' } },
  from: { host: 'localhost', port: config.database.tunnel.local.port },
  status: 'OPEN',
})
```

option 2 (context-aware):
```ts
// no change to user code — declastruct uses context.aws.credentials.account internally
DeclaredAwsVpcTunnel.as({
  via: { mechanism: 'aws.ssm', bastion: { exid: 'vpc-main-bastion' } },
  into: { cluster: { name: 'ahbodedb' } },
  from: { host: 'localhost', port: config.database.tunnel.local.port },
  status: 'OPEN',
})
```

option 3 (lookup to physical id):
```ts
// cluster identity expanded at runtime to include physical identifier
// e.g., { name: 'ahbodedb' } becomes { name: 'ahbodedb', id: 'cvpnwcrisal7' }
```

### what timelines do they go through?

1. configure STAGE env var
2. run use.vpc.tunnel.ts
3. tunnel opens (or confirms extant tunnel is correct for this stage)
4. use database
5. (optionally) close tunnel or let it persist

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| dev and prod tunnels coexist | yes |
| declastruct shows correct state | yes |
| no false "in sync" positives | yes |
| backward compatible | depends on option |

### what are the pros? the cons?

**option 1 (explicit account + region fields)**:
- pro: explicit, clear identity
- pro: user controls the account and region
- pro: hash and unique keys stay in sync
- con: break — requires update of all use.vpc.tunnel.ts files
- con: user must know their account id and region

**option 2 (context-aware comparison)**:
- pro: no user code changes
- pro: context is already available
- con: identity depends on context (not purely declarative)
- con: may require changes to declastruct core

**option 3 (lookup to physical id)**:
- pro: no user code changes
- pro: uses actual AWS resource identity
- con: requires API call to lookup cluster
- con: more complex implementation

### what edgecases are present and how do our contracts keep users in a pit of success?

| edgecase | handling |
|----------|----------|
| user runs with wrong STAGE | each stage has its own cache file — no collision |
| bastion shared across accounts | different hashes due to account differentiation |
| same port different account | different cache files, can't run simultaneously (port collision) |
| tunnel cache from old version | v-tag in hash forces re-creation |

## what is awkward?

### what feels off or forced?

**the logical name problem**: using `{ name: 'ahbodedb' }` is ergonomic but ambiguous. the same logical name is present in multiple accounts, but the physical clusters are different. this is a fundamental tension between:
- ergonomic: user says "connect to ahbodedb"
- precise: user says "connect to cluster cvpnwcrisal7 in account 874711128849"

**the hash includes account + region, unique doesn't**: there's a mismatch between what `getTunnelHash` uses for identity (includes account and region) and what `DeclaredAwsVpcTunnel.unique` declares (excludes both). this suggests the code evolved but the domain object didn't.

### where does the design fight the user's mental model?

users think: "i said STAGE=dev, so everything should be dev"

but the system thinks: "i see a tunnel for ahbodedb via vpc-main-bastion, so it's in sync"

the mental model mismatch is:
- user: "stage determines everything"
- system: "resource identity is independent of stage"

### what tradeoffs feel uncomfortable?

1. **breaking change vs clean design**: adding `account` to the domain object is clean but breaks extant users
2. **declarative vs context-dependent**: making identity depend on context feels less declarative
3. **simplicity vs precision**: lookup to physical IDs adds complexity but solves ambiguity

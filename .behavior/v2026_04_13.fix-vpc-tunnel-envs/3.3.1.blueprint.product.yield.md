# blueprint: fix vpc tunnel env differentiation

## summary

add `account` and `region` fields to `DeclaredAwsVpcTunnel` domain object and include them in the `unique` keys. this enables tunnels for different AWS accounts/regions to have distinct identities, even when they target the same logical resource name.

**break**: consumers must now pass `account` and `region` when they declare a tunnel.

## implementation notes

1. **no version bump in hash**: keep `_v` at `'v2025_11_27'` for now (cache invalidation not required)

2. **hash and unique in sync**: both account and region are now explicit in:
   - domain object unique keys
   - hash serialization
   - this ensures identity comparison and cache keys align

3. **break for consumers**: all consumers must update to pass `account` and `region`
   - declapract-typescript-ehmpathy use.vpc.tunnel.ts must add:
     - `account: config.aws.account`
     - `region: config.aws.region`

4. **account + region types**: both use `string` instead of ref types
   - reason: they are identity differentiators, not relationships
   - the tunnel doesn't "belong to" an account/region — it uses credentials from them
   - other domain objects use RefByPrimary when the resource EXISTS IN the account (relationship)
   - tunnel uses account + region for identity comparison only

# vision: AWS Organizations Service Control Policy support

## the outcome world

### day-in-the-life with SCPs in place

a security engineer at ahbode opens their terminal. they need to ensure no compromised role — even one with AdministratorAccess — can exfiltrate data or cover tracks.

```typescript
import {
  DeclaredAwsOrganizationServiceControlPolicy,
  DeclaredAwsOrganizationServiceControlPolicyAttachment,
  DeclaredAwsIamPolicyDocument,
  DeclaredAwsIamPolicyStatement,
  DeclaredAwsTags,
} from 'declastruct-aws';

// define the guardrail once
const denyDangerousActions = new DeclaredAwsOrganizationServiceControlPolicy({
  name: 'deny-dangerous-actions',
  description: 'block exfiltration, audit tamper, and org escape',
  content: new DeclaredAwsIamPolicyDocument({
    statements: [
      new DeclaredAwsIamPolicyStatement({
        sid: 'DenySnapshotShare',
        effect: 'Deny',
        action: [
          'rds:ModifyDBSnapshotAttribute',
          'rds:ModifyDBClusterSnapshotAttribute',
          'ec2:ModifySnapshotAttribute',
          'ec2:ModifyImageAttribute',
        ],
        resource: ['*'],
      }),
    ],
  }),
  tags: new DeclaredAwsTags({ managedBy: 'declastruct' }),
});

// attach to org root — applies to ALL accounts
const attachment = new DeclaredAwsOrganizationServiceControlPolicyAttachment({
  policy: { name: denyDangerousActions.name },
  target: { id: organization.id },
});

// idempotent apply
await daoServiceControlPolicy.set.findsert(denyDangerousActions, context);
await daoServiceControlPolicyAttachment.set.findsert(attachment, context);
```

they commit, push, CI runs. the SCP is applied org-wide. done.

tomorrow, an attacker compromises a prod role. they try `rds:ModifyDBSnapshotAttribute` to share a snapshot to their account. AWS returns `Access Denied`. the SCP blocked it at the org level — before IAM even evaluated.

### before/after contrast

| before (IAM-only) | after (SCP + IAM) |
|-------------------|-------------------|
| each role needs explicit deny | one policy covers all principals |
| new role = new attack surface | new role inherits SCP guardrails |
| attacker creates new role, bypasses denies | SCP blocks regardless of role |
| incident: "who forgot the deny?" | no forgetting — org-level enforced |

### the "aha" moment

> "wait — even the root user in a member account can't bypass this?"

exactly. SCPs set the ceiling. IAM permissions can only be equal or more restrictive. an AdministratorAccess role in a member account is still bound by the SCP.

## user experience

### usecases & goals

| usecase | goal | contract |
|---------|------|----------|
| block exfiltration | prevent snapshot/AMI sharing | `DeclaredAwsOrganizationServiceControlPolicy` |
| protect audit trail | prevent cloudtrail/config tampering | same |
| enforce region | restrict to approved regions | same |
| attach to org root | apply to all accounts | `DeclaredAwsOrganizationServiceControlPolicyAttachment` |
| attach to specific account | targeted guardrail | same (target = account ID) |

### contract inputs & outputs

**DeclaredAwsOrganizationServiceControlPolicy**

```typescript
interface DeclaredAwsOrganizationServiceControlPolicy {
  // metadata (aws-assigned, readonly)
  id?: string;      // e.g., 'p-abc123'
  arn?: string;     // e.g., 'arn:aws:organizations::123:policy/p-abc123'

  // unique key
  name: string;     // 1-128 chars, unique per org

  // attributes
  description?: string;  // max 512 chars
  content: DeclaredAwsIamPolicyDocument;  // max 5,120 chars when serialized
  tags?: DeclaredAwsTags;
}
```

**DeclaredAwsOrganizationServiceControlPolicyAttachment**

```typescript
interface DeclaredAwsOrganizationServiceControlPolicyAttachment {
  // references
  policy: RefByUnique<typeof DeclaredAwsOrganizationServiceControlPolicy>;  // { name }
  target: RefByPrimary<typeof DeclaredAwsOrganization>                      // { id: 'r-xxx' } (root)
        | RefByPrimary<typeof DeclaredAwsOrganizationAccount>;              // { id: '123456789012' }
  // note: OU support deferred — requires DeclaredAwsOrganizationUnit first
}
```

### what it looks like to use them

```typescript
// 1. create SCP
const scp = await daoServiceControlPolicy.set.findsert(
  new DeclaredAwsOrganizationServiceControlPolicy({
    name: 'deny-snapshot-share',
    description: 'prevent RDS/EBS snapshot exfiltration',
    content: policyDocument,
    tags: null,
  }),
  context,
);

// 2. attach to org root
await daoServiceControlPolicyAttachment.set.findsert(
  new DeclaredAwsOrganizationServiceControlPolicyAttachment({
```

## mental model

### how users describe this to a friend

> "SCPs are like a firewall at the org level. IAM says what you CAN do. SCPs say what you CAN'T do — no matter what IAM allows. think of it as a ceiling on permissions."

### analogies

| analogy | maps to |
|---------|---------|
| building code (construction) | SCP = city-wide ordinance; IAM = per-building rules |
| airport security | SCP = TSA (everyone goes through); IAM = airline ticket (individual access) |
| parental controls | SCP = router-level block; IAM = per-device settings |

### terms: theirs vs ours

| AWS term | declastruct term | why |
|----------|------------------|-----|
| Service Control Policy | `DeclaredAwsOrganizationServiceControlPolicy` | explicit prefix for namespace clarity |
| Policy Attachment | `DeclaredAwsOrganizationServiceControlPolicyAttachment` | explicit — not just "attachment" |
| PolicyId | `id` (metadata) | standard pattern |
| TargetId | `target` (RefByPrimary) | domain object reference |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| prevent exfiltration | yes | block ModifyDBSnapshotAttribute, ModifyImageAttribute |
| protect audit trail | yes | block DeleteTrail, StopLogging |
| prevent org escape | yes | block LeaveOrganization |
| declarative management | yes | domain objects + findsert/upsert |
| idempotent operations | yes | findsert returns extant if name matches |

### pros

- **org-wide ceiling**: one policy, all accounts, all principals
- **cannot bypass**: even AdministratorAccess respects SCPs
- **declarative**: version-controlled, auditable, reproducible
- **idempotent**: safe to re-apply
- **reuses extant patterns**: DeclaredAwsIamPolicyDocument, RefByUnique

### cons

- **management account only**: must run from org management account (or delegated admin)
- **5KB limit**: policy document max 5,120 chars — complex policies need splitting
- **no conditions on some actions**: SCPs can't use all IAM condition keys
- **OU support deferred**: this vision targets org root + accounts; OU requires new domain object

### edgecases & pit-of-success

| edgecase | pit-of-success |
|----------|----------------|
| empty statements array | runtime validation fails before API call |
| policy > 5KB | content.serialize() checks length, throws ConstraintError |
| duplicate attachment | findsert returns extant (idempotent) |
| detach last SCP from root | AWS blocks this; we surface the error clearly |
| name collision | findsert finds extant by name, returns it |

## open questions & assumptions

### assumptions made

1. **OU support deferred**: we attach to org root (r-xxx) or account (12-digit). OU attachment requires `DeclaredAwsOrganizationUnit` which doesn't exist yet.

2. **unique key = name**: AWS doesn't enforce name uniqueness, but declastruct-aws convention is findsert-by-unique. we treat name as unique key.

3. **no upsert for content**: AWS UpdatePolicy exists, but changing SCP content mid-flight is risky. initial scope: findsert + delete. upsert deferred.

4. **reuse DeclaredAwsIamPolicyDocument**: SCP syntax is a subset of IAM policy syntax. we reuse extant domain objects rather than create SCP-specific ones.

### questions for wisher [wisher]

1. **scope: OU support?** [wisher]
   - wish acceptance criteria says "org root, ou, or account"
   - should we add OU in this phase or defer?
   - note: DeclaredAwsOrganizationUnit doesn't exist yet
   - recommendation: defer OU to separate wish

2. **upsert semantics?** [wisher]
   - UpdatePolicy exists in AWS API
   - should we support upsert, or findsert-only?
   - recommendation: findsert + delete initially; upsert if needed

3. **delegation support?** [wisher]
   - AWS supports delegated administrator for organizations
   - do we need to surface this?
   - recommendation: defer — management account is sufficient for ahbode

### questions answered [answered]

1. **duplicate name detection?** [answered]
   - AWS doesn't enforce name uniqueness
   - if duplicate names exist, findsert could silently pick wrong policy
   - resolution: findsert should detect duplicates and throw ConstraintError
   - rationale: fail loud is safer than silent wrong-selection

## what is awkward?

### unique key concern

AWS does not enforce policy name uniqueness. two policies can have the same name with different IDs. declastruct-aws convention expects unique keys to be... unique.

**options:**
1. **trust the user** — document that names should be unique; findsert finds first match
2. **org-scoped unique** — treat (organizationId, name) as composite unique key
3. **use ID as unique** — but then user must know ID upfront (bad UX)

**recommendation:** option 1 — trust the user, document clearly. if they create duplicate names outside declastruct, that's their problem. findsert will find "a" match (first one returned by API).

### attachment as separate domain object

the wish shows `DeclaredAwsOrganizationServiceControlPolicyAttachment` as a domain object. this matches the API (AttachPolicy is a separate call). but it feels heavy:

```typescript
// two domain objects for one logical action
const scp = new DeclaredAwsOrganizationServiceControlPolicy({ ... });
const attachment = new DeclaredAwsOrganizationServiceControlPolicyAttachment({ ... });
```

**alternative:** embed `attachedTo` array in the policy itself:
```typescript
const scp = new DeclaredAwsOrganizationServiceControlPolicy({
  ...,
  attachedTo: [{ id: 'r-xxx' }],  // targets
});
```

**recommendation:** keep separate domain object. reasons:
- matches AWS API granularity
- one policy → many targets is cleaner with separate attachment entity
- allows fine-grained control (detach from one target, keep others)
- consistent with how IAM policies work in declastruct-aws

### 5KB limit

SCP content is capped at 5,120 characters. a comprehensive deny policy can hit this fast. users may need to split across multiple SCPs.

**mitigation:** surface clear ConstraintError when content exceeds limit, with hint to split policy.

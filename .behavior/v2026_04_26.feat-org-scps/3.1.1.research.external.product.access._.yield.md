# research: external product access for AWS Organizations SCP support

## summary

to fulfill the SCP feature, we need to access the AWS Organizations API via the `@aws-sdk/client-organizations` package. the API provides full CRUD operations for SCPs with well-documented contracts, limits, and error types.

---

## remote repositories required

### AWS Organizations API

the primary remote repository is the AWS Organizations service, accessed via REST API.

**endpoint requirement:**
> "For the current release of Organizations, specify the `us-east-1` region for all Amazon Web Services API and CLI calls made from the commercial Amazon Web Services Regions outside of China." [1]

**authorization:**
> "You can only call this operation from the management account or a member account that is a delegated administrator." [2]

---

## API contracts

### CreatePolicy [2]

creates a new SCP in the organization.

**request parameters:**
- `Content` (required): "The policy text content to add to the new policy. The text must be correctly formatted JSON." [2]
- `Name` (required): "The friendly name to assign to the policy." Length: 1-128 characters [2]
- `Description` (optional): "An optional description to assign to the policy." Max 512 chars [2]
- `Type` (required): must be `SERVICE_CONTROL_POLICY` for SCPs [2]
- `Tags` (optional): key-value pairs [2]

**response:**
> "A structure that contains details about the newly created policy." [2]

returns `Policy.PolicySummary` with `Id`, `Arn`, `Name`, `Description`, `Type`, `AwsManaged` [2]

**key errors:**
- `DuplicatePolicyException`: "A policy with the same name already exists." [2]
- `MalformedPolicyDocumentException`: "The provided policy document doesn't meet the requirements of the specified policy type." [2]
- `ConstraintViolationException`: includes `POLICY_CONTENT_LIMIT_EXCEEDED`, `POLICY_NUMBER_LIMIT_EXCEEDED` [2]

### AttachPolicy [3]

attaches an SCP to a target (root, OU, or account).

**request parameters:**
- `PolicyId` (required): pattern `^p-[0-9a-zA-Z_]{8,128}$` [3]
- `TargetId` (required): root (`r-xxxx`), account (12 digits), or OU (`ou-xxxx-xxxx`) [3]

**response:**
> "If the action is successful, the service sends back an HTTP 200 response with an empty HTTP body." [3]

**key errors:**
- `DuplicatePolicyAttachmentException`: "The selected policy is already attached to the specified target." [3]
- `PolicyNotFoundException`: "We can't find a policy with the PolicyId that you specified." [3]
- `TargetNotFoundException`: "We can't find a root, OU, account, or policy with the TargetId that you specified." [3]

### DetachPolicy [4]

detaches an SCP from a target.

**constraint:**
> "Every root, OU, and account must have at least one SCP attached." [4]
> "If you want to replace the default `FullAWSAccess` policy with an SCP that limits the permissions that can be delegated, you must attach the replacement SCP before you can remove the default SCP." [4]

**key errors:**
- `ConstraintViolationException` with `MIN_POLICY_TYPE_ATTACHMENT_LIMIT_EXCEEDED`: cannot detach last SCP [4]
- `PolicyNotAttachedException`: "The policy isn't attached to the specified target in the specified root." [4]

**effect:**
> "If the policy that is detached is a service control policy (SCP), the changes to permissions for AWS Identity and Access Management (IAM) users and roles in affected accounts are immediate." [4]

### DeletePolicy [5]

deletes an SCP from the organization.

**prerequisite:**
> "Before you perform this operation, you must first detach the policy from all organizational units (OUs), roots, and accounts." [5]

**key errors:**
- `PolicyInUseException`: "The policy is attached to one or more entities. You must detach it from all roots, OUs, and accounts before you perform this operation." [5]
- `PolicyNotFoundException`: "We can't find a policy with the PolicyId that you specified." [5]

### UpdatePolicy [6]

updates an SCP's name, description, or content.

> "You can't change a policy's type." [6]

**key errors:**
- `DuplicatePolicyException`: "A policy with the same name already exists." [6]
- `PolicyChangesInProgressException`: "Changes to the effective policy are in progress, and its contents can't be returned." [6]

### ListPolicies [7]

retrieves all policies of a specified type.

**pagination:**
> "When you call List* operations, always check the `NextToken` response parameter value, even if you receive an empty result set. These operations can occasionally return an empty set of results even when more results are available. Continue to make requests until `NextToken` returns null." [7]

**max results:** 1-20 per request [7]

### ListPoliciesForTarget / ListTargetsForPolicy [8]

retrieves policies attached to a target, or targets attached to a policy.

## limits and quotas [9]

| limit | value |
|-------|-------|
| max SCPs per organization | 10,000 |
| max SCP document size | 5,120 characters |
| max SCPs per root | 5 |
| max SCPs per OU | 5 |
| max SCPs per account | 5 |
| min SCPs per entity | 1 (when SCPs enabled) |

> "Every entity must have _at least_ one SCP attached at all times when you enable SCPs. You can't remove the last SCP from an entity." [9]

> "Policies that affect an OU or account by inheritance do **_not_** count against these limits." [9]

**character count:**
> "If you save the policy via the AWS Management Console, extra white space (such as spaces and line breaks) between JSON elements and outside of quotation marks, is removed and not counted. If you save the policy via an SDK operation or the AWS CLI, then the policy is saved exactly as you provided and no automatic removal of characters occurs." [9]

## SCP evaluation model [10]

**intersection rule:**
> "The effective permissions are the logical intersection between what is allowed by the SCP and resource control policies (RCPs) and what is allowed by the identity-based and resource-based policies." [10]

**deny-by-default:**
> "SCP evaluation follows a deny-by-default model; any permissions not explicitly allowed in the SCPs are denied." [11]

**explicit allow required:**
> "For a permission to be **allowed** for a specific account, there must be an **explicit `Allow` statement** at every level from the root through each OU in the direct path to the account (that includes the target account itself)." [11]

**any deny blocks:**
> "For a permission to be **denied** for a specific account, **any SCP** from the root through each OU in the direct path to the account (that includes the target account itself) can deny that permission." [11]

**inheritance:**
> "An SCP restricts permissions for IAM users and roles in member accounts, that includes the member account's root user. Any account has only those permissions permitted by **_every_** parent above it." [10]

## best practices [13][14][15]

### deny-list vs allow-list

> "Use deny-list for most organizations. AWS launches new services constantly. With a deny-list approach, new services are automatically available unless explicitly blocked. With an allow-list, every new service is blocked by default, and you need constant SCP updates to keep pace with AWS releases." [13]

### explicit deny statements

> "AWS updated its SCP guidance in 2025 to emphasize: Use explicit Deny statements rather than rely on implicit denies. Deny statements in SCPs cannot be overridden by IAM policies." [13]

### test before deployment

> "AWS strongly recommends that you don't attach SCPs to the root of your organization without thorough tests of the impact that the policy has on accounts. Instead, create an OU that you can move your accounts into one at a time, or at least in small numbers, to ensure that you don't inadvertently lock users out of key services." [14]

### FullAWSAccess policy

> "You should not remove the **FullAWSAccess** policy unless you modify or replace it with a separate policy with allowed actions, otherwise all AWS actions from member accounts will fail." [10]

---

## anti-patterns [15][16]

### conflicted SCPs

> "Conflicted or overlapped SCPs can cause unexpected Deny behavior, and the fix is to consolidate or merge policies." [15]

### wrong approach: allow-list

> "For most organizations, the deny-list approach (where actions are allowed by default and you explicitly deny specific actions) is best practice." [15]

### global service exemptions omitted

> "Omitted global service exemptions can cause IAM, Route 53, and CloudFront to stop work, and the fix is to use NotAction for global services." [15]

### over-complex OU structures

> "It's desirable to keep the depth of OUs simple, not exceed 3-4 levels." [15]

## management account scope [10][18]

**SCPs do not affect management account:**
> "SCPs don't affect users or roles in the management account. They affect only the member accounts in your organization." [10]

**delegated administrator:**
> "Unlike the management account, delegated administrator accounts are subject to SCPs. A misconfigured SCP can block the delegated administrator from its duties." [18]

**delegation capability:**
> "With delegated administrator for AWS Organizations policies, you can delegate the management of policies of each policy type to a designated member account." [19]

## 2025 updates [13][20]

> "In September 2025, AWS Organizations now offers full IAM policy language support for service control policies (SCPs)." [13]

new capabilities:
- conditions in Allow statements
- individual resource ARNs
- NotAction element with Allow statements
- wildcards anywhere in Action element strings
- NotResource element [20]

## convergence signals

**sources [2][3][4][5][6][7] agree:** all CRUD operations require management account or delegated admin credentials.

**sources [9][10][14] agree:** minimum 1 SCP per entity when SCPs are enabled.

**sources [10][11][13][15] agree:** deny-list strategy is recommended over allow-list for most organizations.

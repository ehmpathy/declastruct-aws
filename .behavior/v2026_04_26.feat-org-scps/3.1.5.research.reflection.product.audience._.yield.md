# research: audience reflection for AWS Organizations SCP support

## who is the audience?

### primary: security engineers

security engineers at organizations who use AWS Organizations with multiple accounts. they are responsible for:
- implementation of org-wide security guardrails
- prevention of data exfiltration vectors
- protection of audit trails from tamper attempts
- response to compromised credentials

**context:** they operate from the org management account (or as delegated admin). they understand IAM permissions model and recognize its limitations — that IAM alone cannot prevent a compromised AdministratorAccess role from data exfiltration.

**constraints:** they need guardrails that:
- cannot be bypassed by any principal, also root user in member accounts
- are declarative and version-controlled (audit trail)
- are idempotent (safe to re-apply, CI/CD friendly)

### secondary: devops engineers / platform teams

engineers who manage infrastructure-as-code for AWS environments. they:
- deploy and maintain declastruct resource declarations
- run CI/CD pipelines that apply changes
- troubleshoot deployment failures

**context:** they may not deeply understand SCP evaluation semantics, but they need clear error messages when an operation fails (e.g., "cannot detach last SCP").

**constraints:** they need:
- familiar patterns (get/set/del, findsert semantics)
- clear error messages with hints
- idempotent operations (re-run safe)

### tertiary: compliance officers / auditors

compliance stakeholders who review security controls for SOC 2, PCI-DSS, HIPAA, etc. they:
- audit what guardrails are in place
- verify controls match documented policies
- review change history

**context:** they don't write code, but they need to understand what SCPs do and verify they match organizational policies.

**constraints:** they need:
- clear documentation of what each SCP blocks
- audit trail of when SCPs were created/modified
- declarative source of truth (version control)

## why do they care?

### pain this relieves

| audience | pain | relief |
|----------|------|--------|
| security engineer | compromised role can exfiltrate data via snapshot share | SCP blocks at org level, regardless of IAM |
| security engineer | attacker creates new role to bypass denies | SCP applies to all principals, even new roles |
| security engineer | manual SCP management via console is error-prone | declarative, version-controlled, CI/CD friendly |
| devops engineer | imperative AWS CLI commands are hard to track | idempotent domain objects with findsert semantics |
| compliance officer | cannot prove guardrails are in place | SCPs are code, auditable in version control |

### goal this enables

| audience | goal | enabled by |
|----------|------|------------|
| security engineer | org-wide security upper-bound | one SCP attached to org root applies everywhere |
| security engineer | sleep soundly with exfiltration blocked | declarative proof of guardrails |
| devops engineer | consistent patterns for all AWS resources | SCP domain objects follow extant declastruct patterns |
| compliance officer | demonstrate control effectiveness | version-controlled SCP definitions |

### frustration this eliminates

| audience | frustration |
|----------|-------------|
| security engineer | "why do we have to add deny to every role?" |
| security engineer | "who forgot to update the new role's permissions?" |
| devops engineer | "is this idempotent? will it fail if I run twice?" |
| compliance officer | "how do I prove this control is actually enforced?" |

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| primary (security engineer) | high | high | data breach prevention, audit protection |
| secondary (devops engineer) | medium | medium | deployment reliability, familiar patterns |
| tertiary (compliance officer) | medium | low | audit readiness, documentation |

**stakes for primary audience:**
- compromised role can share RDS snapshots to external accounts (data exfiltration)
- attacker can disable CloudTrail (cover tracks)
- attacker can leave organization (escape guardrails)

these are high-stakes security concerns that SCPs directly address.

## what do they care about?

### success criteria

| audience | judges success by |
|----------|-------------------|
| security engineer | guardrails enforced org-wide; cannot be bypassed |
| security engineer | idempotent operations (safe re-apply) |
| devops engineer | familiar get/set/del patterns |
| devops engineer | clear error messages when an operation fails |
| compliance officer | declarative SCPs match documented policies |

### tradeoffs they would accept

| audience | acceptable tradeoff |
|----------|---------------------|
| security engineer | OU support deferred (org root + account sufficient for v1) |
| security engineer | upsert deferred (findsert + delete sufficient for v1) |
| devops engineer | must run from management account (acceptable security constraint) |

### tradeoffs they would reject

| audience | unacceptable tradeoff |
|----------|----------------------|
| security engineer | silent failure when duplicate names found |
| security engineer | imperative-only API (no findsert semantics) |
| devops engineer | non-idempotent operations (fail on re-apply) |
| devops engineer | vague error messages without hints |

### what would delight vs disappoint

| audience | delighted | disappointed |
|----------|-----------|--------------|
| security engineer | findsert finds extant, no duplicate | findsert silently picks wrong policy |
| security engineer | clear ConstraintError on 5KB limit with hint | silent truncation |
| devops engineer | familiar RefByUnique/RefByPrimary patterns | new unfamiliar patterns |
| devops engineer | reuse of extant DeclaredAwsIamPolicyDocument | new SCP-specific policy type |

## how will they experience it?

### touchpoints

| touchpoint | audience | experience |
|------------|----------|------------|
| domain object construction | all | `new DeclaredAwsOrganizationServiceControlPolicy({ name, content })` |
| findsert operation | all | `await dao.set.findsert(scp, context)` |
| attachment creation | all | `new DeclaredAwsOrganizationServiceControlPolicyAttachment({ policy, target })` |
| error message | devops | `ConstraintError: policy content exceeds 5,120 character limit` |
| integration test | devops | runs against real AWS Organizations API |

### friction points

| friction | mitigation |
|----------|------------|
| must run from management account | document clearly; error message if wrong account |
| 5KB content limit | clear ConstraintError with hint to split policy |
| duplicate names in org | findsert detects duplicates, throws ConstraintError |
| cannot detach last SCP | surface AWS error clearly with explanation |
| OU not supported in v1 | document scope; defer to separate wish |

### smooth experience

1. security engineer defines SCP as domain object in code
2. SCP reuses familiar DeclaredAwsIamPolicyDocument pattern
3. findsert is idempotent — safe to re-run in CI/CD
4. attachment to org root applies to all accounts
5. if an operation fails, error message explains why and hints at fix
6. changes are version-controlled, auditable

---

## summary

the SCP feature serves security engineers who need org-level guardrails that cannot be bypassed. the stakes are high (data breach, audit tamper). success means:
- familiar declastruct patterns (get/set/del, findsert)
- reuse of extant domain objects (DeclaredAwsIamPolicyDocument)
- idempotent operations
- clear error messages
- declarative, version-controlled SCPs

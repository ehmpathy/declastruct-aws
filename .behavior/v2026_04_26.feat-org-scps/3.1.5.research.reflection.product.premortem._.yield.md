# research: premortem reflection for AWS Organizations SCP support

## premortem: imagine we shipped and it failed

we shipped SCP support. it failed badly. why?

### plausible failure scenarios

1. **silent wrong-policy selection**: user has duplicate-named SCPs in their org (created outside declastruct). findsert finds "a" policy but not "the" policy. user thinks guardrail is in place, but wrong policy is attached. security breach occurs through the intended-to-be-blocked vector.

2. **findsert creates when it should find**: race condition or eventual consistency causes get.byUnique to return null even though policy exists. findsert creates duplicate. user now has two policies with same name. later operations pick the wrong one.

3. **policy content exceeds 5KB silently**: user writes a comprehensive deny policy. serialized content is 5,500 chars. if we don't validate before API call, AWS rejects with unclear error. user wastes hours on what seems like API misconfiguration.

4. **attachment succeeds but no effect**: user attaches SCP to wrong target (e.g., account ID typo). attachment operation succeeds, but intended accounts are unprotected. user believes guardrail is in place; it is not.

5. **cannot detach last SCP confusion**: user tries to remove all SCPs via delete. AWS blocks detach of last SCP. error message is unclear. user thinks declastruct is broken.

6. **integration tests brittle**: tests require org management account. test passes in CI, fails locally. devs skip tests. regressions ship.

---

## what obvious signs did we ignore?

look back from the imagined failure — what signs were present that we dismissed?

- **AWS allows duplicate policy names**: we knew this but assumed "users will be careful." they weren't.

- **findsert-by-unique assumes uniqueness**: the pattern assumes unique key is actually unique. AWS doesn't enforce this. we should have added duplicate detection.

- **5KB limit is small**: complex deny policies hit this fast. we should have documented split strategies and enforced limit client-side.

- **org management account requirement**: we knew this but didn't surface clear errors when credentials are wrong.

- **eventual consistency**: AWS Organizations API has eventual consistency on some operations. we assumed immediate consistency.

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| policy names are unique per org | findsert picks wrong policy; security gaps | medium — AWS allows duplicates via console/CLI |
| AWS API is immediately consistent | race conditions; duplicate creates | low — but documented for some operations |
| users run from management account | unclear auth errors; wasted debug time | medium — credential misconfiguration is common |
| 5KB is enough for typical SCPs | users hit limit; frustration | high — comprehensive deny lists are verbose |
| OU support can be deferred | users need OU attachment now | low — org root + account covers most usecases |
| upsert can be deferred | users need to update content | medium — policy changes are common |
| reuse of DeclaredAwsIamPolicyDocument works | SCP-specific syntax not supported | low — SCP syntax is subset of IAM |

## what is the nightmare scenario?

### the breach that should have been blocked

security engineer deploys SCP to block snapshot exfiltration. they have an old, manually-created SCP with the same name (from a previous attempt). findsert finds the old policy, returns it as extant. attachment succeeds — but to the old policy, which has weaker rules.

attacker compromises a role. attempts `rds:ModifyDBSnapshotAttribute`. security engineer expects `Access Denied`. instead, it succeeds. attacker exfiltrates production database snapshot.

**aftermath:**
- security engineer: "I deployed the SCP! why didn't it block?"
- investigation: "you have two policies named 'deny-snapshot-share'. findsert found the old one."
- blame: "declastruct silently picked the wrong policy"

this is the worst outcome: **false confidence**. user believes guardrail is in place. it is not.

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| duplicate name → wrong policy | findsert detects duplicates, throws ConstraintError | low — add ListPolicies check |
| 5KB limit exceeded | validate content.length before API call; throw ConstraintError with hint | low — pre-flight check |
| wrong account credentials | detect non-management account; throw BadRequestError with clear message | low — auth check |
| race condition on create | use idempotency key if AWS supports; document limitation | medium — may need retry logic |
| cannot detach last SCP | surface AWS error with explanation; suggest replace-then-remove pattern | low — clear error message |
| OU support needed | document scope clearly; track as follow-up wish | low — documentation |
| upsert needed | document scope; track as follow-up wish | low — documentation |
| attachment to wrong target | validate target exists via get before attach? | medium — extra API call |

### priority mitigations (must have for v1)

1. **duplicate name detection**: findsert must check for duplicates and fail loud
2. **5KB limit validation**: check content length before API call
3. **clear auth error**: detect wrong account and surface helpful error
4. **clear "last SCP" error**: explain why detach failed and how to fix

### deferred mitigations (acceptable risk for v1)

- OU support: document as out of scope; org root + account sufficient
- upsert: document as out of scope; findsert + delete sufficient
- target validation: trust user-provided target ID; AWS will reject invalid

---

## summary

the nightmare scenario is **false confidence** — user believes guardrail is in place, but findsert silently picked the wrong policy due to duplicate names. mitigation: detect duplicates, fail loud.

other key mitigations:
- validate 5KB limit client-side
- clear error on wrong account credentials
- clear error on "cannot detach last SCP"

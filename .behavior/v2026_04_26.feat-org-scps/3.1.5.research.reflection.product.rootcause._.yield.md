# research: root cause reflection for AWS Organizations SCP support

## what is the symptom?

### observable problem

security engineers at organizations with multiple AWS accounts cannot enforce org-wide security guardrails that are immune to bypass.

### how it manifests

- per-role IAM deny policies must be added to every role
- attacker who compromises a role can create a new role without the deny
- even AdministratorAccess roles can share snapshots, disable CloudTrail
- no declarative way to manage org-level constraints in declastruct-aws

### when it occurs

whenever an organization needs to:
- prevent data exfiltration across all accounts
- protect audit trails from all principals
- enforce compliance controls that cannot be bypassed

---

## 5 whys tree

### branch 1: security gaps with IAM-only approach

1. **why do security engineers need SCP support in declastruct-aws?**
   → because they cannot declaratively manage org-level guardrails today

2. **why can't they use IAM policies alone for org-level guardrails?**
   → because IAM policies apply per-identity; attacker can create new identity without denies

3. **why can an attacker create a new identity to bypass IAM denies?**
   → because compromised roles often have IAM:CreateRole or AdministratorAccess

4. **why do production roles have such broad permissions?**
   → because many legitimate workloads require administrative access to function

5. **why is there no upper bound that applies regardless of IAM permissions?**
   → **ROOT CAUSE: IAM operates at identity level; SCPs operate at organization level. AWS designed SCPs to fill the "upper bound" gap that IAM cannot address.**

### branch 2: declarative management gap

1. **why do engineers need SCP support in declastruct-aws specifically?**
   → because declastruct-aws is their infrastructure-as-code tool for AWS resources

2. **why not use AWS Console or CLI for SCPs?**
   → because imperative management is error-prone and not version-controlled

3. **why is imperative management problematic?**
   → because changes are not tracked, not auditable, not reproducible

4. **why do they need audit trail and reproducibility?**
   → because compliance requirements demand proof of controls; "who changed what when?"

5. **why can't they get this from extant tools?**
   → **ROOT CAUSE: declastruct-aws doesn't support SCP domain objects yet; no declarative SCP management in their stack.**

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| symptom | cannot enforce org-wide guardrails | no |
| cause | IAM denies are per-identity, bypassable | no |
| cause | attackers can create new identities | no |
| cause | production roles need broad permissions | no |
| **root cause** | **IAM is identity-level; SCPs are the org-level upper bound that IAM cannot provide** | **yes** |
| **root cause** | **declastruct-aws lacks SCP domain objects for declarative management** | **yes** |

---

## recommendation

### address both root causes

**root cause 1 (architectural)**: IAM cannot provide org-level upper bound. AWS created SCPs to solve this. we cannot change AWS architecture; we adopt their solution.

**root cause 2 (tools)**: declastruct-aws lacks SCP support. this is what we build.

### the fix

add to declastruct-aws:
- `DeclaredAwsOrganizationServiceControlPolicy` domain object
- `DeclaredAwsOrganizationServiceControlPolicyAttachment` domain object
- DAO with findsert/delete semantics
- clear error messages for edge cases

### why this is root cause, not symptom

to address these root causes provides:
- **org-level upper bound**: SCPs block actions regardless of IAM permissions
- **declarative management**: version-controlled, auditable, reproducible
- **idempotent operations**: safe to re-apply, CI/CD friendly

this is not a bandaid — it fills a genuine architectural gap (SCPs) with proper tools (declastruct domain objects).

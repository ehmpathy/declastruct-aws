# research: external product claims for AWS Organizations SCP support

## summary

this document catalogs facts, assumptions, open questions, and opinions found via external research about AWS Organizations SCPs. each claim is labeled as [FACT], [SUMP], [KHUE], or [OPIN].

---

## claims: SCP fundamentals

### [FACT] SCPs do not grant permissions

> "SCPs do not grant permissions to the IAM users and IAM roles in your organization. No permissions are granted by an SCP. An SCP defines a permission guardrail, or sets limits, on the actions that the IAM users and IAM roles in your organization can perform." [1]

convergence: sources [1][2][3][4] agree on this fundamental property.

### [FACT] SCPs set the upper bound on permissions

> "An SCP defines a permission guardrail, or sets limits, on the actions that the IAM users and IAM roles in your organization can perform." [1]

> "For an action to be permitted, both the IAM policy and the SCP must explicitly allow it." [4]

### [FACT] SCPs do not affect management account

> "SCPs don't affect users or roles in the management account. They affect only the member accounts in your organization." [1]

convergence: sources [1][5][6] consistently state this.

### [FACT] effective permissions = intersection of all policies

> "The effective permissions are the logical intersection between what is allowed by the SCP and resource control policies (RCPs) and what is allowed by the identity-based and resource-based policies." [1]

> "An IAM entity (user or role) can make a request that is affected by an SCP, a permissions boundary, and an identity-based policy. In this case, the request is allowed only if all three policy types allow it." [2]

### [FACT] SCPs affect even AdministratorAccess

> "A user or role in the affected account can't use that permission, even if the account administrator attaches the `AdministratorAccess` IAM policy with \*/\* permissions to the user." [1]

## claims: SCP vs IAM policies

### [FACT] SCPs and IAM policies use similar syntax but differ in scope

> "SCPs are similar to AWS Identity and Access Management permission policies and use almost the same syntax." [1]

> "SCPs impose boundaries at the organizational or account level, while IAM policies regulate permissions at the identity level." [3]

### [FACT] SCPs are preventive, not detective

> "SCPs are preventive controls, not detective controls. They stop mistakes before they happen rather than alert you after damage is done." [7]

### [OPIN] SCPs are more effective than per-role IAM denies

> "A developer cannot accidentally delete production databases if an SCP prevents it. An attacker cannot expand into unsupported regions if an SCP blocks it." [7]

rationale: per-role denies require maintenance on each role; SCPs apply org-wide.

---

## claims: API behavior

### [FACT] DuplicatePolicyException exists for same-name policies

> "DuplicatePolicyException: A policy with the same name already exists." [8]

this implies AWS attempts to enforce name uniqueness at creation time.

### [KHUE] does AWS enforce policy name uniqueness?

the CreatePolicy API returns `DuplicatePolicyException` for same-name policies [8], but AWS Control Tower has been observed to create duplicate SCPs [9]. need to verify whether this is a soft or hard constraint.

### [FACT] DuplicatePolicyAttachmentException prevents double-attach

> "DuplicatePolicyAttachmentException: The selected policy is already attached to the specified target." [10]

this makes AttachPolicy idempotent for the same policy-target pair.

### [FACT] minimum 1 SCP per entity required

> "Every entity must have _at least_ one SCP attached at all times when you enable SCPs. You can't remove the last SCP from an entity." [11]

### [FACT] detach required before delete

> "Before you perform this operation, you must first detach the policy from all organizational units (OUs), roots, and accounts." [12]

> "PolicyInUseException: The policy is attached to one or more entities." [12]

### [SUMP] API operations are not explicitly idempotent

the AWS Organizations API documentation does not explicitly guarantee idempotency for CreatePolicy. findsert pattern (lookup then create if absent) is a client-side implementation concern.

## claims: limits and constraints

### [FACT] max 5,120 characters per SCP document

> "Maximum size of a policy document: Service control policies: 5120 characters" [11]

### [FACT] max 5 SCPs per entity (direct attachment)

> "Root: Min 1, Max 5. Organizational Unit (OU): Min 1, Max 5. Account: Min 1, Max 5." [11]

### [FACT] inherited SCPs do not count against limit

> "Policies that affect an OU or account by inheritance do **_not_** count against these limits." [11]

### [FACT] max 10,000 SCPs per organization

> "Number of policies of each type in an organization: Service control policies: 10,000" [11]

### [FACT] whitespace counts if saved via SDK

> "If you save the policy via an SDK operation or the AWS CLI, then the policy is saved exactly as you provided and no automatic removal of characters occurs." [11]

## claims: best practices

### [OPIN] deny-list approach preferred over allow-list

> "Use deny-list for most organizations. AWS launches new services constantly. With a deny-list approach, new services are automatically available unless explicitly blocked." [13]

convergence: sources [6][7][13][14] agree deny-list is better for most organizations.

### [OPIN] explicit deny statements preferred

> "AWS updated its SCP guidance in 2025 to emphasize: Use explicit Deny statements rather than rely on implicit denies." [13]

### [OPIN] test SCPs before production attachment

> "AWS strongly recommends that you don't attach SCPs to the root of your organization without thorough tests of the impact that the policy has on accounts." [1]

### [OPIN] keep OU depth to 3-4 levels

> "It's desirable to keep the depth of OUs simple, not exceed 3-4 levels." [6]

---

## claims: anti-patterns

### [OPIN] conflicted SCPs cause unexpected deny behavior

> "Conflicted or overlapped SCPs can cause unexpected Deny behavior, and the fix is to consolidate or merge policies." [6]

### [OPIN] global service exemptions must not be omitted

> "Omitted global service exemptions can cause IAM, Route 53, and CloudFront to stop work, and the fix is to use NotAction for global services." [6]

### [OPIN] do not remove FullAWSAccess without replacement

> "You should not remove the **FullAWSAccess** policy unless you modify or replace it with a separate policy with allowed actions, otherwise all AWS actions from member accounts will fail." [1]

---

## claims: 2025 updates

### [FACT] full IAM policy language support added September 2025

> "In September 2025, AWS Organizations now offers full IAM policy language support for service control policies (SCPs)." [13]

new capabilities:
- conditions in Allow statements
- individual resource ARNs
- NotAction element with Allow statements
- wildcards anywhere in Action element strings
- NotResource element [15]

---

## claims: infrastructure as code

### [FACT] SCPs can be managed as code via Terraform or CloudFormation

> "Manage AWS Organizations' Service Control Policies (SCPs) as infrastructure as code (IaC) through CI/CD pipelines via AWS CloudFormation or HashiCorp Terraform can reduce the burden." [16]

### [FACT] AWS provides SCP management reference architecture

> "AWS provides reference architectures to deploy SCPs through CI/CD pipelines via Terraform as the Infrastructure-as-Code tool." [17]

### [OPIN] declarative management reduces operational burden

> "You create, delete, and update authorization policies via manifest files (scp-management.json and rcp-management.json)." [16]

---

## claims: delegated administration

### [FACT] delegated administrators are subject to SCPs

> "Unlike the management account, delegated administrator accounts are subject to SCPs. A misconfigured SCP can block the delegated administrator from its duties." [18]

### [SUMP] management account should have minimal direct use

rationale: since SCPs don't apply to management account, it represents a security boundary exception. industry practice is to minimize direct use and delegate to member accounts where possible.

---

## claims: security value

### [OPIN] SCPs prevent breaches before they happen

> "These policies represent real-world lessons learned from security incidents, cost overruns, and compliance failures across dozens of organizations." [7]

### [KHUE] what breaches have SCPs actually prevented?

search results do not contain specific documented cases of breaches prevented by SCPs. the security value is inferred from the preventive control model rather than publicized incidents.

### [FACT] SCP changes take effect immediately

> "If the policy that is detached is a service control policy (SCP), the changes to permissions for AWS Identity and Access Management (IAM) users and roles in affected accounts are immediate." [19]

---

## convergence signals

**sources [1][2][3][4][6] agree:** SCPs and IAM permissions work via intersection — both must allow for action to succeed.

**sources [1][5][6][18] agree:** management account is exempt from SCPs.

**sources [6][7][13][14] agree:** deny-list strategy is preferred for most organizations.

**sources [8][10][12] agree:** API provides clear error types for constraint violations.

---

## conflict detection

### policy name uniqueness

**partial conflict:** source [8] shows `DuplicatePolicyException` exists, but source [9] reports AWS Control Tower can create duplicate-named SCPs. likely explanation: the exception is thrown at CreatePolicy time, but Control Tower may use different code paths or older SCPs may predate the constraint.

---

## open questions for vision

### [KHUE] should findsert detect and fail on duplicate names?

if AWS does not strictly enforce uniqueness, multiple SCPs with the same name could exist. should findsert silently pick one, or fail loud?

recommendation from vision: fail loud on duplicates [20].

### [KHUE] should upsert be supported?

UpdatePolicy API exists and is atomic. is upsert needed for v1, or is findsert + delete sufficient?

recommendation from vision: defer upsert to later [20].

### [KHUE] is OU support in scope?

wish mentions OU support, but DeclaredAwsOrganizationUnit does not exist. is OU attachment required for v1?

recommendation from vision: defer OU to separate wish [20].

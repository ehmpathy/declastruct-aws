# vision: DeclaredAwsLambda should auto-detect code changes

## the outcome world

### what does a day-in-the-life look like with this in place?

a developer modifies their lambda's source code, rebuilds the zip, and runs `npx declastruct plan`. the plan shows `UPDATE` with a diff that shows `code.hash` changed. they apply the plan, and the new code deploys.

### what's the before/after contrast?

| before | after |
|--------|-------|
| change lambda code, rebuild zip | change lambda code, rebuild zip |
| run `declastruct plan` | run `declastruct plan` |
| plan shows `KEEP` (code change invisible) | plan shows `UPDATE` with `code.hash` diff |
| developer confused, manually forces update | developer reviews diff, applies |

### what's the "aha" moment where the value clicks?

the moment a developer changes their lambda code and sees `declastruct plan` correctly detect the change via `code.hash` diff.

## user experience

### what usecases do folks fulfill?

1. **deploy code changes** â€” modify lambda source, rebuild zip, plan shows UPDATE, apply deploys new code
2. **verify no drift** â€” run plan, see KEEP, confirm lambda code matches expectations
3. **version pins** â€” publish immutable versions with explicit `hash: { code, config }` fingerprint

### what contract inputs & outputs do they leverage?

**lambda declaration with explicit hash:**
```ts
const zipUri = './dist/lambda.zip';
const lambda = DeclaredAwsLambda.as({
  name: 'my-lambda',
  runtime: 'nodejs20.x',
  handler: 'index.handler',
  timeout: 30,
  memory: 128,
  role: RefByUnique.as<typeof DeclaredAwsIamRole>(lambdaRole),
  envars: { ENV: 'prod' },
  code: {
    zipUri,
    hash: calcAwsLambdaCodeHash({ of: { zipUri } }),
  },
  tags: { app: 'my-app' },
});
```

**version declaration reuses lambda's code hash:**
```ts
const lambdaVersion = DeclaredAwsLambdaVersion.as({
  lambda: RefByUnique.as<typeof DeclaredAwsLambda>(lambda),
  hash: {
    code: lambda.code!.hash,
    config: calcAwsLambdaConfigHash({ of: lambda }),
  },
});
```

## mental model

### how would users describe this to a friend?

> "declastruct computes a hash of my lambda zip and compares it against what's deployed. if they differ, it knows to update."

### what analogies or metaphors fit?

- **git diff for lambda code** â€” just like git knows a file changed by content hash comparison, declastruct knows lambda code changed by code.hash comparison
- **content-addressed updates** â€” the zip content determines whether an update is needed

### what terms would they use vs what terms would we use?

| user term | our term |
|-----------|----------|
| "the code changed" | `code.hash` differs |
| "the config changed" | `config.hash` differs (memory, timeout, etc.) |
| "same version" | both `hash.code` and `hash.config` match |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| detect code changes | yes â€” via `code.hash` comparison |
| detect config changes | yes â€” via `config.hash` comparison |
| version deduplication | yes â€” versions keyed by `hash: { code, config }` |

### what are the pros? the cons?

**pros:**
- explicit hash computation â€” no magic, clear what's compared
- composable utilities â€” `calcAwsLambdaCodeHash`, `calcAwsLambdaConfigHash`
- format consistency â€” hex hashes via `hash-fns`, converted to/from base64 for AWS

**cons:**
- user must call `calcAwsLambdaCodeHash()` for lambda and version declarations
- requires local zip file at plan time

### what edgecases exist and how do our contracts keep users in a pit of success?

| edgecase | how we handle |
|----------|---------------|
| `code.zipUri` doesn't exist | `calcAwsLambdaCodeHash` throws clear error on read |
| `code.hash` not computed | type error â€” `hash: Hash` is required, not nullable |
| different code, same config | new version (code hash differs) |
| same code, different config | new version (config hash differs) |
| both same | return extant version (idempotent) |

## open questions & assumptions

### what assumptions have we made?

1. **local zip path** â€” `code.zipUri` is always a local file path
2. **sha256 is sufficient** â€” collision-resistant for our purposes
3. **hex format** â€” consistent with `hash-fns` ecosystem

### what questions have been answered?

1. **why was code change not detected?**
   - `codeSha256` was in `readonly` array, excluded from `omitReadonly()` comparison

2. **why restructure to `code: { zipUri, hash }`?**
   - groups related properties
   - `hash` is required â€” forces explicit computation upfront via `calcAwsLambdaCodeHash`

3. **why explicit `calcAwsLambdaCodeHash` / `calcAwsLambdaConfigHash`?**
   - clear name per domain (not generic `calcCodeSha256`)
   - composable for version declarations

## groundwork

### external research

- **AWS Lambda CodeSha256**: base64-encoded sha256 (44 chars with pad)
- **hash-fns format**: hex-encoded sha256 (64 chars)
- **conversion**: `asHashFromBase64()` and `asBase64FromHash()` utilities

### internal research

**root cause (the bug):**

`src/domain.objects/DeclaredAwsLambda.ts` (before):
```ts
public static readonly = ['codeSize', 'codeSha256'] as const;
```

`codeSha256` in `readonly` meant `omitReadonly()` excluded it from comparison â€” code changes were invisible.

**fix:**

restructured to `code: { zipUri, hash }` where `hash` is NOT in readonly. comparison now sees hash changes.

**utilities created:**

- `calcAwsLambdaCodeHash({ of: { zipUri } })` â€” compute sha256 of zip file, return as hex Hash
- `calcAwsLambdaConfigHash({ of: lambda })` â€” compute sha256 of config properties
- `asHashFromBase64(base64)` â€” convert AWS base64 to hex Hash
- `asBase64FromHash(hash)` â€” convert hex Hash to AWS base64

## what is awkward?

### what feels off or forced?

users must explicitly call `calcAwsLambdaCodeHash()` for both lambda and version declarations. not fully automatic, but explicit is better than magic.

### where does the design fight the user's mental model?

users might expect `code.hash` to auto-compute. instead, they must compute it explicitly upfront. this is intentional â€” domain objects shouldn't do I/O, and change detection requires explicit declaration.

### what tradeoffs feel uncomfortable?

**explicit vs automatic**: chose explicit computation over automatic (via DAO hydrate or getter). tradeoff: more user code, but no hidden I/O in domain objects.

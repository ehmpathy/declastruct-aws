# reflection: root cause

## what is the symptom?

### observable problem

platform engineers cannot programmatically manage AWS account aliases via declastruct-aws. they must either:
- use terraform separately from declastruct
- set aliases manually via AWS console

### how it manifests

- inconsistent toolchain: auth resources in declastruct, aliases in terraform
- manual steps in account provision workflow
- risk of drift: terraform state and AWS state can diverge

### when it occurs

- every time a new AWS account is provisioned
- every time an account needs rename (environment rebrand, org restructure)

---

## 5 whys tree

### branch 1: why can't platform engineers manage aliases via declastruct?

1. **why can't they manage aliases via declastruct?**
   → declastruct-aws doesn't expose account alias operations

2. **why doesn't declastruct-aws expose alias operations?**
   → no one has implemented `DeclaredAwsAccountAlias` domain object and operations

3. **why hasn't anyone implemented it?**
   → it wasn't prioritized; other resources (organizations, IAM, SSO) took precedence

4. **why was it lower priority?**
   → accounts can function without aliases; it's quality-of-life, not critical functionality

5. **why is it needed now?**
   → **inconsistent toolchain pain: terraform for aliases, declastruct for all else. team wants unified infrastructure-as-code.**

### branch 2: why do teams want readable aliases at all?

1. **why do teams want readable aliases?**
   → 12-digit account ids are hard to remember and distinguish

2. **why are account ids hard to distinguish?**
   → they're arbitrary numbers with no semantic content

3. **why does that matter?**
   → wrong-account mistakes are costly (apply changes to prod instead of prep)

4. **why do wrong-account mistakes happen?**
   → humans can't visually verify which account they're in from the id alone

5. **why is visual verification important?**
   → **human error prevention: clear visual indicator prevents costly mistakes**

### branch 3: why is the current workaround (terraform) insufficient?

1. **why is terraform insufficient?**
   → it works, but splits infrastructure management across two tools

2. **why is split toolchain bad?**
   → cognitive overhead, maintenance burden, drift risk

3. **why does drift risk matter?**
   → terraform state and actual AWS state can diverge silently

4. **why can't terraform handle this alone?**
   → team chose declastruct for auth resources; want consistency

5. **why did they choose declastruct?**
   → **declarative domain model with idempotent operations fits their mental model better than HCL for auth resources**

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| surface | can't manage aliases via declastruct | no — symptom |
| mid | declastruct-aws lacks alias support | no — gap to fill |
| mid | terraform workaround splits toolchain | no — consequence of gap |
| deep | 12-digit account ids lack semantic content | no — AWS design, out of scope |
| **root** | **human error prevention requires visual disambiguation** | **yes** |
| **root** | **unified toolchain desire requires feature parity** | **yes** |

---

## recommendation

### address root cause at the right level

we cannot change AWS account id design (that's AWS's domain). but we can:

1. **enable human error prevention** — expose alias management in declastruct, so teams can ensure every account has a readable identifier

2. **enable unified toolchain** — add alias support, so declastruct-aws achieves feature parity for auth-related account configuration

### what we build

`DeclaredAwsAccountAlias` with `getAccountAlias`, `setAccountAlias`, `delAccountAlias` operations.

this directly addresses both root causes:
- human error prevention: aliases become easy to provision
- unified toolchain: no more terraform detour for aliases

### what we don't build

- auto-suggestion of unique alias names (out of scope)
- multi-alias support (AWS doesn't support it)
- alias enforcement policy (org-level concern, not declastruct scope)

# vision: account alias support

## the outcome world

### day-in-the-life

**before**: you deploy infrastructure to an AWS account. the console header shows "123456789012". you're 90% sure it's the prep account, but you double-check by way of IAM settings. or worse — you don't check, and you accidentally apply changes to prod.

**after**: every account has a human-readable alias. the console shows "ahbode-prep" in the header. the sign-in url is `https://ahbode-prep.signin.aws.amazon.com`. you know at a glance where you are.

to set this up takes one line in your declastruct config:

```ts
await setAccountAlias({ upsert: { alias: 'ahbode-prep' } }, context);
```

> **critical constraint**: aliases must be unique within your AWS partition. if another AWS account in commercial AWS already uses `ahbode-prep`, you cannot use it. always namespace with your org name (e.g., `ahbode-prep` not just `prep`).

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| console header | `123456789012` | `ahbode-prep` |
| sign-in url | `https://123456789012.signin.aws.amazon.com` | `https://ahbode-prep.signin.aws.amazon.com` |
| management | terraform or manual | declastruct, consistent with other auth resources |
| cognitive load | memorize account ids | names are self-evident |

### the "aha" moment

> "wait, i can just read the console header and know which account i'm in? and the sign-in bookmark tells my team which environment they enter?"

safety through clarity. no more account id memorization. no more "which one is 789012 again?"

## user experience

### usecases

1. **set alias for new account** — org creates account, immediately assigns alias
2. **update alias** — rebrand or rename environment (rare but supported)
3. **verify alias** — read current alias to confirm state
4. **remove alias** — revert to account-id-only (rare)

### contract inputs & outputs

```ts
// get current alias for authenticated account
const alias = await getAccountAlias({ by: { auth: true } }, context);
// => { account: { id: '123456789012' }, alias: 'ahbode-prep' } | null

// set alias (upsert semantics) — account derived from auth context
const result = await setAccountAlias(
  { upsert: { alias: 'ahbode-prep' } },
  context,
);
// => { account: { id: '123456789012' }, alias: 'ahbode-prep' }

// delete alias for authenticated account
await delAccountAlias({ by: { auth: true } }, context);
```

### domain object

```ts
/**
 * a human-readable alias for an AWS account
 *
 * .what
 * - replaces the 12-digit account id in console header and sign-in url
 * - e.g., `https://ahbode-prep.signin.aws.amazon.com`
 *
 * .constraints
 * - **partition-unique**: the alias must be unique across ALL aws accounts
 *   in your partition (aws, aws-cn, or aws-us-gov). if another org already
 *   uses "dev", you cannot use "dev". always namespace with your org name.
 * - 3-63 characters
 * - lowercase letters, digits, hyphens only
 * - must start and end with letter or digit
 * - no consecutive hyphens
 *
 * .note
 * - despite the use of IAMClient, this is account metadata, not an IAM identity
 * - one alias per account (singleton pattern)
 */
export interface DeclaredAwsAccountAlias {
  /**
   * the account this alias belongs to
   *
   * .note = one alias per account (1:1 relationship)
   */
  account: RefByPrimary<typeof DeclaredAwsOrganizationAccount>;

  /**
   * the account alias
   *
   * .note = partition-unique: this must be unique across all AWS accounts
   *         in your partition, not just your org. use namespaced names like
   *         "ahbode-prep" not generic names like "prep" or "dev".
   */
  alias: string;
}

export class DeclaredAwsAccountAlias
  extends DomainEntity<DeclaredAwsAccountAlias>
  implements DeclaredAwsAccountAlias
{
  /**
   * .what = alias is primary because it's partition-unique and serves as the identifier
   * .note = like an AWS-assigned ID, you reference the alias by its alias string
   */
  public static primary = ['alias'] as const;

  /**
   * .what = account is unique because each account can have at most one alias
   * .note = enforces the 1:1 relationship — you can't create two aliases for the same account
   */
  public static unique = ['account'] as const;
}
```

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| human-readable identifier | yes | alias shows in console header |
| prevent wrong-account mistakes | yes | visual confirmation at a glance |
| friendly sign-in url | yes | `https://{alias}.signin.aws.amazon.com` |
| consistent with declastruct | yes | same get/set/del pattern |

### pros

- simple: one attribute, one operation each
- safe: upsert semantics, idempotent
- consistent: follows declastruct patterns exactly
- valuable: prevents costly mistakes

### cons

- partition-unique: alias must be unique within your AWS partition (aws, aws-cn, or aws-us-gov)
- character constraints: 3-63 chars, lowercase, hyphens, no consecutive hyphens
- singleton: only one alias per account (this is actually fine — it's the intended design)

### edge cases and pit of success

| edge case | how we handle it |
|-----------|------------------|
| alias already taken in partition | aws returns error, we surface it clearly |
| invalid characters | validation at domain object level before api call |
| no alias set | `getAccountAlias` returns `null` |
| upsert when alias exists | delete old, create new (aws limitation) |

## open questions & assumptions

### assumptions

1. **one alias per account** — confirmed via aws docs
2. **partition-unique** — confirmed; alias must be unique within your aws partition (aws, aws-cn, aws-us-gov)
3. **iam client is correct** — confirmed; despite account-level scope, managed via iam api
4. **upsert = delete + create** — aws doesn't have an update command; to change alias, must delete then create

### decided questions

2. **[answered] should we include account reference in domain object?**

   **no**. aws api constraint: you can only manage your own account's alias. cross-account capability does not exist, so an account reference would be meaningless. follow the `DeclaredAwsOrganization` singleton pattern — `by: { auth: true }`.

### external research needed

none additional — api constraints are well-documented.

## what is awkward?

### the iam prefix debate

aws puts this in the iam namespace (`IAMClient`), but it's not an iam identity — it's account metadata. to call it `DeclaredAwsIamAccountAlias` feels like a leak of implementation detail into the domain model.

**resolution**: prefer `DeclaredAwsAccountAlias`. the domain model should reflect what things *are*, not which sdk manages them.

### upsert requires delete+create

aws doesn't have an "update alias" api. to change an alias, you must delete the old one and create the new one. this creates a brief window where no alias exists.

**resolution**: accept this limitation. document it. the window is milliseconds; no practical impact.

### partition-unique constraint

the alias must be unique across all aws accounts in your partition (aws, aws-cn, or aws-us-gov), not just yours. this means:
- you can't use generic names like "dev" or "prod"
- you should namespace with org/company name: "ahbode-dev", "mycompany-prod"

**resolution**: document the constraint. users will naturally discover this when common names fail.

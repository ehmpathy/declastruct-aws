# reflection: premortem

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

### most plausible failure scenarios

1. **partition-unique collision not surfaced clearly** — user tries to set alias "mycompany-prod", it fails with cryptic AWS error. user doesn't understand why. they give up or file a confused bug report. we didn't validate alias availability before the call, and we didn't translate the AWS error into an actionable message.

2. **upsert deletes alias but create fails** — user calls `setAccountAlias({ upsert: { alias: 'new-alias' } })`. we delete the old alias successfully, but `CreateAccountAlias` fails (rate limit, transient error, or alias already taken by another org milliseconds before). account is now alias-less. user panics.

3. **validation mismatch with AWS** — our alias validation regex accepts a string, but AWS rejects it. or vice versa: we reject a valid alias. user's alias works in terraform but not in declastruct, which causes frustration and distrust.

---

## what obvious signs did we ignore?

- **aws api doesn't support atomic upsert** — the two-step delete+create is inherently non-atomic. we accepted this risk without serious consideration of recovery patterns.

- **partition-unique is poorly understood** — users expect uniqueness within their org, not across all AWS accounts in the partition. the external research flagged this, but we might not communicate it clearly enough.

- **no integration test for the actual alias** — we can test create/delete/list, but verification that the alias appears in console header requires manual inspection. we might ship a feature that works via API but doesn't render correctly.

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| IAM client region doesn't matter (IAM is global) | SDK might reject or route differently based on region | low |
| alias validation regex matches AWS exactly | mismatch causes accept-then-reject or reject-then-accept confusion | medium |
| upsert's delete+create window is imperceptible | concurrent requests or rate limits cause visible gap | low |
| `EntityAlreadyExistsException` means alias taken in partition | could mean an alternative we didn't anticipate | low |
| users will namespace their aliases | users will try "prod" and get frustrated when it fails | high |

---

## what is the nightmare scenario?

### worst plausible outcome

user runs `setAccountAlias({ upsert: { alias: 'new-name' } })` on their production account:

1. `delAccountAlias` succeeds — production account loses its alias
2. `CreateAccountAlias` fails — rate limit, or `new-name` was taken by another org between validation and create
3. production account is now alias-less
4. team bookmarks to `https://ahbode-prod.signin.aws.amazon.com` stop functional
5. sign-in confusion ensues; some team members can't find the prod account
6. worst: team member makes changes to wrong account because visual indicator is gone

**what would make us deeply regret this?**
- production incident caused by our upsert implementation
- team loses access patterns to critical account

**what would cause the most damage?**
- visible outage or security incident traced back to declastruct

**what would be hardest to recover from?**
- trust loss: "declastruct broke our prod account's alias"

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| upsert fails between delete and create | wrap in try-catch, attempt rollback (re-create old alias) if create fails | medium: adds complexity, rollback might also fail |
| alias validation mismatch | copy exact AWS regex pattern from API documentation, add integration test with edge cases | low: straightforward |
| users try generic names like "prod" | document partition-unique constraint prominently, suggest namespace pattern in error message | low: documentation only |
| `EntityAlreadyExistsException` unclear | translate error: "alias 'X' is already in use by another AWS account in this partition. try a namespaced alias like 'yourorg-X'" | low: error message enhancement |
| sign-in URL breaks in upsert window | document the brief window; if critical, recommend set-if-different pattern (check first, only upsert if alias actually needs to change) | low: documentation + optional optimization |

## risk summary

| risk | severity | likelihood | mitigation priority |
|------|----------|------------|---------------------|
| upsert non-atomic (delete succeeds, create fails) | high | low | high — add rollback attempt |
| partition-unique collision unclear | medium | high | high — clear error message |
| validation mismatch | medium | medium | medium — copy exact AWS regex |
| users try generic names | low | high | low — documentation |

---

## action items for blueprint

1. **error translation** — `EntityAlreadyExistsException` must explain partition-unique constraint
2. **rollback on create failure** — if delete succeeds but create fails, attempt to restore old alias
3. **exact validation** — copy AWS regex pattern verbatim, test edge cases
4. **documentation** — warn about partition-unique, recommend namespace pattern

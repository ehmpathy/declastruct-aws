# evaluation: account alias support

evaluation of what was implemented against the blueprint.

---

## summary (as implemented)

implemented `DeclaredAwsAccountAlias` domain object with `getOneAccountAlias`, `setAccountAlias`, and `delAccountAlias` operations.

key design decisions followed:
- **singleton pattern**: one alias per account, retrieved via `by: { auth: true }`
- **no account reference**: aws api constraint вЂ” can only manage own account's alias
- **upsert semantics**: setAccountAlias uses upsert for rebrand use case
- **delete-then-create**: aws has no update api, so upsert = delete old + create new
- **validation first**: validate alias format before any aws api call
- **del returns confirmation**: `delAccountAlias` returns `{ deleted: true }`

additional implementation detail (not in blueprint):
- **named transformer extracted**: `asAccountAliasFromListResponse` extracted per rule.forbid.inline-decode-friction

## divergence analysis

### divergences found

| section | blueprint declared | actual implemented | divergence type |
|---------|-------------------|-------------------|-----------------|
| filediff | 10 files in domain.operations/accountAlias/ | 11 files | added |
| codepath | inline array access in getOneAccountAlias | extracted to asAccountAliasFromListResponse | added |
| codepath | delAccountAlias error paths | added NoSuchEntityException handler | hardened |
| codepath | DeclaredAwsAccountAlias has runtime validation (via zod or joi) | no built-in validation (delegated to validateAccountAliasFormat) | simplified |
| codepath | validateAccountAliasFormat uses single regex | uses separate checks for clearer error messages | improved |
| tests | partition collision test in validateAccountAliasFormat.test.ts | not implemented (blueprint misplacement вЂ” validateAccountAliasFormat is pure) | omitted |

### divergence resolution

| divergence | resolution | rationale |
|------------|------------|-----------|
| `asAccountAliasFromListResponse.ts` added | **retain** | extracted per rule.forbid.inline-decode-friction; improves code quality without altered behavior; blueprint did not anticipate this extraction but it aligns with architectural rules |
| `__snapshots__/` directory | **retain** | auto-generated by jest for acceptance test snapshots; expected artifact |
| `NoSuchEntityException` handler in delAccountAlias | **retain** | hardened idempotency for race condition where alias is deleted between lookup and delete; improves reliability |
| domain object validation | **retain** | validation at operation level (validateAccountAliasFormat) achieves same goal; domain object trusts pre-validated input per blueprint note |
| separate validation checks | **retain** | separate checks produce detailed error messages (which constraint violated) vs single regex (just "invalid"); improves user experience |
| partition collision test | **acknowledge** | blueprint placed test under validateAccountAliasFormat but this is a pure validator; partition collision occurs in setAccountAlias via EntityAlreadyExistsException; the error path exists in code but is untestable without a known-taken alias from another account; blueprint acknowledged this as "untestable in integration" |

**note**: both divergences are ADDITIONS that improve the implementation. neither removes or alters declared functionality. the blueprint declared the *what*, execution discovered the *how* required an additional transformer to satisfy code quality rules.

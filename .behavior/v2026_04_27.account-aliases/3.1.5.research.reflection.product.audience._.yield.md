# reflection: audience

## who is the audience?

### primary: platform engineers at multi-account AWS organizations

**who they are**:
- engineers who manage AWS infrastructure via IaC (terraform, cdk, declastruct)
- work with 3+ AWS accounts (dev, prep, prod at minimum)
- responsible for account provisioning, access management, security guardrails

**their context**:
- switch between accounts multiple times daily
- console header shows 12-digit account ids by default
- risk of accidental changes to wrong account is real and costly
- already use declastruct-aws for other AWS resource control

### secondary: developers who access AWS console

**who they are**:
- application developers who occasionally access AWS console for debugging
- may not remember which account id maps to which environment
- rely on visual cues to confirm they're in the right place

**their context**:
- infrequent AWS console use (daily to weekly)
- bookmark sign-in URLs for quick access
- cognitive load: remembering account ids is error-prone

### tertiary: security and compliance teams

**who they are**:
- auditors who review AWS configurations
- security engineers who verify access patterns
- compliance officers who trace actions to accounts

**their context**:
- need to quickly identify which account logs belong to
- readable identifiers simplify audit trails
- cloudtrail entries tagged with alias improve readability

## why do they care?

### primary: platform engineers

| motivation | detail |
|------------|--------|
| **prevent costly mistakes** | wrong-account changes can cause outages, data loss, or security incidents |
| **reduce cognitive load** | no need to memorize 12-digit account ids |
| **consistency** | alias management via same declastruct patterns they already use |
| **team onboarding** | new team members can understand account structure at a glance |

### secondary: developers

| motivation | detail |
|------------|--------|
| **confidence** | "am i in prod?" answered by glancing at console header |
| **friendly bookmarks** | `https://ahbode-prep.signin.aws.amazon.com` is self-documenting |
| **reduced anxiety** | clear visual indicator reduces fear of wrong-account actions |

### tertiary: security/compliance

| motivation | detail |
|------------|--------|
| **audit clarity** | cloudtrail logs tagged with readable alias |
| **faster review** | no mental translation from account id to environment |
| **documentation** | compliance docs reference aliases instead of ids |

---

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| platform engineers | high | medium | operational safety, reduced mistakes |
| developers | medium | low | convenience, confidence |
| security/compliance | low | low | audit readability |

**note**: urgency is medium (not high) because:
- accounts can function without aliases
- workaround exists (manual setup via terraform or console)
- this is a quality-of-life improvement, not a blocker

---

## what do they care about?

### success criteria (what they judge by)

| audience | criteria |
|----------|----------|
| platform engineers | alias sets correctly, idempotent, consistent with other declastruct operations |
| developers | console header shows alias, sign-in URL works |
| security/compliance | alias appears in cloudtrail, audit tools |

### tradeoffs they accept

| acceptable | unacceptable |
|------------|--------------|
| brief alias-less window during upsert | broken alias that shows wrong name |
| namespace requirement (ahbode-prep not prep) | silent failure when alias taken |
| only one alias per account | complex multi-alias workflow |

### delight vs disappointment

| delighted | disappointed |
|-----------|--------------|
| single line: `setAccountAlias({ upsert: { alias } })` | verbose multi-step process |
| clear error when alias taken in partition | cryptic AWS error message |
| validation before AWS call | invalid alias accepted, fails at AWS |
| consistent with getOne/set/del pattern | different API shape |

## how will they experience it?

### platform engineer journey

1. **discover** — reads declastruct-aws docs or changelog, sees account alias support
2. **integrate** — adds `setAccountAlias` call to account provisioning procedure
3. **run** — executes provisioning, alias is set
4. **verify** — opens AWS console, confirms alias in header
5. **maintain** — occasional updates (rebrand, environment rename)

### developer journey

1. **receive** — gets notified of friendly sign-in URL from platform team
2. **bookmark** — saves `https://ahbode-prep.signin.aws.amazon.com`
3. **use** — opens console via bookmark, sees alias in header
4. **trust** — glances at header to confirm environment before actions

### touchpoints

| touchpoint | experience |
|------------|------------|
| SDK call | `setAccountAlias({ upsert: { alias: 'ahbode-prep' } })` |
| success response | `{ account: { id: '...' }, alias: 'ahbode-prep' }` |
| console header | "ahbode-prep" instead of "123456789012" |
| sign-in URL | `https://ahbode-prep.signin.aws.amazon.com` |
| error: alias taken | clear message: "alias 'ahbode-prep' already in use in partition" |

### friction points

| friction | mitigation |
|----------|------------|
| alias taken by another org in partition | namespace with org name; clear error message |
| invalid alias format | validate before AWS call; explain constraints in error |
| upsert window (delete → create) | milliseconds; no practical impact |

### smooth experience

1. platform engineer adds one line to provisioning procedure
2. alias is set on first run, idempotent on reruns
3. developers access via friendly URL, see alias in header
4. no ongoing maintenance required unless rename needed

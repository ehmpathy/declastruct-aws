# cancellation: account aliases

## .decision

cancelled — feature provides zero value for SSO-based workflows.

## .why

### historical context

| feature | launched |
|---------|----------|
| IAM (with aliases) | May 2010 |
| AWS SSO | December 2017 |

account aliases predate SSO by ~7 years. they solved the problem of human-readable sign-in URLs when IAM users were the only option.

### claimed benefits vs SSO reality

| benefit | IAM users | SSO users |
|---------|-----------|-----------|
| friendly sign-in URL | `alias.signin.aws.amazon.com` | moot — SSO portal used instead |
| console header clarity | shows alias | shows role name, not alias |
| multi-account identification | alias helps | account name (Organizations) in picker |

### key insight

with SSO, the console header top-right shows the **role name**, not the account alias. the alias only appears when signed in via IAM (the alias sign-in URL).

the account name (set in Organizations) is what shows in the SSO account picker — not the alias.

### only use case left

the only scenario where aliases provide value:
- **break-glass IAM users** — emergency accounts that bypass SSO

without break-glass IAM users, this feature provides zero value.

## .recommendation

if break-glass IAM users are needed in the future, revisit this feature. otherwise, manage account identification via:
- **account name** (Organizations) — shows in SSO picker
- **SSO portal URL** — already customizable (one-time, console-only)

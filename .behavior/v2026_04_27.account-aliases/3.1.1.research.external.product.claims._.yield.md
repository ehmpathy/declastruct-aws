# research: external product claims

## summary

research on AWS account alias claims, facts, assumptions, questions, and opinions gathered from official documentation, practitioner blogs, terraform registry, and community discussions.

---

## facts

### [FACT] one alias per account

aws accounts support exactly one alias. if a new alias is created, the old one is replaced [1][5][17].

> "Your AWS account can have only one alias" — AWS IAM Documentation [1]

> "if you create a new alias for your AWS account, the new alias overwrites the previous alias, and the URL containing the previous alias stops working" — AWS Account Management [5]

### [FACT] partition-unique, not globally unique

aliases must be unique within the AWS partition (aws, aws-cn, aws-us-gov) [1][5][18].

> "The account alias must be unique across all Amazon Web Services products within a given network partition" — AWS Documentation [1]

note: some practitioner sources claim global uniqueness [2][18], but official AWS documentation specifies partition scope.

### [FACT] alias constraints: 3-63 characters

aliases must follow strict character rules [1][5][6].

> "The alias must be between 3 and 63 characters in length" — AWS Documentation [6]

> "must contain only digits, lowercase letters, and hyphens" — AWS IAM Documentation [1]

> "You cannot start or finish with a dash, nor can you have two dashes in a row" — Bitcot Guide [2]

### [FACT] alias appears in console header

the alias replaces the 12-digit account ID in the AWS console navigation [1][3][5].

> "An account alias replaces the account ID in the sign-in URL" — AWS Documentation [1]

### [FACT] both sign-in URLs remain active

after alias creation, both the numeric and alias URLs work [1][5].

> "The original URL containing your AWS account ID remains active and can be used after you create your AWS account alias" — AWS Documentation [1]

### [FACT] alias changes are logged in CloudTrail

CreateAccountAlias and DeleteAccountAlias API calls are captured in CloudTrail [9][13].

> "CloudTrail tracks the who, what, where, and when of activity that occurs in your AWS environment" — AWS CloudTrail Documentation [13]

### [FACT] account alias distinct from account name

these are separate concepts in AWS [3][5][12].

> "Account Name and Account Alias in AWS are not the same thing!" — Tymik.me Blog [3]

> "account name is displayed in the console navigation bar, billing documents, and Organizations hierarchy" while "account alias is the custom URL used to sign in" — AWS re:Post [12]

### [FACT] alias not available in IAM Identity Center

account aliases are IAM-specific; SSO users cannot set them [10].

> "It is not possible to set an account alias for an AWS account in IAM Identity Center" — AWS re:Post [10]

## assumptions

### [SUMP] aliases improve operational safety

practitioners assume clear aliases reduce wrong-account mistakes [2][4][8].

> "When you assume a role in the AWS web console, you can use the account alias, so instead of cryptic 'Admin @ 123456654321' you see 'Admin @ bboq-dev-website,' which reduces confusion" — Bad Boys of DevOps [4]

note: this assumes humans read the header. no empirical evidence cited.

### [SUMP] naming convention should include environment

multiple sources assume environment indicators help distinguish accounts [2][7][8][14].

> "company-unit-env-purpose... creates a predictable pattern where the environment is always in the same position" — Business Compass LLC [8]

> "Use different aliases for different environments (dev, prep, prod)" — Bitcot Guide [2]

### [SUMP] aliases have zero performance impact

sources state no performance cost without citation [2][8].

> "Account aliases provide significant organizational benefits with zero direct costs and no impact on AWS service performance" — Bitcot Guide [2]

note: likely true but unverified — no AWS performance documentation found.

### [SUMP] generic names will be taken

practitioners assume common names are unavailable due to partition uniqueness [2][14].

> "globally unique, which could result in name conflicts if your preferred alias is already in use" — Accend Networks [18]

### [SUMP] terraform is the standard IaC approach

sources assume terraform for alias management [15][16][17].

> "resource 'aws_iam_account_alias' 'alias' { account_alias = var.account_alias }" — Terraform Registry [15]

## questions

### [KHUE] should aliases match account names?

no clear guidance on whether alias should match AWS Organizations account name [3][12].

> "Account Name and Account Alias in AWS are not the same thing!" — Tymik.me [3]

question: should they be synchronized for consistency, or intentionally different?

### [KHUE] how to handle alias change propagation?

some sources mention propagation delay [12].

> "Changes to an AWS account can take up to four hours to propagate everywhere" — AWS re:Post [12]

question: does this apply to alias changes? what is the actual propagation time?

### [KHUE] what happens to bookmarks when alias changes?

old alias URLs stop working [1][5], but no guidance on migration strategy.

> "the URL containing the previous alias stops working" — AWS Documentation [1]

question: how should teams communicate alias changes to avoid broken bookmarks?

### [KHUE] terraform data source fails on new accounts

terraform has a known issue with accounts without aliases [6].

> "The aws_iam_account_alias data source, when operating in a brand new AWS account where the administrator has not set an IAM alias yet, returns an error" — GitHub Issue [6]

question: should declastruct handle this edge case explicitly?

### [KHUE] EntityAlreadyExists but no alias visible?

users report orphaned aliases [6].

> "system reports that an alias already exists with an error 'EntityAlreadyExists,' but running aws iam list-account-aliases returns an empty list" — AWS re:Post [6]

question: what causes this? how to recover?

## opinions

### [OPIN] aliases are essential for multi-account management

practitioner consensus that aliases are necessary, not optional [2][4][7][8].

> "Account aliases provide significant organizational benefits" — Multiple sources

opinion: strong consensus, though some orgs may function without them.

### [OPIN] 15-25 characters is ideal length

one source recommends specific length [2].

> "keep aliases short (15-25 characters)" — Bitcot Guide [2]

opinion: reasonable but arbitrary; no empirical basis cited.

### [OPIN] alias naming should follow organization-purpose-environment

practitioner preference for consistent naming pattern [7][8][14].

> "consider using a naming standard such as organization-purpose-environment (for example, sales-catalog-prod)" — AWS Best Practices [7]

opinion: one of several valid patterns; alternatives include env-first or flat naming.

### [OPIN] aliases are not a security control

official and practitioner sources agree aliases provide no security [1][5][11].

> "an alias doesn't provide additional security and is purely for convenience" — Accend Networks [11]

opinion: accurate characterization; aliases are UX, not security.

### [OPIN] avoid PII in aliases

AWS recommends not including personal information [7][14].

> "For privacy and security reasons, avoid using account names that reflect personally identifiable information (PII)" — AWS Best Practices [7]

opinion: sensible guidance given public visibility of alias URLs.

### [OPIN] color-coding is better than aliases for account identification

some practitioners suggest browser extensions or themes over aliases [19].

> "enable colour theme for the browser" as alternative identification method — AWS re:Post [19]

opinion: aliases and color-coding serve different purposes; not mutually exclusive.

## conflict detection

**sources [2] and [1] conflict on uniqueness scope:**
- [2] Bitcot claims "globally unique"
- [1] AWS Documentation says "unique within a given network partition"
- resolution: trust official documentation — partition-unique is correct

**sources conflict on Identity Center compatibility:**
- [10] confirms aliases unavailable in Identity Center as of Sept 2024
- older sources may imply broader compatibility
- resolution: treat as IAM-only feature

## anti-patterns

### warn: do not rely on alias for security [1][11]

> "an alias doesn't provide additional security" — Accend Networks [11]

aliases are publicly visible in URLs. do not assume obscurity provides protection.

### warn: do not use generic names [2][18]

> "name conflicts if your preferred alias is already in use" — Accend Networks [18]

always namespace with organization identifier (e.g., `ahbode-dev` not `dev`).

### warn: do not include PII [7][14]

> "avoid using account names that reflect personally identifiable information" — AWS Best Practices [7]

aliases appear in public sign-in URLs.

### warn: do not assume instant propagation [12]

> "can take up to four hours to propagate" — AWS re:Post [12]

plan for delay when changing aliases.

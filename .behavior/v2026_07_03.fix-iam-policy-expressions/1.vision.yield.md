# 1.vision — support resource/action/principal include+exclude via a `Scope` on `DeclaredAwsIamPolicyStatement`

> wish: execute ehmpathy/declastruct-aws#58

> **decided contract** (wisher call): rather than mirror AWS's raw `NotResource` /
> `NotAction` / `NotPrincipal` fields, model each match field as `bare | Scope`, where
> `Scope = { include?, exclude? }`. a bare scalar/array is the shorthand for
> `{ include: X }` (positive); `{ exclude: X }` maps to the AWS `Not*` element. the
> `Scope` is a nested `DomainLiteral`, so it hydrates cleanly and manipulates safely.
> the bare shorthand survives via two upstream `domain-objects` fixes (see "upstream
> dependency"). one field carries both modes, and the domain object speaks intent while
> the cast speaks AWS. see "the selector contract".

> **revision note**: an earlier draft of this vision modeled exclusion as a `{ except }`
> variant with no `nested` entry for the widened fields. an acceptance test exposed that
> `domain-objects` flags any plain-object field value not declared in `nested` as unsafe
> to manipulate (it crashed `checkAreResourcesEquivalent` on the very first iam role in
> the plan). the fix is to nest the fields as a `Scope` domain object — which in turn
> needs `domain-objects` to leave bare scalars/arrays un-hydrated under a nested key. the
> contract below is the corrected, nested-safe design.

## the outcome world

### day-in-the-life

vlad is hardening the prod `everyday-reader` role in ahbode/infrastructure. the
role attaches the aws-managed `ReadOnlyAccess` policy, which grants
`ssm:GetParameter*` on `*`. he needs to allow reads of ONE ssm path
(`.../cicd/scope=plan/*`) and deny every other ssm parameter.

the canonical AWS idiom is a single deny statement with `NotResource`:

```json
{ "Effect": "Deny", "Action": "ssm:GetParameter*", "NotResource": ["arn:.../cicd/scope=plan/*"] }
```

today he cannot express this in declastruct-aws — `DeclaredAwsIamPolicyStatement`
models only `resource`. so he drops down to hand-written json or terraform, which
breaks the declarative, domain-driven flow the rest of his infra uses.

after this wish: he writes a normal `DeclaredAwsIamPolicyStatement` with
`resource: { except: X }`, and it serializes to the exact iam json AWS expects
(`NotResource`).

### before / after

| | before | after |
|---|--------|-------|
| express "deny all except X" | impossible in declastruct-aws | `resource: { except: '...' }` |
| fallback | raw json / terraform outside declastruct | stays in declastruct |
| silent-drop risk | untyped field would be dropped on serialize | typed + tested round-trip |
| "both set" foot-gun | n/a | impossible by construction (one field, one variant) |

### the "aha" moment

the value clicks when `resource: { except: 'arn:...' }` serializes to
`"NotResource": "arn:..."` with **no** `"Resource"` key — the exclusion idiom is
first-class, structurally safe, and load-critical all the way to the iam api.

## user experience

### usecases / goals

- **primary**: express a `Deny` statement scoped by resource exclusion
  (`resource: { exclude: X }` → `NotResource`) — the deny-all-but-X idiom.
- **uniform**: the same `Scope` shape covers action exclusion (→ `NotAction`) and
  principal exclusion (→ `NotPrincipal`), so one pattern serves all three elements.
- **ergonomic**: bare scalar/array shorthand for the common positive case stays intact
  (`resource: '*'`, `action: ['s3:GetObject', ...]`), so extant statements do not churn.

### the selector contract

each match field is `bare | Scope`, where `Scope` is a nested `DomainLiteral` that holds
`include` XOR `exclude`. bare stays the shorthand — NOT new parallel fields:

```ts
// Scope for string-valued match fields (resource, action)
interface DeclaredAwsIamStatementScope {
  include?: string | string[];  // → Resource / Action
  exclude?: string | string[];  // → NotResource / NotAction
}

// Scope for the principal field (principal payload)
interface DeclaredAwsIamPrincipalScope {
  include?: DeclaredAwsIamPrincipal;  // → Principal
  exclude?: DeclaredAwsIamPrincipal;  // → NotPrincipal
}

// resource: bare shorthand (= include), OR an explicit Scope
resource?: string | string[] | DeclaredAwsIamStatementScope;

// action: same treatment (bare stays required; a Scope carries include/exclude)
action: string | string[] | DeclaredAwsIamStatementScope;

// principal: bare '*', bare principal shorthand (= include), OR an explicit Scope
//   (note: exclude never wraps '*' — AWS NotPrincipal: "*" is nonsensical; guarded)
principal?: '*' | DeclaredAwsIamPrincipal | DeclaredAwsIamPrincipalScope;
```

**four author shapes per field** (e.g. resource): `'*'` · `['a','b']` · `{ include: '*' }`
· `{ exclude: 'a' }`. the two bare forms are shorthand for `{ include: ... }`.

**serialize rule** (`castFrom*`):
- bare value → positive element (`Resource` / `Action` / `Principal`)
- `Scope.include` → positive element; `Scope.exclude` → negated element (`Not*`)

**parse rule** (`castInto*`, both read paths):
- `Resource` → `resource: X` (bare); `NotResource` → `resource: { exclude: X }` (and same
  for action/principal) — round-trip symmetric.

**one runtime guard**: a `Scope` with BOTH `include` and `exclude` set maps to a statement
with both `Resource` and `NotResource`, which AWS forbids — so the write cast fails fast on
a both-set `Scope`. (with the prior `{ except }` design this was type-impossible; the
`Scope` trades that for the explicit-`{ include }` ergonomic, at the cost of one guard.)
`action` stays required (bare or `Scope`), so it does NOT become optional.

### what leverage looks like

```ts
new DeclaredAwsIamPolicyStatement({
  sid: 'DenyNonPlanSsmParameters',
  effect: 'Deny',
  action: 'ssm:GetParameter*',
  resource: { exclude: 'arn:aws:ssm:*:*:parameter/*/svc-*/database/role/cicd/scope=plan/*' },
});
```

→ serializes (via `castFromDeclaredAwsIamPolicyStatement`) to:

```json
{
  "Sid": "DenyNonPlanSsmParameters",
  "Effect": "Deny",
  "Action": "ssm:GetParameter*",
  "NotResource": "arn:aws:ssm:*:*:parameter/*/svc-*/database/role/cicd/scope=plan/*"
}
```

### timeline

1. land the two upstream `domain-objects` fixes (bare-scalar-in-array skip; structural
multi-option nested) → 2. add `DeclaredAwsIamStatementScope` + `DeclaredAwsIamPrincipalScope`
domain objects → 3. nest them on `DeclaredAwsIamPolicyStatement` (`resource`/`action`/
`principal`) → 4. thread through the write cast (`include`→positive, `exclude`→`Not*`,
both-set→fail-fast) → 5. thread through both read casts (`Not*`→`{ exclude }`) → 6. tests:
round-trip for each element across all four author shapes → 7. acceptance coverage.

### upstream dependency

the nested-safe `Scope` design needs two `domain-objects` behaviors that do not exist
today. both are dispatched as separate tasks against `ehmpathy/domain-objects`:

| # | upstream change | unlocks | needed for |
|---|-----------------|---------|-----------|
| 1 | skip hydration for a bare scalar **inside an array** under a nested key (mirror the field-level guard at the element level) | `resource: ['a','b']` bare-array shorthand stays bare, not `[Wrapper('a'), Wrapper('b')]` | resource, action |
| 2 | structural multi-option nested: for a plain object under a multi-option nested key, try each option and keep the one that validates (no `_dobj` tag) | `principal: { service }` bare shorthand coexists with `principal: { include/exclude }` | principal |

fix 1 alone makes `resource`/`action` fully work (single-option nest + bare scalar/array).
fix 2 is only needed to keep the bare `{ service }` principal shorthand; absent it, principal
requires the explicit `{ include: { service } }` form. this vision expects both land, so all
four author shapes work uniformly across resource, action, and principal.

## mental model

- **to a friend**: "it's the `NOT IN (...)` of iam — deny all but this one path."
- **analogy**: a bouncer's list. `resource: [...]` (or `{ include: [...] }`) = "let these
  in". `resource: { exclude: [...] }` = "let all in except these". one field names the
  list and the mode — you name include XOR exclude, never both (the cast enforces it).
- **terms**: users think "all but X"; the `{ exclude: X }` shape says exactly that, and
  `{ include: X }` (or the bare shorthand) says "just these". we speak AWS's `NotResource`
  only at the serialize boundary (the cast), not in the domain object — the domain object
  speaks intent, the cast speaks AWS.

## evaluation

### how well it solves the goal

fully, and more safely than raw `not*` fields. AWS `NotResource` is the ONLY
mechanism for this shape (no ssm condition key for parameter name; a plain `Deny` on
`Resource: *` also blocks the allowed path since explicit deny beats allow). the
`{ except }` variant carries that mechanism end to end while it deletes the whole
"both set" bug class.

### pros

- unblocks the ahbode prod `everyday-reader` scoping without a drop out of declastruct
- **bare shorthand preserved** — extant statements stay `resource: '*'` / `action: [...]`;
  zero churn to the ~25 author sites that never touch exclusion
- **manipulation-safe** — the `Scope` is a nested domain object instance, so declastruct's
  plan/equivalence machinery handles it (the plain-object form crashed on this)
- `action` stays required (no required→optional loosen), no "neither set" guard
- reads as plain intent (`include`/`exclude`), not iam jargon; AWS vocab lives at the cast
- covers all three elements uniformly (resource, action, principal) in one pattern

### cons / tradeoffs

- **diverges from strict AWS-field mirroring** — the cast now decodes `Scope` ↔
  positive/`Not*` instead of a trivial rename; a reader compares against AWS docs via the
  cast, not a literal `notResource` field. (precedent: `condition` is already a nested
  domain object, not a raw passthrough.)
- three cast sites carry the variant (one write, two read) — duplication is extant; we
  extend the same pattern rather than refactor (wet-over-dry; not in scope)
- **an upstream `domain-objects` dependency** — the nested-safe `Scope` needs two library
  fixes to keep the bare shorthand; this wish is gated on them (dispatched as tasks)
- the union type (`bare | Scope`) plus a both-set runtime guard is marginally more surface
  than a single scalar field; the payoff is the manipulation-safety + the bare shorthand

### edgecases & pit-of-success

| edgecase | how we keep the pit of success |
|----------|-------------------------------|
| `Scope` with both `include` and `exclude` | **fail-fast** at the write cast — maps to both `Resource` and `NotResource`, which AWS forbids; the cast rejects it before any AWS call |
| bare array corrupts under a nested key | **prevented upstream** — fix 1 leaves bare scalars inside arrays un-hydrated, so `['a','b']` stays `['a','b']` |
| plain-object field crashes `checkAreResourcesEquivalent` | **prevented** — the field is a nested `Scope` domain object (an instance), which is manipulation-safe; this was the exact acceptance-test crash that drove the revision |
| `notAction` forces `action` optional | avoided — `action` stays required, as bare or `Scope` |
| `NotPrincipal: "*"` (dangerous) | **guarded** — `exclude` wraps a principal object; the cast rejects an exclude of `'*'` or an empty principal |
| read path round-trip | both read casts map `Not*` → `{ exclude }`, so a policy read from AWS re-serializes identically |
| empty exclusion `{ exclude: [] }` | **fail-fast** at the write-cast (serialize) boundary — an empty exclusion is almost certainly a mistake. placed at the cast, not the domain object, since this repo has no `static schema` validation pattern on domain objects; the cast still fires before any AWS call, which meets the intent |

## open questions & assumptions

> **note**: the `{ except }` selector decision (wisher call) settles several prior
> open questions. what it settled and what remains is below.

### assumptions

1. **scope covers all three cast sites**. the issue names only the write path as
   load-critical, but round-trip tests require the two read casts too. read-path
   symmetry is now a decided part of the contract (`Not*` ↔ `{ except }` both ways),
   not an open assumption.
2. **the selector applies uniformly to `resource`, `action`, `principal`.** decided:
   one `{ except }` pattern for all three, rather than three bespoke `not*` fields.
3. **`{ except }` for principal wraps a `DeclaredAwsIamPrincipal`, never `'*'`.** this
   is deliberate — AWS `NotPrincipal: "*"` is dangerous; the type cannot express it.

- ~~Q1 fail-fast placement~~ — **decided: write cast**. the `Scope` CAN hold both
  `include` and `exclude`, which maps to both `Resource` and `NotResource` (AWS forbids),
  so the write cast fails fast on a both-set `Scope`. (this flipped from the earlier
  `{ except }` draft, where both-set was type-impossible.)
- ~~Q2 read-path symmetry~~ — **decided: include**; round-trip is intrinsic to the
  `Scope` contract (`Not*` → `{ exclude }`).
- ~~Q3 include the `not*` trio~~ — **decided: yes, uniformly**, via the one `Scope`
  pattern (not three separate fields; cost is now one pattern, not per-field).
- ~~Q4 `notPrincipal: '*'`~~ — **decided: guarded** at the cast (`exclude` of `'*'` or an

## what is awkward?

- **three cast sites, no shared statement-mapper** — the camel↔Pascal mapping is
  duplicated across one write + two read casts. the `{ except }` variant must land in
  all three, consistently; miss one and it silently drops or fails to round-trip. a
  shared mapper is tempting but is a refactor beyond this wish's scope (wet-over-dry:
  the duplication predates us). we accept the friction and lean on round-trip tests to
  catch drift.
- **the union diverges from AWS-field mirroring** — declastruct's norm is a 1:1
  field-name map to the SDK; `{ except }` breaks that literal correspondence and moves
  the AWS vocab into the cast. this is the real cost of the ergonomic win. mitigated
  by precedent (`condition` is already a richer nested shape) and by the fact that the
  cast is the correct home for AWS-shape knowledge.
- **`{ except }` on `principal` is only valid in trust policies** — the domain object
  is shared between permission and trust policies, so the type permits a principal
  exclusion in contexts AWS would reject. this matches the extant looseness around
  `principal`/`resource` (already optional/context-dependent); we won't newly enforce
  context-specific validity — out of scope, and it would fight the shared shape.
- **`Scope` on the match fields + `nested` hydration** — the fields are nested `Scope`
  domain objects, so they hydrate to instances (manipulation-safe). the ONE subtlety is
  the bare shorthand: bare scalars/arrays must NOT hydrate under a nested key, and a bare
  principal object must pick the right multi-option. both are handled by the two upstream
  `domain-objects` fixes. this vision expects them landed; absent fix 2, principal loses
  only its bare `{ service }` shorthand (the explicit `{ include: { service } }` still
  works). settled — see Q6.

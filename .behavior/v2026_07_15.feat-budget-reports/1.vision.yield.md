# 1.vision — feat-budget-reports

> illustrate the world where, now that budgets + cost guards ship, we can **see**
> where our money goes and where it is **expected** to go — declaratively, as
> read-only cost reports that sit beside the extant budget resources.

---

## the outcome world

### day-in-the-life

**before.** we can *cap* and *guard* spend (feat-budget shipped v1.8.0), but to
answer "where did the $21 go?" or "are we on track to blow the cap this month?" a
human still opens the Cost Explorer console, clicks through date pickers and
group-by dropdowns, and eyeballs a chart. no part is versioned, no part is
reproducible, no part is scriptable. the budget knows the *ceiling*; it cannot tell
you the *composition* or the *trajectory*.

**after.** cost visibility is a declared, read-only resource — you `plan` it and read
the result, exactly like the extant CloudWatch log-group reports:

```bash
npx declastruct plan  --wish provision/aws.cost/account=demo/reports.ts --into .temp/plan.json
```

three new report composites answer the wisher's three questions:

1. **where the money goes** ("where does our budget go?") —
   `DeclaredAwsCostReportSpendObserved` runs `GetCostAndUsage`
   over a range, at a granularity (DAILY/MONTHLY), grouped by a dimension (SERVICE,
   LINKED_ACCOUNT) or a TAG. one read returns *both* the composition (groups) AND the
   trend (time buckets). "EC2 was $12, S3 was $4, and here's each day of the month."
2. **where the money is EXPECTED to go** — `DeclaredAwsCostReportSpendForecast` runs
   `GetCostForecast` to project spend over a future window, with a mean and a
   confidence interval. "at the current trajectory you'll hit ~$19 by month-end,
   80% confidence band $17–$22."
3. **where we can SAVE money** ("where can we save money?") —
   `DeclaredAwsCostReportRecommendationsToRightsize` runs `GetRightsizingRecommendation` to surface idle +
   underutilized EC2, each with a downsize/terminate action and an estimated monthly
   saving, plus a summary total. "3 boxes are oversized; downsize them to save ~$8/mo."

all three are **read-only** — a `getOne` composite, no `set`/`del` — exactly like
`DeclaredAwsCloudwatchLogGroupReportCostOfIngestion`. you never *apply* a report; you
*read* it. the value is that the query (range, group, filter) is declared in a
git-tracked file and the result is a typed domain object you can pipe into anything.

### before/after contrast

| concern                    | before                              | after                                              |
| -------------------------- | ----------------------------------- | -------------------------------------------------- |
| where did the money go?    | click Cost Explorer, eyeball chart  | `DeclaredAwsCostReportSpendObserved` (grouped)             |
| how has spend trended?     | manual date-range compare           | `DeclaredAwsCostReportSpendObserved` (DAILY/MONTHLY buckets) |
| where is spend headed?     | gut feel, or wait for the bill      | `DeclaredAwsCostReportSpendForecast` (mean + interval)  |
| where can we save money?   | read a blog, guess which boxes to resize | `DeclaredAwsCostReportRecommendationsToRightsize` (per-box saving + total) |
| reproducible across accts  | re-click per account                | one per-account `reports.ts`, declared             |
| pipe into alerts/dashboards| screenshot                          | typed `HasReadonly<...>` domain object             |

### the "aha" moment

it clicks when you realize the report is the **same shape** as the log-group reports
already in the repo — a read-only composite that hides a paged SDK call + a cast
behind one clean domain read. we are not inventing a new mechanism; we're pointing the
*exact* report-composite pattern at the Cost Explorer client that the cost-anomaly
resources already use. budgets set the ceiling, guards enforce it, and now **reports
show you the picture** — all declared, all git-tracked, all one `plan` away.

## user experience

### the resource set (canonical — this is the whole feature)

this feature adds **4 new declared resources**, all read-only report composites, in two
symmetric prefix-pairs. the `Spend*` pair (`Observed`/`Forecast`) answers "where does the
money go?"; the `RecommendationsTo*` pair (`Rightsize`/`PurchasePlan`) answers "where can we
save?". each pair shares a noun with a distinguishing suffix (per `rule.require.order.noun_adj`),
so both cluster in autocomplete:

| # | resource | answers (wisher's words) | AWS SDK command | identity |
|---|----------|---------|-----------------|----------|
| 1 | `DeclaredAwsCostReportSpendObserved` | "where does our budget go?" (+ how it trended) | `GetCostAndUsage` | `unique = [range, granularity, groupBy, filter, metric]`, no `primary` |
| 2 | `DeclaredAwsCostReportSpendForecast` | where spend is EXPECTED to go | `GetCostForecast` | `unique = [range, granularity, metric, filter]`, no `primary` |
| 3 | `DeclaredAwsCostReportRecommendationsToRightsize` | "where can we save money?" (idle/oversized EC2) | `GetRightsizingRecommendation` | `unique = [service, filter, recommendationTarget, benefitsConsidered]`, no `primary` |
| 4 | `DeclaredAwsCostReportRecommendationsToPurchasePlan` | "where can we save money?" (commitment purchase) | `GetSavingsPlansPurchaseRecommendation` | `unique = [savingsPlansType, termInYears, paymentOption, lookbackDays, accountScope, filter]`, no `primary` |

> **the two prefix-pairs.** `Spend*` = *factual* reads (what spend happened / will happen);
> `RecommendationsTo*` = *advisory* reads (AWS's own recommendation nouns — what action to
> take to save). the verb-suffix names the action each recommends: `Rightsize` (resize/kill
> idle EC2) vs `PurchasePlan` (buy a Savings Plan to cut on-demand rates). a name built from
> the AWS "Recommendation" noun + the concrete action is more honest than a vague "Savings" —
> it tells the reader exactly what the report advises. (name detail: `PurchasePlan` = "purchase
> a Savings Plan"; a clearer `PurchaseSavingsPlan` is a possible [wisher] tweak.)

> **observed vs forecast — the `Spend*` pair.** `SpendObserved` reads *what already
> happened* (`GetCostAndUsage` — settled + estimated buckets over a PAST range);
> `SpendForecast` reads *what is projected* (`GetCostForecast` — mean + confidence band
> over a FUTURE range). same `Spend` noun, opposite time-direction.

> **why RecommendationsTo\* are their own resources, not a slice of `SpendObserved`.** the
> wisher named two question-families: "where does our budget go?" (composition/trend →
> `Spend*`) and "where can we save money?" (optimization → `RecommendationsTo*`). these are
> *categorically* different reads: `SpendObserved` AGGREGATES what you already spent;
> `RecommendationsTo*` RECOMMEND what you could stop spending. AWS models them as different
> APIs (`GetCostAndUsage` vs the `Get*Recommendation` family) with different shapes —
> `SpendObserved` returns buckets+groups+amounts; the recommendation reports return per-item
> findings each with an `EstimatedMonthlySavings`, plus a summary
> `EstimatedTotalMonthlySavingsAmount` + `SavingsPercentage`. so #3/#4 are genuine new
> composites, not narrowed #1s.
>
> **rightsize vs purchase-plan — and how the cache makes #4 viable.**
> `RecommendationsToRightsize` (`GetRightsizingRecommendation`) is the clean single-call,
> member-safe read (no prior generation step). `RecommendationsToPurchasePlan`
> (`GetSavingsPlansPurchaseRecommendation`) is more awkward: it needs a two-step
> `StartSavingsPlansPurchaseRecommendationGeneration`-then-`Get` async flow (AWS generates the
> rec set, then you fetch it) and leans PAYER scope. **the cache resolves this** — the getOne
> composite for #4 wraps its read in `withSimpleCache` backed by `simpleOnDiskCache` (local
> dir or an s3 target, with a TTL). on a cache miss it runs the generate-then-get dance once
> and stores the result; on a hit (the common case — recs move slowly) it serves the cached
> rec set with no async round-trip and no re-billed request. so #4 is IN v1, its async
> two-step shape hidden behind a cached read (see the cache note below).

> **overlap disclosure — SpendForecast vs the already-shipped budget forecast.** feat-budget
> (v1.8.0) already ships `DeclaredAwsBudgetNotification` with `basis: 'FORECASTED'`,
> which makes AWS Budgets project spend against a *declared cap* and email when the
> projection breaches a threshold. so a *form* of "expected to go" is already live. the
> new `DeclaredAwsCostReportSpendForecast` is NOT a duplicate — it differs on three axes:
> (a) it needs **no cap** (a budget forecast only exists relative to a budget limit);
> (b) it returns the **projected number + a confidence band** (mean/lower/upper), not a
> boolean breach; (c) it is **sliceable** by service/filter. rule of thumb: budget
> FORECASTED = "will I breach *this* ceiling?"; Forecast report = "what's the *number*,
> and how confident, sliced how I like". the two compose (alert on breach, report the
> figure) rather than compete — but the wisher should know the budget already answers
> the capped-breach case, so this report is the *cap-independent, quantified* half.

all four:
- use the extant `getAwsCostExplorerClient` (pinned to `us-east-1`)
- are **read-only** — `getOne` composite only; `set.findsert` throws `readonly`,
  `set.upsert`/`set.delete` are `null` (mirrors the log-group report DAOs)
- earn their `DeclaredAws*` as **decode-friction composites** (paged SDK call + cast
  into a coherent report AWS has no single resource for) — per
  `rule.forbid.dao-for-narrow-usecase-resource`, allowed
- ship domain-object + DAO + provider registration + sdk export + acceptance, per
  `rule.require.dao-and-acceptance-per-declared-resource`

### the cache — how we make slow/paid reads cheap (`withSimpleCache` + `simpleOnDiskCache`)

the Cost Explorer reads have two costs: **money** ($0.01 per paged request, re-billed on
every `plan`) and **latency** (the recommendation APIs are slow, and
`RecommendationsToPurchasePlan` needs a two-step async generate-then-get). a cache layer
absorbs both:

- each getOne composite wraps its CE read in **`withSimpleCache`** (the ehmpathy HOF —
  `with-simple-cache`), keyed on the report's `@unique` identity (the query shape).
- the cache is backed by **`simpleOnDiskCache`** (`simple-on-disk-cache` — supports a **local
  dir** OR a **remote s3 target**, with **time-based expiration**). the TTL matches the
  data's real freshness: CE cost data settles over ~24–48h and recommendations move slowly,
  so a multi-hour TTL is honest — a cached read is not stale within the window AWS itself has
  not refreshed.
- on a cache **hit**, the read returns the stored domain object with **no CE request** (no
  $0.01, no latency, no async round-trip). on a **miss**, it runs the real read (the two-step
  generate-then-get for #4), stores it under the TTL, and returns it.

why this matters per resource:
- **`SpendObserved` / `SpendForecast` / `RecommendationsToRightsize`** — the cache turns a
  plan-in-a-loop from N billed requests into one, and makes re-`plan` instant.
- **`RecommendationsToPurchasePlan`** — the cache is what makes it *viable in v1*: the ugly
  async `Start*`-then-`Get` generation runs once per TTL window on a miss, and every other
  read is a cheap cached hit. the awkward two-step shape never surfaces to the caller.

this also **softens the acceptance-cost tension** (below): with an s3-backed cache shared
across CI runs, the acceptance suite can read a cached rec set instead of a fresh billed
request every run — though a cold cache still bills once, and the cache-vs-fresh choice for
acceptance is itself a [wisher] call (a cached acceptance proves the cast, not a live read).

> **`options` vs `context`** (per `rule.require.input-options-pattern`): the cache target
> (local vs s3) + TTL are **configuration**, so they belong in an `options`/`context` arg on
> the getOne composite, not in the report's `@unique` identity — the cache is an execution
> concern, not part of what the report IS.

### contract inputs & outputs — testdrive

these mirror the extant report shape (`DeclaredAwsCloudwatchLogGroupReport*` +
`UniDateTimeRange`) and the `.as()` / `getResources()` convention.

**(1) where the money goes — `DeclaredAwsCostReportSpendObserved`** (`GetCostAndUsage`)

```ts
import { asIsoTimeStamp } from 'iso-time';

const spendByService = DeclaredAwsCostReportSpendObserved.as({
  // --- @unique: the query that defines this report's identity ---
  range: {                                   // TimePeriod (start inclusive, end exclusive)
    since: asIsoTimeStamp('2026-07-01T00:00:00Z'), // runtime-validated cast, not a bare string
    until: asIsoTimeStamp('2026-08-01T00:00:00Z'),
  },
  granularity: 'MONTHLY',                    // DAILY | MONTHLY  (the trend buckets)
  groupBy: { dimension: 'SERVICE' },         // { dimension: 'SERVICE' | 'LINKED_ACCOUNT' | ... } | { tag: 'env' }
  filter: null,                              // null = whole account; or { dimension: 'SERVICE', values: ['Amazon Elastic Compute Cloud - Compute'] }
  metric: 'UNBLENDED_COST',                  // which cost metric to sum

  // --- @readonly: derived from AWS on read ---
  // buckets: [                              // one per time granule → the TREND
  //   {
  //     range: { since, until },
  //     total: { amount: '16.42', unit: 'USD' },
  //     groups: [                           // the COMPOSITION within the bucket
  //       { keys: ['Amazon Elastic Compute Cloud - Compute'], amount: '12.10', unit: 'USD' },
  //       { keys: ['Amazon Simple Storage Service'],          amount: '4.32',  unit: 'USD' },
  //     ],
  //   },
  // ],
  // total: { amount: '16.42', unit: 'USD' }, // sum across all buckets
});
```

**(4) where we can SAVE money (commitment) — `DeclaredAwsCostReportRecommendationsToPurchasePlan`** (`GetSavingsPlansPurchaseRecommendation`)

```ts
const savingsPlan = DeclaredAwsCostReportRecommendationsToPurchasePlan.as({
  // --- @unique: the recommendation query that defines this report's identity ---
  savingsPlansType: 'COMPUTE_SP',              // COMPUTE_SP | EC2_INSTANCE_SP | SAGEMAKER_SP | DATABASE_SP
  termInYears: 'ONE_YEAR',                     // ONE_YEAR | THREE_YEARS
  paymentOption: 'NO_UPFRONT',                 // NO_UPFRONT | PARTIAL_UPFRONT | ALL_UPFRONT

  filter: null,                                // or { dimension: 'LINKED_ACCOUNT', values: [...] }
  // note: NO `range` — like Rightsize, AWS derives the window from `lookbackDays`

  // --- @readonly: derived from AWS on read (behind the cache) ---
  // summary: {                                // SavingsPlansPurchaseRecommendationSummary
  //   estimatedMonthlySavings: { amount: '4.20', unit: 'USD' },  // EstimatedMonthlySavingsAmount
  //   estimatedSavingsPercentage: '12',
  //   estimatedRoi: '0.30',
  //   hourlyCommitmentToPurchase: '0.021',
  //   currentOnDemandSpend: { amount: '35.00', unit: 'USD' },
  // },
  // recommendations: [ /* per-plan detail: commitment, upfront, est savings */ ],
});

// the getOne composites wrap their reads in withSimpleCache(simpleOnDiskCache(...)) —
// #4's two-step async generate runs once per TTL on a miss; see the cache note above
export const getResources = async () => [
  spendByService, forecastMonthEnd, savingsEc2, savingsPlan,
];
```

> **⚠️ both `RecommendationsTo*` reports have NO `range` — they sidestep the
> identity-vs-recency tension.** unlike the `Spend*` pair, neither
> `GetRightsizingRecommendation` nor `GetSavingsPlansPurchaseRecommendation` takes a
> `TimePeriod`; AWS picks the lookback window itself (`Rightsize` reports it as
> `LookbackPeriodInDays`; `PurchasePlan` takes it as the enum `lookbackDays`). so each
> recommendation report's identity is its *query shape* — naturally stable — and its numbers
> refresh as `@readonly`. this is a point IN FAVOR of the `RecommendationsTo*` pair: they are
> the reports where "declare once, re-plan → KEEP, read anytime" holds unconditionally, with
> no relative-range churn.

outputs: `plan` shows each report as a change whose read derives the `@readonly`
fields (buckets/points/totals). because they are read-only, a re-`plan` is a `KEEP`
(the identity — the query — is unchanged; only the derived numbers move, and those
are `@readonly` so they don't drive drift). reads return `HasReadonly<...>` domain
objects.

> **⚠️ the identity-vs-recency contradiction (core tension — must be settled).** the
> `range` is part of the report's `@unique` identity, and the testdrive hardcodes
> *absolute* dates (`2026-07-01` → `2026-08-01`). that is what makes the KEEP claim above
> true — a *fixed* range is a *stable* identity. BUT it means the report is **frozen to
> July 2026**: to hold "this month's spend" current, a human must hand-edit the dates
> every month, and a stale file silently reports last month forever. the natural desire
> — "always show me THIS month / the last 30 days" — wants a *relative* range, but a
> relative range is NOT a stable unique key: expanded to absolute dates it churns to a
> **new identity every day** (so every plan spawns a new report row and KEEP never
> holds), and left symbolic it can't be a deterministic key at all. so the vision's
> headline promise — "declare once, re-plan → KEEP, read anytime" — holds ONLY for
> frozen absolute ranges, and those are exactly the ranges a user does NOT want for a
> report that must stay current. this is the deepest design tension in the feature and a
> **[wisher]** decision: (a) absolute-range-only (honest KEEP, but manually stale),
> (b) support a relative range as a first-class field that the read expands to absolute
> *at read time* while it holds the SYMBOLIC form (e.g. `last: 'P30D'`) as the stable
> identity — so `last: P30D` is one durable row whose numbers refresh (this is the
> ergonomic users actually want, but it means the unique key is the symbolic range, not
> the expanded dates), or (c) accept daily identity churn for relative ranges. option (b)
> is likely right but changes the identity model — it must be chosen deliberately, not
> defaulted.

### timeline the user goes through

1. write `reports.ts` for the account (copy the shape above, tweak range/groupBy)
2. `plan` → each report reads against live bill data
3. read the derived domain object (buckets = trend, groups = composition, points =
   forecast) — pipe into a dashboard, an alert, or just eyeball it
4. next day, re-`plan` → *for an absolute range*, same identity + refreshed numbers =
   `KEEP`; *for a relative range* (if adopted per the identity-vs-recency note above),
   KEEP holds only if the symbolic form is the identity — see that ⚠️ callout
5. widen the range or change the groupBy → a *new* identity → a new report row

### how a user describes it to a friend

> "it's a saved Cost Explorer query, as code. i write down the date range and how i

## evaluation

### how well it solves the goals

- **"where does our budget go?"** → `DeclaredAwsCostReportSpendObserved` groups → ✅
- **how it trended over time** → same report's DAILY/MONTHLY buckets → ✅
- **where money is expected to go** → `DeclaredAwsCostReportSpendForecast` → ✅
- **"where can we save money?"** → `DeclaredAwsCostReportRecommendationsToRightsize` (idle/oversized
  EC2) + `DeclaredAwsCostReportRecommendationsToPurchasePlan` (commitment purchase) → ✅ (both
  in v1; the cache absorbs #4's async two-step + per-request cost)
- **reproducible across accounts** → per-account `reports.ts` → ✅

### pros

- **pure reuse of a proven pattern** — read-only report composite already exists twice
  (CloudWatch); we point it at the CE client the anomaly resources already use. no new
  client, no new auth reach, no new mechanism.
- **read-only = trivially idempotent** — no `set`/`del`, no convergence subtlety; a
  re-plan is a `KEEP` because the identity (the query) is stable and the numbers are
  `@readonly`.
- **one report answers two of the wish's phrasings at once** — Spend's buckets ARE the
  trend; its groups ARE the composition. no separate "trend" resource needed.
- **composes with feat-budget** — same account, same CE client, same `Cost*` family;
  reports sit beside budgets/guards as the "observability" half of cost control.
- **the cache turns two big cons into non-issues** — `withSimpleCache` +
  `simpleOnDiskCache` (local or s3, TTL) means repeat reads cost $0 and are instant, and #4's
  ugly async two-step generate runs once per TTL window. the cache is what lets us ship all
  four reports (incl. commitment-purchase) without a plan-loop bill or a slow read.

### cons / tradeoffs

- **Cost Explorer API costs money (mitigated by the cache)**: each paged CE request is
  **$0.01**, re-billed on every un-cached `plan`. the `withSimpleCache` layer collapses a
  plan-loop to one billed request per TTL window — but a COLD cache still bills, and a
  mis-set TTL could serve stale numbers, so the charge is softened, not erased. still flag it
  (`rule.forbid.surprises`).
- **region-pinned to `us-east-1`** — same deviation the anomaly resources already carry
  (the CE client hardcodes it); consistent, but still a footgun for a reader who assumes
  region flows from context.
- **forecast needs history** — `GetCostForecast` fails with `DataUnavailableException`
  on a young account (needs enough past cost to project). the demo account may be too
  new for a meaningful forecast — acceptance can prove the resource *reads* against a
  seeded account but may hit `DataUnavailable` on a cold one.
- **recommendations need usage** — the rightsize API only surfaces recs for EC2 that has run
  long enough to analyze; a low-spend demo account may return an EMPTY rec set (a valid,
  non-error read, but a thin showcase). also `Service` is EC2-only today — no resize recs for
  RDS/Lambda/etc — so v1 resize coverage is scoped to EC2, disclosed.
- **acceptance for read-only reports is a read, not an apply** — we assert the report
  reads + matches a masked snapshot (numbers masked, structure asserted), not a
  CREATE/KEEP lifecycle. slightly weaker than a mutable resource's plan/apply/KEEP, but
  identical to how the extant log-group reports are covered.
- **management-account gravity (maybe)** — `GetCostAndUsage` on a *member* account
  returns only that account's costs; org-wide roll-ups need payer creds. the demo
  acceptance profile is a member account, so a `LINKED_ACCOUNT` group-by is trivial but
  a full-org report needs the payer — same wall feat-budget's guards hit. the
  `PurchasePlan` report inherits this most sharply: its `accountScope: PAYER` mode rolls up
  the whole org, so a payer-scope read from a member account yields only that member's recs —
  a [wisher] scope call (below).

### edgecases & pit-of-success

| edgecase                               | how the contract keeps users safe                                |
| -------------------------------------- | ---------------------------------------------------------------- |
| `end` before `start`, or `end` == `start` | validate `range` at cast; AWS 400s otherwise — fail loud early |
| forecast `start` in the past           | `GetCostForecast` requires start ≥ today; validate + fail loud   |
| forecast horizon > limits (DAILY >3mo, MONTHLY >18mo) | validate against the documented caps          |
| empty result (no spend in range)       | read to empty `buckets`/`groups`, not an error                   |
| `predictionInterval` outside 51–99     | validate the range; AWS 400s otherwise                           |
| `groupBy` tag key that is absent       | AWS returns empty-string tag values, not an error — surface as-is |
| plan-in-a-loop cost                    | doc the $0.01/request charge; encourage a wider range over frequent replans |
| young account, no forecast history     | catch `DataUnavailableException` → fail loud with a clear "account too new to forecast" message |
| rightsize: no recs (low-usage account) | read to an empty `recommendations` + zero `summary`, not an error — an empty save-list is valid |
| rightsize: non-EC2 service requested   | `service` accepts only `AmazonEC2` today → validate + fail loud (no silent wrong-service read) |
| purchase-plan: generate not yet done   | the getOne composite runs `Start*` then polls `Get*` on a cache miss; a not-ready poll is retried, not surfaced as an error |
| purchase-plan: stale cache vs live recs | TTL bounds staleness; a shorter TTL trades freshness for more billed reads — a documented `options` knob |

## open questions & assumptions

### assumptions made

1. **read-only composites, no mutable state.** reports are `getOne`-only, exactly like
   the CloudWatch log-group reports. **assumed** — a report is a read, never an apply.
2. **four resources — two symmetric pairs.** the `Spend*` pair — `SpendObserved` (where does
   our budget go? + trend) and `SpendForecast` (expected spend) — answers "where the money
   goes"; the `RecommendationsTo*` pair — `RecommendationsToRightsize` (resize/kill idle EC2)
   and `RecommendationsToPurchasePlan` (buy a Savings Plan) — answers "where can we save?". i
   did NOT split a separate `DeclaredAwsCostReportTrend` because `SpendObserved`'s buckets
   already ARE the trend. **wisher-confirmed** — the wisher named both question-families and
   both save-report members; the residual confirmation is scope depth per resource (see
   [wisher] below), not the count.
3. **default metric = `UNBLENDED_COST`.** the most common "what did it cost me" number —
   BUT it is GROSS (pre-credit) and the wish literally says "where our money goes" (net
   cash-out). this default now carries a [wisher] fork (gross vs `NetUnblendedCost`); see
   the metric-semantics question below. flagged, not silently assumed.
4. **`us-east-1` pin, member-account scope** for the demo/acceptance path (consistent
   with the anomaly resources).
5. **the wish's "budget & cost reports" both mean Cost Explorer.** the wish literally says
   "**budget** & cost reports". i assumed BOTH halves = Cost Explorer read APIs. but
   "budget reports" could instead mean **budget-vs-actual figures on the
   `DeclaredAwsBudget` resources we already shipped** — AWS exposes
   `DescribeBudgetPerformanceHistory` (planned vs actual per budget period), which is a
   *different service* (Budgets, not Cost Explorer) and a *different resource*. so there
   is a real fork in what the wisher even asked for: (a) two Cost-Explorer reports (my
   read), or (b) one Cost-Explorer spend/forecast report + one **budget-performance**
   report keyed to a `DeclaredAwsBudget` ref. **needs the wisher** — this changes which
   AWS API the "budget" half hits. i lean (a) but flag (b) as a plausible literal read of
   the word "budget".
6. **AWS cost `Amount` is kept as a decimal STRING, not parsed to `number`.** AWS returns
   amounts like `"39.1603300457"` — high-precision decimal strings. i assumed we preserve
   them as strings in the domain object (like the `{ amount, unit }` shape the budget
   resources use), NOT coerce to JS `number` (which is float64 and would lose precision on
   long decimals / large sums). the extant log-group report uses `number` for *bytes*
   (integers, safe), but cost amounts are NOT integers — a naive mirror that reused
   `number` would silently corrupt cents. this assumption must hold at execution: amounts
   stay strings (or an `iso-price`-style money type), never bare `number`.
7. **dogfood on the DEMO account, like feat-budget did.** i inherited feat-budget's
   dogfood target (the demo account) without question. but that inheritance is a category
   error for *reports*: feat-budget dogfooded on demo because a *guard* needs a cheap,
   low-stakes account to protect — the cheaper the better. a *report*, by contrast, draws
   its entire value from spend VOLUME to slice and trend; the demo account is deliberately
   capped at **$21/mo**, so its cost report is near-empty and its forecast is
   `DataUnavailable`-prone. so the very account that made feat-budget's dogfood sensible
   makes reports' dogfood *vacuous* — you cannot showcase "see where the money goes" on an
   account engineered to spend almost zero. **needs the wisher**: dogfood/acceptance
   should probably target a HIGHER-spend account (or the payer account with a
   `LINKED_ACCOUNT` filter), NOT the demo account by reflex. this also compounds the
   acceptance-cost tension (peer review #2): a near-empty demo report makes the masked
   snapshot assert almost zero.

  family already shipped as `DeclaredAwsCostAnomaly*` in v1.8.0 — a retrofit to
  `CostExplorer*` is a fresh back-compat break on a just-released public SDK for zero

  (`rule.require.symmetry-with-peer-resources` is satisfied — the family is NOT split);
  (3) `Cost` is the reader-friendly feature word we deliberately chose over the terse `Ce`
  service acronym in feat-budget, and consistency with that prior decision beats a churn.
  the wisher CAN override, but logic gives a clear default — this is a decided default, not
  an open wisher fork.
- **how fresh is Cost Explorer data?** a question the vision failed to surface at all until
  this triage — and a load-bearing one for a "see where the money goes RIGHT NOW" feature.
  answerable NOW from known AWS behavior: CE cost data is NOT real-time; it refreshes at
  most a few times per day and commonly lags actual usage by up to ~24h (some records
  settle over 24–48h). so a report whose range includes "today" reads a PARTIAL/stale
  today, and `ResultsByTime` marks such buckets `Estimated: true`. implication for the
  contract: the domain object should expose that `estimated` flag per bucket so a reader is
  not misled that a fresh number is final. this is [answered] (the latency is a known fact)
  but it MUST shape the contract — surfaced here so execution carries the `estimated` field
  rather than a silent presentation of stale data as settled.
- **can a MEMBER account report on other accounts' spend?** answerable NOW from the AWS
  doc, so split out of the [wisher] "scope" item: NO — `GetCostAndUsage` on a member
  account returns only THAT account's costs; a `LINKED_ACCOUNT` group-by or any org-wide
  roll-up requires PAYER (management) creds. so the technical constraint is a fact, not a
  wisher call. what REMAINS [wisher] is only the intent question "do we WANT org-wide
  reports (→ run from payer) or per-account (→ member is fine)?" — the capability boundary
  is decided here.
- **does the default `metric` match the wish's literal words?** the wish asks to see where
  our money **goes** — which literally means the *net cash that leaves the account*. but
  the vision defaults `metric` to `UNBLENDED_COST`, and that is the GROSS list-price cost
  BEFORE credits, refunds, and RI/SP discounts are netted out. the FACTUAL half is answerable
  NOW from the CE metric enum (no wisher needed): `UnblendedCost` = gross; `NetUnblendedCost`
  and `NetAmortizedCost` = net-of-credits (what actually hits the bill). so a report that says
  "$16.42 went to EC2" on `UnblendedCost` can OVERSTATE the money that truly left the bank if
  the account carries credits — a silent mismatch between the number and the wish's own word
  "goes". this is a compound question (per the r4 decompose discipline): the metric-semantics
  FACT is [answered] here (gross vs net enum), and only the INTENT — "does the wisher mean
  gross list-price or net-of-credits cash-out?" — is a genuine [wisher] fork (below). i lean
  net for the literal wish, but the vision's stated default is gross, so this contradiction
  must be settled deliberately, not defaulted silently.

- **pagination loop shape?** answerable NOW, so moved out of [research]: `GetCostAndUsage`
  returns a `NextPageToken` and accepts it back as `NextPageToken` — a token-cursor loop,
  the SAME shape the extant anomaly ops already use for `GetAnomalyMonitors`
  (`NextPageToken` in/out). so the pattern to mirror is decided (token-cursor while-loop);
  only the exact field names get confirmed when the SDK types are open — which is the
  separate [research] item below, not this one. `GetCostForecast` is NOT paginated (single
  response), so no loop there.

### [wisher] — only the wisher can decide

- **the wish-interpretation fork (highest leverage)**: does "budget reports" mean a Cost
  Explorer report, or a **budget-vs-actual** report on the shipped `DeclaredAwsBudget`
  resources (`DescribeBudgetPerformanceHistory`)? this changes which AWS service the
  "budget" half calls (see assumption #5). only the wisher knows the intent.
- **scope intent**: do we WANT org-wide roll-ups (must run from the payer account) or
  per-account reports (member creds suffice)? the capability boundary is already decided
  in [answered] (member = self-only, org = payer); only the intent remains for the wisher.
- **metric semantics — gross or net-of-credits?**: the wish's word "goes" literally means
  cash-out (net), but the vision defaults to gross `UnblendedCost`. the metric enum is
  already decided in [answered]; only the INTENT remains — does the wisher want the gross
  list-price number (`UnblendedCost`) or the net-of-credits number (`NetUnblendedCost` /
  `NetAmortizedCost`) that matches "money that actually left"? i lean net, but the wrong
  default silently overstates spend on a credited account, so the wisher must pick.
- **which reports ship first**: all four, or the `Spend*` pair before the `RecommendationsTo*`
  pair? (`SpendForecast` has the young-account `DataUnavailable` wrinkle; `RecommendationsTo*`
  need enough EC2 usage to produce recs, so on a low-spend demo they may return an empty rec
  set.) the wisher pulled `PurchasePlan` INTO scope, so the default is all four; sequence is
  the residual call.
- **`PurchasePlan` account scope — `PAYER` or `LINKED`?** `GetSavingsPlansPurchaseRecommendation`
  takes `AccountScope: PAYER` (roll up mgmt + members) vs `LINKED` (this member only). from a
  member account, `PAYER` yields only that member's recs — so the useful org-wide commitment
  view needs payer creds. which scope (and thus which account runs it) is a wisher intent call.
- **cache config — local dir or s3 target, and what TTL?** the reports wrap reads in
  `withSimpleCache` + `simpleOnDiskCache`. a LOCAL dir is simplest for a single dev; an S3
  target lets CI + teammates share a warm cache (and softens the acceptance-cost tension). the
  TTL trades freshness for fewer billed reads — a few hours matches CE's own ~24–48h settle.
  target + TTL are `options` (config), so wisher-tunable without touching identity.
- **name tweak — `PurchasePlan` or `PurchaseSavingsPlan`?** the AWS product is "Savings
  Plans"; `PurchasePlan` is terse, `PurchaseSavingsPlan` is unambiguous. a small wisher call.
- **dogfood**: declare a demo `reports.ts` and wire it into `provision/`, like
  feat-budget's `resources.budget.ts`? (proves the read path end-to-end.)
- **the acceptance-requirement tension (must be settled, not glossed).**
  `rule.require.dao-and-acceptance-per-declared-resource` makes acceptance *mandatory*
  for all three reports — but these particular resources fight acceptance three ways at once,
  and that collision is a wisher-level call, not a detail:
  (a) **every CI run bills money** — each acceptance plan issues a real
  `GetCostAndUsage`/`GetCostForecast` at $0.01/request; unlike the CloudWatch log-group
  reports (whose metric reads are effectively free), this puts a per-run dollar cost on
  the test suite itself.
  (b) **the numbers are non-deterministic and near-zero on the demo account** — a masked
  snapshot can assert *structure*, but the demo account's actual spend is tiny + it
  drifts, so the report may read as empty or jitter run-to-run; the assertion degrades to
  "did it read without error", which is thin.
  (c) **Forecast may `DataUnavailableException` in CI** — a young/low-spend account can
  make the forecast read *fail*, so its acceptance cannot even prove a clean read.
  the rule's own escape-hatch says "fix the fixture, don't skip". options for the wisher:
  seed/point acceptance at an account with real history, assert read-without-error +
  structure only (accept the thin coverage + the per-run $), or split Forecast to a
  hand-run acceptance. this is the single biggest execution risk and needs a wisher
  decision before build.
- **the FUNDAMENTAL shape question: declared resources, or just SDK read-operations?**
  the entire vision assumes reports must be `DeclaredAws*` resources (with DAOs, provider
  registration, and resource-acceptance). but a cost report is a pure *read* — it has no
  desired state to converge, no lifecycle, no drift. an honestly simpler alternative:
  ship them as plain exported read-operations — `getCostReportSpend(input, context)` /
  `getCostReportForecast(...)` — returning typed domain objects, WITHOUT the
  `DeclaredAws*` + DAO + provider + resource-acceptance machinery. this would:
  (a) **dissolve the acceptance-cost tension entirely** (issue found in self-review) —
  operations get normal integration tests the author runs deliberately, not a resource
  that bills $0.01 on every CI plan; (b) drop ~5 files of DAO/provider boilerplate per
  report; (c) still be fully declarative-adjacent (same client, same casts, same domain
  objects) — just *invoked* rather than *declared*. the COUNTER-argument (why the vision
  still leans declared-resource): the extant CloudWatch log-group reports set the
  precedent as `DeclaredAws*` composites, and `plan`-time visibility (seeing the report
  in a wish alongside budgets) is the ergonomic the wish implies. so this is a genuine
  fork the wisher must pick: **declared-resource** (consistent with the log-group
  precedent, plan-visible, but carries the acceptance-cost tension) vs **read-operation**
  (simpler, cheaper to test, but breaks symmetry with the extant reports and isn't
  plan-visible). i lean declared-resource for symmetry, but the simpler path is real and
  directly retires the biggest execution risk above — the wisher should weigh it.

- none beyond the above — both CE report APIs are verified against AWS docs (see

## what is awkward?

1. **read-only "resources" in a declarative provisioner.** a report you `plan` but never
   `apply` bends the mental model of "declare desired state." it's the same shape the
   log-group reports already set, so the precedent exists — but a newcomer may expect a
   report to be a query you *run*, not a resource you *declare*. the read-only DAO
   (findsert-throws, upsert/delete-null) makes it safe, but the ergonomics are a little
   odd.
2. **per-plan dollar cost.** every `plan` that includes a report bills $0.01/request.
   trivial in absolute terms, but a *plan* silently costing money is a surprise the
   feature must call out loudly — no other resource in the repo charges you to plan.
3. **forecast fragility on young accounts.** the demo account is exactly the kind of
   young account `GetCostForecast` refuses to project. the most demo-relevant account is
   the one where the forecast half is hardest to prove in acceptance.
4. **name-symmetry pressure, again.** feat-budget just shipped `DeclaredAwsCostAnomaly*`
   minutes ago; these reports must join the *same* CE family symmetrically. keeping
   `Cost*` avoids a retrofit but leaves the family named after the reader-friendly
   feature word rather than the service (`CostExplorer`) — a papered-over inconsistency
   the wisher may want to settle one way or the other.
5. **"report" as a noun now spans four very different SDK calls — but two clean pairs.**
   `SpendObserved` (historical aggregation), `SpendForecast` (ML projection with a confidence
   band), `RecommendationsToRightsize` (per-resource resize/kill actions), and
   `RecommendationsToPurchasePlan` (commitment-purchase advice) all share the word "report" and
   the read-only shape, yet their `@readonly` shapes diverge sharply (groups/buckets vs
   mean/lower/upper vs per-box recs vs per-plan recs). the two prefix-pairs make this honest:
   `Spend*` clusters the factual reads, `RecommendationsTo*` clusters the advisory reads — a
   reader sees the split in the name and does not assume the four are interchangeable.
6. **deliberate narrows of a plural API (must be disclosed, not silent).** the
   testdrive models `groupBy` as a *single* dimension-or-tag and `metric` as a *single*
   value, but `GetCostAndUsage` actually accepts **up to two** `GroupBy` entries (AWS's
   own sample groups `SERVICE` x an `Environment` tag at once) and a **plural**
   `Metrics[]`. i narrowed both to keep the report's identity clean and the "one number,
   one slice" mental model simple — a report that answers "spend by service" is easier
   to reason about than one that answers "spend by service AND tag, across three
   metrics". this is a defensible v1 simplification, BUT it under-models the API, so it
   is a **[wisher] decision**, not a silent default: ship single-group/single-metric v1
   (my lean), or model the two-level groupBy + metric list from the start? either way the
   contract must not *pretend* the API is singular — the narrow is a choice, disclosed
   here. **the same narrow also applies to `filter`**: the testdrive models it flat
   (`{ dimension, values }`), but AWS's `Filter` is a recursively nestable `Expression`
   (`And`/`Or`/`Not` of dimensions/tags/cost-categories). so the v1 narrow-scope decision
   spans FOUR fields — `groupBy` (1 vs 2), `metric` (single vs `Metrics[]`), `filter`
   (flat predicate vs full `Expression` tree), and `granularity` (the testdrive drops the
   API's `HOURLY` option, exposing only DAILY/MONTHLY) — and the wisher's "simple v1" vs
   "full-fidelity" call should cover all four together.

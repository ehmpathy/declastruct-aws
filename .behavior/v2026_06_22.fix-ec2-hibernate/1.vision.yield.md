# vision: verify ssh connection & hibernate capacity

## the outcome world

### day-in-the-life with this in place

an engineer declares an EC2 instance with hibernation enabled and deploys it via declastruct. they know it works because:

1. **acceptance tests prove the contract** — the declastruct CLI workflow (plan → apply → idempotency) is verified for EC2 resources
2. **integration tests prove the operations** — individual get/set operations are verified for hibernation state transitions
3. **dogfood proves real-world usage** — `provision/aws.infra` demonstrates the pattern for demo account infrastructure

the engineer can confidently:
- declare `status: 'hibernated'` and know the instance will hibernate
- declare `status: 'active'` and know it will resume
- establish SSM sessions to active instances for debug

### before/after contrast

| before | after |
|--------|-------|
| EC2 resources in acceptance tests but no assertions | acceptance tests verify EC2 plan/apply/idempotency |
| hibernation tested in integration but not acceptance | hibernation verified end-to-end via declastruct |
| no proof we can connect to instances | SSM connectivity verified via integration test |
| dogfood exists but not exercised regularly | dogfood runs as part of acceptance suite |

### "aha" moment

> "wait — i can declare `status: 'hibernated'` and declastruct handles the whole state machine?"

the value clicks when users realize:
- hibernation is a declarative intent, not a procedural command
- declastruct handles the AWS waiter complexity
- state transitions are idempotent — re-apply of same status is a no-op

### contract inputs & outputs

```typescript
// declare launch template with hibernation
const template = DeclaredAwsEc2LaunchTemplate.as({
  exid: 'my-template',
  instanceType: 't3.micro',
  imageId: 'ami-xxx',
  hibernation: true,            // <-- enable hibernation
  rootVolumeEncrypted: true,    // <-- required for hibernation
  rootVolumeSize: 16,           // <-- must hold RAM
  // ...
});

// declare instance
const instance = DeclaredAwsEc2Instance.as({
  exid: 'my-instance',
  template: { exid: template.exid },
  subnet: { exid: 'my-subnet' },
  securityGroups: [{ exid: 'my-sg' }],
  tags: { purpose: 'dev' },
});

// control lifecycle via session
const session = DeclaredAwsEc2InstanceSession.as({
  instance: { exid: instance.exid },
  status: 'active',    // or 'stopped', 'hibernated'
});

// authorize SSH key on instance (via Instance Connect bootstrap)
const sshKey = DeclaredAwsEc2SshKeyAuthorized.as({
  instance: { exid: instance.exid },
  publicKey: fs.readFileSync('~/.ssh/id_ed25519.pub', 'utf8'),
  comment: 'vlad-laptop',
});

// open SSH tunnel via SSM
const sshTunnel = DeclaredAwsSsmSshTunnel.as({
  instance: { exid: instance.exid },
  localPort: 2222,
  status: 'OPEN',    // or 'CLOSED'
});
```

after `npx declastruct apply`, SSH in:
```bash
ssh -i ~/.ssh/id_ed25519 -p 2222 ec2-user@localhost
```

### what it looks like to leverage

```bash
# plan deployment
npx declastruct plan --wish resources.ts --into plan.json

# apply — instance starts in 'active' state
npx declastruct apply --plan plan.json

# later: change status to 'hibernated' in resources.ts
# re-plan and apply — instance hibernates
npx declastruct plan --wish resources.ts --into plan.json
npx declastruct apply --plan plan.json

# verify via AWS CLI or SSM
aws ec2 describe-instances --instance-ids i-xxx
```

### timelines

- **plan**: <5s — no AWS mutations, just diff calculation
- **apply (create instance)**: ~60s — EC2 launch + waiter
- **apply (hibernate)**: ~120s — stop with hibernate flag + waiter
- **apply (resume)**: ~60s — start + waiter
- **apply (idempotent)**: <5s — no-op when already in desired state

## mental model

### how users describe this to a friend

> "it's like terraform for EC2 but with first-class hibernation support. you declare 'i want this instance hibernated' and it figures out the rest."

### analogies

- **hibernation = laptop sleep** — state preserved, quick resume
- **session = power button** — controls active/stopped/hibernated
- **declastruct = desired state reconciler** — converges reality to declaration

### terms

| user term | our term | AWS term |
|-----------|----------|----------|
| sleep / hibernate | `status: 'hibernated'` | StopInstances with Hibernate=true |
| wake up / resume | `status: 'active'` | StartInstances |
| shutdown | `status: 'stopped'` | StopInstances with Hibernate=false |
| SSH | SSM session | StartSession via aws.ssm |

## evaluation

### how well does it solve the goals?

| goal | coverage |
|------|----------|
| verify hibernation works | integration tests cover state transitions |
| verify SSH over SSM works | needs fck-nat + IAM role + integration test |
| acceptance test coverage | needs assertions in declastruct.acceptance.test.ts |
| dogfood in provision/aws.infra | exists but needs verification |

### pros

- **declarative** — state intent, not procedure
- **idempotent** — safe to re-apply
- **typed** — domain objects catch config errors at compile time
- **testable** — integration tests verify real AWS behavior

### cons

- **hibernation constraints** — requires encrypted EBS, sufficient volume size, specific instance types
- **cold start latency** — hibernate/resume takes ~2 minutes
- **cost while hibernated** — EBS storage still incurs charges

### edgecases & pit-of-success

| edgecase | how we handle |
|----------|---------------|
| hibernation on non-hibernation-capable instance | AWS API error surfaced immediately |
| unencrypted root volume | launch template requires `rootVolumeEncrypted: true` |
| volume too small for RAM | user must size appropriately — we document in types |
| transition from 'hibernated' to 'stopped' | AWS handles — we just call StopInstances (no hibernate flag) |

### infrastructure decision

**NAT approach:** ephemeral fck-nat on t4g.nano

how it works:
1. CI findserts NAT instance before tests (find extant by tag, or create)
2. if extant and active → reuse (no new instance)
3. tests run (SSM connectivity via NAT)
4. instance self-terminates after 90 min (hard timeout via user data)
5. safety net: if CI fails to clean up, instance dies anyway

idempotency:
- tag: `purpose: declastruct-acceptance-nat`
- findsert: lookup by tag, create only if not found
- prevents duplicate instances from parallel/overlapping test runs

age check:
- if extant instance age < 87 min → reuse
- if extant instance age >= 87 min → terminate and create new
- ensures tests have at least 3 min safe window before self-destruct

cost:
- ~$0.10/mo (only runs during test execution)
- vs $32/mo for NAT Gateway
- vs $6.72/mo for always-on NAT

user data (self-destruct command):
```bash
#!/bin/bash
# terminate after 90 minutes
sleep 5400 && \
  INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id) && \
  aws ec2 terminate-instances --instance-ids $INSTANCE_ID &
```

trade-offs accepted:
- ~60s setup time per test run (NAT launch)
- single AZ (acceptable for acceptance env)
- 90 min hard limit on test duration

## groundwork

### external research

**AWS SSM Session Manager**
- SSM sessions work without public IP if:
  - instance has SSM agent (AL2023 includes it)
  - instance has IAM role with SSM permissions
  - VPC has SSM endpoints OR instance has internet access
- current acceptance VPC has internet gateway but no SSM VPC endpoints
- instance needs IAM instance profile with `AmazonSSMManagedInstanceCore` policy

**AWS EC2 Hibernation Requirements** (verified via AWS docs)
- EBS-backed instance with encrypted root volume
- instance type supports hibernation (t3.micro: yes)
- root volume >= instance RAM (t3.micro has 1 GiB RAM, 16 GiB volume is plenty)
- hibernation enabled at launch (cannot enable after)
- Amazon Linux 2, AL2023, Ubuntu, Windows supported

## what is awkward?

### SSM requires IAM instance profile — in scope

the launch template has `iamInstanceProfile: null` — SSM won't work without it. **solution:** add IAM role with `AmazonSSMManagedInstanceCore` policy and attach to launch template.

### acceptance instance is stopped by default

`resources.acceptance.ts:271-274` declares `status: 'stopped'` to avoid charges. to verify connectivity, we'd need to temporarily set `status: 'active'` in test, then stop after.

### no public IP in acceptance VPC — resolved

current subnet doesn't assign public IP. **solution:** ephemeral fck-nat instance (~$0.10/mo) launched only for test runs. this avoids:
- public IP exposure on test instances
- $32/mo NAT Gateway cost
- $6.72/mo always-on NAT cost
- new VPC endpoint domain objects

fck-nat instance launches in public subnet before tests, self-terminates after 90 min.

### session test duration

hibernation tests take ~5 minutes per transition. acceptance tests already take significant time — add of hibernate/resume verification adds more.

**recommendation**: keep hibernation verification in integration tests (where it already exists), add SSM connectivity test to integration tests, add plan/apply/idempotent assertions for EC2 to acceptance tests without exercise of hibernation (too slow).

# vision: EC2 launch templates + hibernate support

## the outcome world

### day-in-the-life with this in place

a developer declares their EC2 resources:

```typescript
// layer 1: what the machine is (AMI, type, hibernate)
const template = DeclaredAwsEc2LaunchTemplate.as({
  exid: 'my-template',
  instanceType: 't3.medium',
  imageId: 'ami-0abcd1234567890ef',
  hibernation: true,
  rootVolumeSize: 30,
  rootVolumeEncrypted: true,
  iamInstanceProfile: 'my-role',
  userData: '#!/bin/bash\necho "ready"',
});

// layer 2: where to place it + identity
const instance = DeclaredAwsEc2Instance.as({
  exid: 'my-instance',
  template: RefByUnique.as(template),
  subnetId: 'subnet-abc123',
  securityGroupIds: ['sg-xyz789'],
});

// layer 3: lifecycle state (safe to change)
const session = DeclaredAwsEc2InstanceSession.as({
  instance: RefByUnique.as(instance),
  status: 'active',
});

// apply all
await declastruct.apply({ resources: [template, instance, session] });
```

at end of work — hibernate by declaration change:
```typescript
const session = DeclaredAwsEc2InstanceSession.as({
  instance: RefByUnique.as(instance),
  status: 'hibernated',
});

await declastruct.apply({ resources: [template, instance, session] });
// RAM preserved to EBS, compute charges stop
```

next day — resume by declaration change:
```typescript
const session = DeclaredAwsEc2InstanceSession.as({
  instance: RefByUnique.as(instance),
  status: 'active',
});

await declastruct.apply({ resources: [template, instance, session] });
// ~30-60s later: RAM restored, processes intact
```

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| cost model | always-on or cold start | hibernate = pay storage only |
| resume time | 2-5 min (fresh boot) | typically 30-60s for small instances (RAM preserved) |
| state continuity | lost on stop | preserved via hibernate |
| config management | inline in instance | 3-layer separation: template → instance → session |
| monthly cost (t3.medium, 2hr/day) | ~$30/mo (always on) | ~$5/mo (hibernate when idle) |
| lifecycle safety | risk of accidental instance mutation | session layer protects instance config |

### "aha" moment

"wait, i can hibernate my dev machine at 5pm, come back at 9am, and my terminal sessions, processes, even my vim buffers are still there? and it only cost me storage overnight? and i can't accidentally break the instance config when i toggle hibernate?"

## user experience

### usecases

1. **bursty workloads**: machines that work 2hr, hibernate, resume 1hr later
2. **dev environments**: personal dev VMs that hibernate overnight
3. **cost optimization**: reduce EC2 spend by 6-7x for intermittent workloads
4. **state preservation**: avoid cold start penalties for stateful applications

### contract inputs & outputs

#### DeclaredAwsEc2LaunchTemplate (new)

```typescript
interface DeclaredAwsEc2LaunchTemplate {
  id?: string;                              // @metadata, assigned by AWS
  exid: string;                             // @unique, declarative reference
  instanceType: string;                     // e.g., 't3.medium'
  imageId: string;                           // e.g., 'ami-0abcd...'
  hibernation: boolean;                     // enable hibernate support
  rootVolumeSize: number;                   // GiB
  rootVolumeEncrypted: boolean;             // required for hibernate
  iamInstanceProfile: string | null;        // IAM role name
  userData: string | null;                  // base64-encoded user data
  tags?: Record<string, string>;            // AWS tags
}
```

> **AWS API note**:
> - `hibernation` → `HibernationOptions.Configured`
> - cite: https://docs.aws.amazon.com/AWSEC2/latest/APIReference/API_RequestLaunchTemplateData.html

#### DeclaredAwsEc2Instance (updated)

```typescript
interface DeclaredAwsEc2Instance {
  id?: string;                              // @metadata
  exid: string;                             // @unique
  template: Ref<typeof DeclaredAwsEc2LaunchTemplate> | null;  // optional
  subnetId: string;                         // where to place
  securityGroupIds: string[];               // network rules
  privateIp?: string;                       // @readonly
}
```

> **note**: `status` moved to `DeclaredAwsEc2InstanceSession` — instance is identity only.
> **note**: `template` is optional — `null` for instances without launch template (backwards compat).
> **note**: template = **what** the machine is, instance = **where** to place it.

#### DeclaredAwsEc2InstanceSession (new)

```typescript
interface DeclaredAwsEc2InstanceSession {
  instance: Ref<typeof DeclaredAwsEc2Instance>;  // @unique
  status: 'active' | 'stopped' | 'hibernated';
}
```

> **status values**:
> - `active` → AWS status `running`
> - `stopped` → AWS status `stopped`
> - `hibernated` → AWS status `stopped` (with hibernate flag)

> **why separate?** session status changes cannot accidentally mutate instance config. each layer protects the one below.

### timeline

1. **declare** launch template with hibernate enabled
2. **declare** instance with template reference
3. **declare** session with desired status
4. **apply** — declastruct creates template, instance, and sets status
5. **change session status** — re-declare with new status, re-apply
6. declastruct handles all transitions (start, stop, hibernate, resume)

## mental model

### how users describe it to a friend

"it's like laptop sleep mode. when you wake it, all your work is exactly where you left it. except it's a cloud server, and you only pay for storage while it sleeps."

### analogies

| concept | analogy |
|---------|---------|
| launch template | motherboard + RAM + disk config |
| instance | the physical machine |
| session | power state (on/sleep/off) |
| hibernate | laptop sleep mode |
| resume | wake from sleep |

### 3-layer model

| layer | resource | what it holds |
|-------|----------|---------------|
| 1 | LaunchTemplate | **what** the machine is (AMI, type, hibernate, volume) |
| 2 | Instance | **where** to place it (subnet, security groups) + identity |
| 3 | Session | **lifecycle** state (active/stopped/hibernated) |

### terms: user vs system

| user says | session status | AWS status |
|-----------|---------------|------------|
| "turn it on" | `active` | running |
| "turn it off" | `stopped` | stopped |
| "sleep" | `hibernated` | stopped (hibernated) |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| cost reduction | yes | 6-7x cheaper for bursty workloads |
| fast resume | yes | 30-60s vs 2-5min cold boot |
| state preservation | yes | RAM persisted to EBS |
| declarative management | yes | pure declarations, no imperative set* calls |
| safety | yes | 3-layer model prevents accidental mutations |

### pros

- **massive cost savings**: hibernate + resume = pay storage only when idle
- **fast iteration**: ~30s resume vs 2-5min cold start
- **state continuity**: processes, files, shell sessions preserved
- **clean separation**: template → instance → session, each layer protected
- **pure declarative**: no imperative operations, just declare and apply

### cons

- **hibernate constraints**: not all instance types/AMIs support it
- **root volume must be encrypted**: security requirement, slight overhead
- **RAM < 150 GiB**: limitation for very large instances
- **can't retrofit**: must be enabled at launch, not on extant instances

### edgecases and pit-of-success

| edgecase | pit-of-success design |
|----------|----------------------|
| hibernate on unsupported instance | AWS fails at apply time (source of truth) |
| unencrypted root volume | require `rootVolumeEncrypted: true` when `hibernation: true` |
| resume timeout | declastruct uses AWS SDK waiters with 5min timeout |
| template change requires new instance | 3-layer model makes this explicit |
| accidental instance mutation | session layer isolates lifecycle from config |

## groundwork

### external research

**AWS EC2 Hibernate Prerequisites** (verified via docs):
- supported instance families: M3-M8, T2-T4g, C3-C8, R3-R8, I3-I8, X2gd-X8i
- root volume: EBS, encrypted, gp2/gp3/io1/io2
- RAM: < 150 GiB for Linux
- AMI: AL2023, Amazon Linux 2, Ubuntu 20.04+, RHEL 8+
- must be enabled at launch (cannot retrofit)
- cite: https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/hibernating-prerequisites.html

**AWS SDK Commands**:
- `StopInstancesCommand` with `Hibernate: true` for hibernate
- `StartInstancesCommand` for resume from hibernated
- status transitions: running → stopped (with hibernation state)

### internal research

**current state** (verified in codebase):

the current `DeclaredAwsEc2Instance` is minimal — only `exid`, `status` (readonly), `privateIp` (readonly). no ami, instanceType, subnet, or template. current operations:
- `getEc2Instance` — read extant instance by exid
- `setEc2InstanceStatus` — start/stop extant instance

**this vision adds instance creation** — not a refactor. launch templates + instance placement + session lifecycle are all new scope.

**extant patterns** (verified in codebase):

vocab:

patterns:

## what is awkward?

### hibernate status from AWS

AWS reports `stopped` for both stopped and hibernated instances. to distinguish:
- query `HibernationOptions.Configured` + `StateReason.Code` from DescribeInstances
- AWS sets `StateReason.Code` to `Client.UserInitiatedHibernate` for hibernated instances

### launch template versions

AWS launch templates support versions. for now: always use `$Latest`. version support can be added later if needed.

# blocker.1: Race condition in findsert check-then-act for EC2 instances

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/ec2Instance/setEc2Instance.ts:55

The setEc2Instance function performs a non-atomic read (getEc2Instance by exid) followed by a create (RunInstances) without locks, transactions, or AWS-side idempotency keys. Under concurrent calls, two instances of the code can both observe no existing instance and both execute the create, producing duplicate EC2 instances with the same exid tag. This is a classic read-modify-write race that manifests intermittently under load or parallel declarative updates, leading to unpredictable state and hard-to-debug production incidents. The findsert path claims idempotency but violates it due to the unprotected window between check and mutation.

**snippet**:
```ts
  // check if instance already exists
  const instanceFound = await getEc2Instance(
    { by: { unique: { exid: instance.exid } } },
    context,
  );

  // if findsert and found, return extant
  if (input.findsert && instanceFound) return instanceFound;

  // if upsert and found, EC2 instances cannot be updated in place
  // .note = EC2 instances are immutable — must terminate and recreate for changes
  if (input.upsert && instanceFound)
    return UnexpectedCodePathError.throw(
      'EC2 instance upsert not supported — instances are immutable; terminate and recreate for changes',
      { instance, instanceFound },
    );
```

# blocker.4: TOCTOU races and non-idempotent external state management in VPC tunnel

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/vpcTunnel/setVpcTunnel.ts:70

setVpcTunnel performs a series of check-then-act operations against shared external state (port usage via isPortInUse, cache files via isFilePresent + fs.readFile, process liveness via isProcessAlive, and AWS instance sessions) without any synchronization, locks, or atomic primitives. Multiple concurrent invocations (or retries) can interleave, causing duplicate tunnel processes, port conflicts, stale cache reads, or orphaned processes. The port-in-use + cache-present logic is a textbook time-of-check-to-time-of-use race; the subsequent spawn, writeFile, and kill operations are not protected, violating predictability and enabling silent data corruption or impossible-to-debug production incidents.

**snippet**:
```ts
  // check if port is already in use
  const portInUse = await isPortInUse({ port: input.from.port });

  // handle port in use scenario
  if (portInUse) {
    const cacheFilePresent = await isFilePresent({ path: cachePath });

    // port used by unknown process; kill it to reclaim
    if (!cacheFilePresent) {
      killProcessOnPort({ port: input.from.port });
    }

    // check if existing tunnel is ours and healthy
    if (cacheFilePresent) {
      const cacheContent = await fs.readFile(cachePath, 'utf-8');
      const cache: TunnelCacheFile = JSON.parse(cacheContent);

      // return early if existing tunnel is alive and healthy (idempotent)
      if (
        isProcessAlive({ pid: cache.pid }) &&
        (await isTunnelHealthy({ port: input.from.port }))
      )
        return castIntoDeclaredAwsVpcTunnel({
          unique: input,
          status: 'OPEN',
          pid: cache.pid,
        });

      // cleanup stale tunnel
      if (isProcessAlive({ pid: cache.pid }))
        process.kill(cache.pid, 'SIGTERM');
      await fs.rm(cachePath, { force: true });
    }
  }
```

# blocker.5: Time assumptions and non-deterministic waits in VPC tunnel startup

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/vpcTunnel/setVpcTunnel.ts:180

The tunnel startup logic uses a hard-coded 300_000ms timeout, event-driven waits on stdout for the literal string 'Waiting for connections', and multiple waitUntilInstance* calls with maxWaitTime: 300. These embed assumptions about wall-clock latency, AWS SSM behavior, and network conditions that are not guaranteed (the code itself comments 'takes up to 2.5min every once in a while; that's a known "feature"'). Under variable load, network jitter, or delayed AWS responses the promise can reject unpredictably or the tunnel can be killed after partial side effects, producing non-deterministic outcomes and intermittent failures that are impossible to reproduce in tests.

**snippet**:
```ts
  // wait for tunnel to be ready
  await new Promise<void>((resolve, reject) => {
    const timeout = setTimeout(
      () =>
        reject(
          new UnexpectedCodePathError('tunnel startup timeout', {
            input,
            logPath,
          }),
        ),
      300_000, // takes up to 2.5min every once in a while; that's a known "feature" w/ aws here
    );

    tunnelProcess.stdout?.on('data', (data: Buffer) => {
      if (data.toString().includes('Waiting for connections')) {
        clearTimeout(timeout);
        resolve();
      }
    });

    tunnelProcess.stderr?.on('data', (data: Buffer) => {
      context.log?.debug?.('tunnel stderr', { data: data.toString() });
    });

    tunnelProcess.on('error', (err) => {
      clearTimeout(timeout);
      reject(
        new UnexpectedCodePathError(`tunnel error: ${err.message}`, {
          input,
          logPath,
        }),
      );
    });

    tunnelProcess.on('exit', (code) => {
      if (code !== 0) {
        clearTimeout(timeout);
        reject(
          new UnexpectedCodePathError(`tunnel exited with code ${code}`, {
            input,
            logPath,
          }),
        );
      }
    });
  });
```

The function performs mutations (spawn detached 'aws' process, createWriteStream + pipe logs, fs.writeFile cache, process.kill, fs.rm) that are not obvious from the signature and are not guarded for idempotency beyond best-effort cache checks. Retries or concurrent calls can duplicate processes, duplicate log files, or leave orphaned processes/files. The 'aws' CLI is a hidden external dependency; its absence or version differences produce different runtime behavior. These side effects make the operation non-deterministic and unsafe to retry, directly matching the documented blocker patterns for hidden side effects and non-idempotent operations.

**snippet**:

# blocker.7: Post-mutation get assumes immediate AWS visibility (eventual consistency hazard)

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/ec2Instance/setEc2Instance.ts:120

After RunInstancesCommand and CreateLaunchTemplateCommand, the code immediately calls get* by primary id and throws UnexpectedCodePathError if the resource is not found. This embeds an implicit assumption that AWS will make the newly created resource visible within the next Describe call, without retries, backoff, or handling of transient inconsistency. Under load or in certain regions/accounts this can fail spuriously, turning a successful create into an error and leaving the caller with no way to recover the created resource id.

**snippet**:
```ts
  // get the created instance to return full domain object
  const created = await getEc2Instance(
    { by: { primary: { id: instanceId } } },
    context,
  );

  if (!created)
    return UnexpectedCodePathError.throw(
      'could not find newly created EC2 instance',
      { instanceId, instance },
    );
  return created;
```
